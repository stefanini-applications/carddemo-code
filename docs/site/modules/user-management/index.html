👥 User Management

User Story Development Guide — CardDemo Mainframe Application

📋 Overview

Module ID: user-management

Purpose: Admin-only CRUD operations on the USRSEC VSAM file — list, add, update, and delete user records. Admin menu (CA00) is the entry point.

Business Context: Controls who can access the CardDemo application and with what privileges. Separate from RACF (environment security) — this is application-level user management.

CA00: COADM01C (admin menu) CU00: COUSR00C (list) CU01: COUSR01C (add) CU02: COUSR02C (update) CU03: COUSR03C (delete)

🎯 User Story Development Guide

📝 US Templates

  • As an admin, I want to list all system users so that I can see who has access
  • As an admin, I want to add a new user account so that a new employee can access the system
  • As an admin, I want to update a user's password so that I can reset credentials when requested
  • As an admin, I want to change a user's type from regular to admin so that I can promote a user
  • As an admin, I want to delete a user account so that I can revoke access when an employee leaves
  • As an admin, I want confirmation before deleting a user so that I don't accidentally remove the wrong account

⚡ Development Acceleration Factors

  • COUSR00C–COUSR03C: Complete CRUD implementation — all four operations available as reference
  • CSUSR01Y copybook: 80-byte user record layout shared between all user management programs
  • Admin-only access: SEC-USR-TYPE='A' check enforced by COSGN00C routing — no explicit check needed in user management programs (routing handles it)
  • Full VSAM CRUD pattern: READ, WRITE, REWRITE, DELETE all demonstrated across the four programs

📊 Complexity Guidelines

  • Simple (1-2 pts): User list (browse USRSEC), user detail view
  • Medium (3-5 pts): User add with duplicate check (CU01); user update (CU02); user delete with confirmation (CU03)
  • Complex (5-8 pts): Bulk user import, password policy enforcement, audit logging of user management actions

📋 User Record Data Model

01  SEC-USER-DATA.         -- 80 bytes, VSAM KSDS
    05  SEC-USR-ID     PIC X(08). -- Primary key (e.g. USER0001)
    05  SEC-USR-FNAME  PIC X(20). -- First name
    05  SEC-USR-LNAME  PIC X(20). -- Last name
    05  SEC-USR-PWD    PIC X(08). -- Password (plain text - tech debt)
    05  SEC-USR-TYPE   PIC X(01). -- 'U'=regular, 'A'=admin
    05  SEC-USR-FILLER PIC X(23).

⚙️ Business Rules

  • Only admin users (SEC-USR-TYPE='A') can access the admin menu CA00 and user management functions
  • SEC-USR-ID must be unique — add fails with DUPREC if already exists
  • User type must be 'U' (regular) or 'A' (admin) — other values invalid
  • Passwords stored in plain text in USRSEC — known tech debt, flagged for modernization
  • Delete requires confirmation screen interaction (two-step pseudo-conversational pattern)
  • Admin cannot delete their own account while logged in (implementation-dependent)

✅ Acceptance Criteria Patterns

  • Given the admin is on the user list screen, when the list loads, then all users in USRSEC are displayed with ID, name, and type
  • Given valid new user data, when add user is submitted, then record is written to USRSEC and confirmation displayed
  • Given a duplicate user ID, when add user is submitted, then "User ID already exists" error is displayed
  • Given a user is selected for deletion, when admin confirms, then record is deleted from USRSEC and success message displayed
  • Given a regular user attempts to access CA00, when the admin menu is requested, then access is denied or the user is never routed to that transaction
