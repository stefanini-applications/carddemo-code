👥 User Management

User Story Development Guide — Admin User CRUD Operations

📋 Overview

Module ID: user-management

Purpose: Provides full CRUD (Create, Read, Update, Delete) operations for CardDemo application users. Accessible only to admin users (SEC-USR-TYPE = 'A').

Business Context: Admin users manage the user directory for CardDemo. User records control who can log in and what role they have (Regular vs Admin). The USRSEC VSAM file is the authoritative source for application-level credentials.

⚠️ Security Note: This module is accessible only from the Admin Menu (CA00), which is only available after login as an admin user (SEC-USR-TYPE = 'A'). Regular users cannot navigate to these transactions.

🎯 User Story Templates

List Users

As an admin, I want to see a list of all users so that I can review who has access to the CardDemo system.
Acceptance Criteria:
  • Transaction CU00 (COUSR00C) displays all records from USRSEC VSAM
  • List shows: SEC-USR-ID, SEC-USR-FNAME, SEC-USR-LNAME, SEC-USR-TYPE
  • Password (SEC-USR-PWD) is NOT displayed in the list
  • Paging supported for large user lists
  • Admin can select a user to view/update details

Add User

As an admin, I want to add a new user so that new employees can log into CardDemo.
Acceptance Criteria:
  • Transaction CU01 (COUSR01C) provides add user form
  • Required fields: SEC-USR-ID, SEC-USR-FNAME, SEC-USR-LNAME, SEC-USR-PWD, SEC-USR-TYPE
  • SEC-USR-ID must be unique — duplicate rejected with "User ID already exists"
  • SEC-USR-TYPE must be 'U' (Regular) or 'A' (Admin)
  • On success, record written to USRSEC VSAM; success message displayed
  • On error, screen redisplays with entered data and error message

Update User

As an admin, I want to update a user's name or password so that their account information is current.
Acceptance Criteria:
  • Transaction CU02 (COUSR02C) shows current user data for editing
  • All fields except SEC-USR-ID are editable
  • After submit, USRSEC VSAM record rewritten
  • Admin can change SEC-USR-TYPE to promote/demote user privileges
  • Validation: password 1–8 chars; type must be 'U' or 'A'

Delete User

As an admin, I want to delete a user so that terminated employees lose access to the system.
Acceptance Criteria:
  • Transaction CU03 (COUSR03C) shows user to be deleted with confirmation prompt
  • Admin must confirm deletion (e.g., press PF key) before record is deleted
  • After confirmation, record deleted from USRSEC VSAM
  • Deleting a user does not affect account/transaction records linked to that user's activity
  • Admin cannot delete their own user ID (to prevent lockout)
  • Deletion of non-existent user displays "User not found" error

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Add user search by last name on list screen
  • Display user count on list header
  • Mask password field on add/update screen

⚠️ Medium (3-5 pts)

  • Add password complexity validation (min length, mixed case)
  • User status flag (active/suspended) separate from delete
  • Audit log for user additions and deletions

🔴 Complex (5-8 pts)

  • Batch user load from external file (IEBGENER pattern)
  • Role-based access control with granular permissions
  • User activity report — last login, transaction count

SEC-USER-DATA (CSUSR01Y, 80 bytes)

SEC-USR-ID      PIC X(08)  -- User ID (VSAM key, max 8 chars)
SEC-USR-FNAME   PIC X(20)  -- First name
SEC-USR-LNAME   PIC X(20)  -- Last name
SEC-USR-PWD     PIC X(08)  -- Password (plain text, max 8 chars)
SEC-USR-TYPE    PIC X(01)  -- 'U'=Regular user, 'A'=Admin
SEC-USR-FILLER  PIC X(23)

Initial Setup

  • DUSRSECJ — JCL job to initially load USRSEC from sample data file using IEBGENER
  • Sample data includes ADMIN001 (admin) and USER0001 (regular) as default users
