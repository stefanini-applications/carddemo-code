👥 User Management

User Story Development Guide — Admin User CRUD

← Back to Documentation Index

Module ID: user-management

Purpose: Admin-only module providing full CRUD operations on application users stored in the USRSEC VSAM file. Controls who can access the CardDemo application and with what level of privilege.

⚠️ Admin Only: Only users with SEC-USR-TYPE = 'A' can access these functions.

🎯 User Story Development Guide

📝 US Templates

  • List Users: As an admin, I want to list all users so that I can review who has access to the system.
  • Add User: As an admin, I want to add a new user so that new employees can access the application.
  • Update User: As an admin, I want to update a user's password or name so that user information stays current.
  • Delete User: As an admin, I want to delete a user so that terminated employees lose system access.
  • User Type: As an admin, I want to set a user's type (Regular/Admin) so that the correct menu is presented at login.

⚡ Development Acceleration Factors

  • CSUSR01Y copybook: User record structure shared with authentication module — same 80-byte record layout
  • COUSR00C–COUSR03C pattern: Consistent CICS pseudo-conversational CRUD pattern across all four programs
  • USRSEC KSDS: Simple indexed file with SEC-USR-ID as key — straightforward CICS READ/WRITE/REWRITE/DELETE
  • COADM01C routing: Admin menu already routes to all user management transactions

📊 Complexity Guidelines

🔧 Technical Foundation

🏗️ Key Components

  • COADM01C — Admin menu; routes to CU00–CU03 and optional DB2 transaction type management
  • COUSR00C — List users; sequential browse of USRSEC KSDS
  • COUSR01C — Add user; CICS WRITE to USRSEC
  • COUSR02C — Update user; CICS READ UPDATE / REWRITE
  • COUSR03C — Delete user; CICS DELETE from USRSEC
  • COADM01 / COUSR00–COUSR03 (BMS maps) — Admin and user management 3270 screens
  • CSUSR01Y (copybook) — User security record (80 bytes)
  • COADM02Y (copybook) — Admin screen common area

🔗 Dependencies

  • USRSEC (KSDS): User security file; SEC-USR-ID (8 chars) is the KSDS key
  • Authentication module: Same USRSEC file used for signon validation
  • DUSRSECJ (JCL): Initial load/reset of USRSEC file using IEBGENER

🌐 CICS Transactions

CA00  → COADM01C  — Admin menu (entry point for admin users)
CU00  → COUSR00C  — List users (sequential browse)
CU01  → COUSR01C  — Add user
CU02  → COUSR02C  — Update user
CU03  → COUSR03C  — Delete user

📋 User Security Data Model (CSUSR01Y, 80 bytes)

SEC-USER-DATA (USRSEC KSDS):
  SEC-USR-ID      PIC X(08)   — User ID (KSDS key, 8 chars max)
  SEC-USR-FNAME   PIC X(20)   — First name
  SEC-USR-LNAME   PIC X(20)   — Last name
  SEC-USR-PWD     PIC X(08)   — Password (plain text, 8 chars max)
  SEC-USR-TYPE    PIC X(01)   — 'R'=Regular, 'A'=Admin
  SEC-USR-FILLER  PIC X(23)   — Available for extension

Default Users:
  ADMIN001 / PASSWORD / Admin
  USER0001 / PASSWORD / Regular

📋 Business Rules

  • BR-USR-01: Only Admin users (SEC-USR-TYPE = 'A') can access the CA00 admin menu and CU-series transactions
  • BR-USR-02: User IDs are 8 characters maximum; must be unique in USRSEC
  • BR-USR-03: Passwords are stored in plain text in USRSEC (8 chars max) — security consideration for migration
  • BR-USR-04: User type must be 'R' (Regular) or 'A' (Admin)
  • BR-USR-05: Deleting a user removes their USRSEC record; they can no longer log in
  • BR-USR-06: The DUSRSECJ job resets the USRSEC file to initial sample data

✅ Acceptance Criteria Examples

  • Access Control: Given a regular user (type 'R'), when they attempt to navigate to CA00, then they are denied access with an appropriate message
  • Add User: Given valid user ID, names, password, and type, when admin submits CU01, then the new user record is written to USRSEC and the user can log in via CC00
  • Delete User: Given an existing user, when admin confirms deletion in CU03, then the USRSEC record is deleted and the user can no longer log in
  • Unique ID: Given an attempt to add a user with an existing user ID, when CU01 is submitted, then an error is displayed and the duplicate is rejected
  • Performance: User list must display within 3 seconds; all CRUD operations must complete within 3 seconds
