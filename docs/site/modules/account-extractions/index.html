⚡ Account Extractions

Guide for stories driven by MQ request/response patterns that surface VSAM data asynchronously.

📋 Overview

Purpose: Reveal how MQ-driven request/response flows (`CDRD`/`CDRA`) fetch system dates and account snapshots from VSAM so distributed systems can stay synchronized.

Business Context: Mirrors modern asynchronous APIs by using MQ queues (`CARDDEMO.REQUEST.QUEUE`, `CARDDEMO.RESPONSE.QUEUE`) together with COBOL responders.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Time sync: As a scheduler, I want to call `CDRD` so downstream jobs know the mainframe system date without querying the clock manually.
  • Account lookup: As an integration partner, I want `CDRA` to return account data via MQ so I can show customer details in a portal.
  • MQ extension: As an engineer, I want to add a new field to the `ACCT-RESPONSE-MSG` so outbound services consume extra metadata.

⚡ Development Acceleration Factors

  • Message specs: `app/app-vsam-mq/README.md` documents `DATE-REQUEST-MSG`, `ACCT-REQUEST-MSG`, `DATE-RESPONSE-MSG`, `ACCT-RESPONSE-MSG` with fixed lengths.
  • MQ definitions: Connection (`MQ01`) and queue (`CARDREQ`, `CARDRES`) definitions minimize configuration work for new request types.
  • VSAM reference: Account data reuses copybook `CVACT01Y`, so extra fields map directly to existing structures.

📊 Complexity Guidelines

  • Simple (1-2 pts): Change the text on the `CDRD` map to explain how to use PF keys or display the queue name.
  • Medium (3-5 pts): Add a new data field to the `ACCT-RESPONSE-MSG` and update the COBOL parser to reorder the CSV payload.
  • Complex (5-8 pts): Introduce a new MQ requester that reuses the same queues but aggregates multiple `ACCT-REQUEST-MSG` responses.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs: `CODATE01` (system date responder) and `COACCT01` (account detail responder).
  • MQ definitions: `MQ01` (connection), `CARDREQ`/`CARDRES` (queues) defined under `app/app-vsam-mq/csd`.
  • Message formats: `DATE-REQUEST-MSG`, `ACCT-REQUEST-MSG`, `DATE-RESPONSE-MSG`, `ACCT-RESPONSE-MSG` (fixed-length structures in the module README).
  • VSAM dataset: `AWS.M2.CARDDEMO.ACCTDATA.PS` and `CARDXREF` provide the account data payload.

🔗 Dependencies

  • IBM MQ plus CICS-MQ support (`DEFINE MQCONN`, `DEFINE MQQUEUE`).
  • VSAM account files and cross-reference datasets.
  • MQ listeners that reply on the same queues (any MQ client/partner can be wired to the request queue).

🌐 Public Interfaces

CDRD  (CODATE01) - Sends DATE-REQUEST-MSG to CARDDEMO.REQUEST.QUEUE and reads DATE-RESPONSE-MSG
CDRA  (COACCT01) - Sends ACCT-REQUEST-MSG with account number and receives ACCT-RESPONSE-MSG containing 300-byte ACCOUNT-DATA
CARDDEMO.REQUEST.QUEUE - Input queue for asynchronous requests
CARDDEMO.RESPONSE.QUEUE - Output queue for replies

✅ Acceptance Criteria Examples

  • CDRD must emit a DATE-RESPONSE-MSG with the current system date and the matching REQUEST-ID.
  • CDRA must populate `ACCOUNT-DATA` fields using `CVACT01Y` and correlate replies via the MQ `REQUEST-ID`/`RESPONSE-ID`.
  • MQ queues must be defined with the names in the README and respond within 5 seconds to avoid timeouts.
