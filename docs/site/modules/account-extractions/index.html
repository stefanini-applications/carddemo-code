account-extractions

Detailed technical documentation for the CardDemo optional module that integrates CICS, IBM MQ, and VSAM to support asynchronous account and date extraction workloads.

Module ID: account-extractions Primary Tech: COBOL + CICS + MQ + VSAM Programs: CODATE01, COACCT01 Transactions: CDRD, CDRA

Module Overview and Purpose

The account-extractions module exposes asynchronous request/reply processing around legacy CardDemo account data. Its primary purpose is to allow external systems to request operational values without direct 3270 interaction.

The module has two entry points:

  • CDRD transaction mapped to CODATE01 for system date/time extraction.
  • CDRA transaction mapped to COACCT01 for account detail extraction from VSAM.

Both programs implement a common pattern: open queues, read request message, process payload, write response, and route operational failures to an error queue.

Business value: this module works as a modernization bridge so cloud or distributed services can consume card-account information through MQ while leaving the mainframe persistence model unchanged.

Architectural Context

Within CardDemo, account-extractions is an optional extension that sits at the integration boundary. It does not own persistent data structures, and it does not replace core account maintenance screens. It consumes VSAM data and publishes MQ responses.

Inbound Flow

  • External producer sends request to request queue.
  • CICS transaction is triggered with queue context.
  • Program reads request via MQGET and parses message body.

Processing Flow

  • CODATE01: reads system time via CICS time services.
  • COACCT01: validates request function/key and reads ACCTDAT.
  • Both preserve MQ identity fields for response correlation.

Outbound Flow

  • Success response is placed onto reply queue.
  • Operational errors are written to CARD.DEMO.ERROR.
  • CICS SYNCPOINT and queue close operations complete the unit of work.

Runtime Boundaries

  • No DB2 dependency in this module path.
  • No IMS dependency in this module path.
  • Primary persistence dependency is VSAM account master.

External Client

Business Rules (Code-Verified)

  • Function guard: WS-FUNC must equal INQA for account extraction path.
  • Key guard: WS-KEY must be non-zero numeric for account extraction path.
  • Existence rule: DFHRESP(NOTFND) yields invalid-request style message payload.
  • Error routing rule: MQ/file failures are formatted in MQ-ERR-DISPLAY and sent to error queue.
  • Loop termination rule: MQRC-NO-MSG-AVAILABLE sets NO-MORE-MSGS and exits processing loop.
  • Wait rule: request read uses MQGMO-WAITINTERVAL = 5000 (5 seconds).
  • Syncpoint rule: each main loop pass performs CICS SYNCPOINT before next request.

Development Guidelines and Patterns

Pattern 1: Open/Get/Put/Close Discipline

Follow the current module pattern: open input/output/error queues before loop, perform request reads and reply writes inside loop, and close all opened objects in termination section with condition checks. 

PERFORM 2100-OPEN-ERROR-QUEUE
PERFORM 2300-OPEN-INPUT-QUEUE
PERFORM 2400-OPEN-OUTPUT-QUEUE
PERFORM 3000-GET-REQUEST
PERFORM 4000-MAIN-PROCESS UNTIL NO-MORE-MSGS
PERFORM 8000-TERMINATION

Pattern 2: EVALUATE-Based Error Routing

Use EVALUATE on MQ and CICS response codes. On non-normal cases, populate MQ-ERR-DISPLAY and call 9000-ERROR to centralize operational fault publication.

Pattern 3: Preserve Message Identity

Save incoming MQMD-MSGID and MQMD-CORRELID and set them into outbound descriptor to keep deterministic request/reply matching.

Pattern 4: Keep Reply Payload Backward Compatible

  • Do not reorder existing fixed-position request fields without versioning strategy.
  • When extending response text/payload, append fields to avoid client parser breaks.
  • Document field-level changes in module README and system overview module catalog.

Pattern 5: Reuse Core Copybooks

Avoid duplicate account layout definitions. Use shared copybook structures (for example CVACT01Y) so core and extraction paths stay aligned.

Appendix B: Operational Runbook Notes

Startup Validation Steps

  1. Confirm CICS group with CDRD and CDRA transactions is installed and enabled.
  2. Confirm CODATE01 and COACCT01 programs are available in configured load library.
  3. Verify MQ connectivity from CICS region and test queue open permissions.
  4. Send a synthetic date request and verify response in date reply queue.
  5. Send a valid account request and verify successful account payload reply.
  6. Send an intentionally invalid account request and verify predictable invalid parameter response.
  7. Force an MQ failure scenario and verify error queue receives formatted operational message.

Performance Considerations

  • 5-second wait interval is a latency/CPU tradeoff and should be benchmarked before change.
  • Large response expansions can increase queue depth and downstream processing time.
  • CICS syncpoint frequency affects throughput and recoverability characteristics.
  • Error queue monitoring should be automated to detect integration degradation early.

Security Considerations

  • Protect request/reply/error queues with least-privilege authorities.
  • Treat account payload as sensitive; apply transport and consumer controls accordingly.
  • Constrain transaction invocation rights to approved integration identities.
  • Review operational logs for exposure of account identifiers in error text.

Module Summary

The account-extractions module is a focused integration layer: MQ in, data extraction, MQ out. It keeps core CardDemo data structures in VSAM and offers two stable transaction entry points for external integration workflows.

  • Use CDRD for date/time synchronization scenarios.
  • Use CDRA for account detail extraction scenarios.
  • Use error queue telemetry to drive support and reliability workflows.
