💳 Transaction Type Module

Guide for DB2-backed reference data stories that feed the legacy transaction engine.

📋 Overview

Purpose: Demonstrate DB2 integration through `CTTU`/`CTLI` screens and batch jobs while syncing transaction-type metadata back to VSAM so the core CICS engine understands new categories.

Business Context: Having a single source of truth for transaction types empowers governors to define categories, categories-to-transaction mapping, and ensure reporting/transaction-processing stay aligned.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Create Transaction Type: As an admin, I want to define a new transaction-type code plus description via `CTTU` so the VSAM/DB2 blend accepts it.
  • Cursor Navigation: As a user, I want `CTLI` to page forward/backward through categories (using forward/backward cursors) so I can audit the dataset.
  • Batch Sync: As a release engineer, I want `TRANEXTR` to produce VSAM-compatible files after DB2 updates so nightly posting reads the latest metadata.

⚡ Development Acceleration Factors

  • README guidance: `app/app-transaction-type-db2/README.md` spells out DB2 tables, SQL statements, and installation steps.
  • DB2 artifacts: DDL scripts under `app/app-transaction-type-db2/ddl` describe `TRANSACTION_TYPE` and `TRANSACTION_TYPE_CATEGORY` with FK and PK constraints.
  • Batch jobs: `TRANEXTR`, `CREADDB21`, and `MNTTRDB2` jobs show the extraction path and allow you to verify data propagation into VSAM.

📊 Complexity Guidelines

  • Simple (1-2 pts): Update a BMS label or add a new column to the `CTLI` list screen (no dataset change).
  • Medium (3-5 pts): Add a new field to `TRANSACTION_TYPE_CATEGORY`, update static SQL in `CTTU`, and refresh the DDL/sync scripts.
  • Complex (5-8 pts): Introduce a new DB2 table (e.g., reward tiers) and wire `TRANEXTR` plus posting jobs to include the new data.

🔧 Technical Foundation

🏗️ Key Components

  • `CTTU` (`COTRTUPC`) handles add/edit operations using static embedded SQL and SQLCA error handling.
  • `CTLI` (`COTRTLIC`) demonstrates forward/backward cursor navigation and grouped deletes.
  • `TRANEXTR` (`DSNTIAUL` job) extracts DB2 rows into VSAM-compatible files consumed downstream.
  • `MNTTRDB2` (`COBTUPDT`) is the batch maintenance program for periodic updates.
  • Control files in `app/app-transaction-type-db2/ctl` and precompiler declarations in `dcl/` keep COBOL/DB2 integrations aligned.

🔗 Dependencies

  • DB2 subsystem (DDL, plans, DCLGEN, and SQLCA) is mandatory.
  • CICS with DB2 support for static SQL programs.
  • JCL automations for `TRANEXTR`, `CREADB21`, `MNTTRDB2` to propagate data.
  • VSAM for downstream transaction flows—`TRANEXTR` must keep this file consistent.

🗂️ Public Interfaces

CTTU - Transaction type add/edit screen (embedded SQL)
CTLI - Transaction type list/update/delete (forward/backward cursor)
TRANEXTR - Extract DB2 metadata into VSAM files
CREADB21 - Create DB2 objects used by the module
MNTTRDB2 - Batch maintenance job for transaction types

✅ Acceptance Criteria Examples

  • DB2 constraints (`PRIMARY KEY`, `FOREIGN KEY`) prevent deleting a transaction type that still backs VSAM entries.
  • `TRANEXTR` writes VSAM files that match the layout expected by `app/cpy/CVTRA04Y.cpy` so `CT00` reads valid data.
  • `CTTU` surfaces SQLCA error codes in BMS message lines when DB2 rejects a duplicate key.
