1. Overview

The Transaction Type module is an optional extension for the CardDemo credit-card management platform. It demonstrates IBM DB2 integration patterns using embedded static SQL inside IBM CICS COBOL programs, providing a robust relational approach to managing transaction type reference data while maintaining backward compatibility with the legacy VSAM-based transaction processing system.

The module resides entirely under app/app-transaction-type-db2/ and operates independently of the core application once the DB2 subsystem is configured and the CICS DB2 entry (CARDDEMO) is in place.

Purpose and Business Context

  • Maintain transaction type metadata (codes and descriptions) in a DB2 relational table for improved data integrity and concurrent update safety.
  • Manage transaction type categories with full referential integrity enforced by DB2 foreign keys, preventing orphaned category records.
  • Expose an admin CRUD UI (CICS green-screen transactions CTTU and CTLI) for operators to add, browse, update, and delete type records.
  • Synchronize reference data back to VSAM sequential files via the TRANEXTR batch job so the legacy CT00/COTRN transaction screens continue to render consistent category labels without touching DB2 directly.
  • Serve as a canonical teaching example of DB2 cursor navigation, SQLCA error checking, and the DB2 precompiler workflow in a CICS environment.
ℹ️ Optional Module: This module requires a DB2 subsystem accessible from the CICS region. The base CardDemo application runs entirely on VSAM without DB2. Install this module only when demonstrating DB2 integration patterns or administering transaction type metadata from an online terminal.

Dual-Storage Strategy

The module uses a dual-storage strategy: DB2 is the authoritative source of truth for administrative operations; VSAM sequential files serve as a nightly snapshot consumed by the high-speed 3270 transaction path. Changes made via CTTU/CTLI or MNTTRDB2 are not visible to CT00 screens until TRANEXTR is executed (typically nightly).

3. Key Components

Online (CICS) Components

Transaction ID BMS Mapset COBOL Program Function
CTTU COTRTUP (map CTRTUPA) COTRTUPC Add and update a single transaction type record via embedded static SQL INSERT/SELECT/UPDATE with SQLCA validation.
CTLI COTRTLI (map CTRTLIA) COTRTLIC List, update, and delete transaction types using forward/backward DB2 cursors; displays up to 7 rows per page with PF7/PF8 paging.

COTRTUPC – Transaction Type Add/Edit (CTTU)

  • Accepts a 2-character transaction type code (TR_TYPE) and a description of up to 50 characters.
  • On first invocation (no COMMAREA), displays a blank form prompting the operator to enter new data.
  • Validates both fields via COBOL 88-level flags (FLG-TRANFILTER-ISVALID, FLG-DESCRIPTION-ISVALID) before issuing any SQL.
  • Supports a two-phase confirm flow: PF5 previews changes, a second PF5 commits them; PF12 cancels.
  • Detects concurrent update conflicts: if a record changes between fetch and update it raises "Record changed by someone else. Please review".
  • Invoked from Admin Menu (CA00) option 6 or directly by entering CTTU in the CICS transaction field.

COTRTLIC – Transaction Type List/Update/Delete (CTLI)

  • Displays up to 7 transaction type records per screen page.
  • Uses two declared DB2 cursors:
    • C-TR-TYPE-FORWARD – fetches rows with TR_TYPE >= :WS-START-KEY ordered ascending for PF8 (page down).
    • C-TR-TYPE-BACKWARD – fetches rows with TR_TYPE < :WS-START-KEY ordered descending for PF7 (page up).
  • Supports optional filter by type code or description (LIKE search); filter changes trigger a new cursor open.
  • Row-level selection: enter U in the selection column to update a row's description; enter D to request deletion (with referential integrity enforced by DB2).
  • Transfer to CTTU is performed via XCTL when an update is selected, passing context through COMMAREA.
  • Invoked from Admin Menu (CA00) option 5.

Batch Components

Job Name Utility / Program Function
CREADB21 DSNTIAD, DSNTEP4 One-time setup: creates the CardDemo DB2 database, tablespace, tables, indexes, and loads initial seed data from control files.
TRANEXTR DSNTIAUL (SQL unload) Nightly extract: backs up prior VSAM files to GDG (+1), deletes current files, then unloads TRANSACTION_TYPE and TRANSACTION_TYPE_CATEGORY from DB2 into fixed-format PS datasets (LRECL=60).

COBTUPDT – Batch Maintenance Program

Input record layout (60-byte fixed, LRECL=60):

Column  1     : Operation code  A=Add  U=Update  D=Delete  *=Comment
Columns 2-3   : TR_TYPE code (2-char alphanumeric)
Columns 4-53  : TR_DESCRIPTION (50-char alphanumeric)

The program reads the input file sequentially, evaluates the operation code, and performs the corresponding embedded SQL statement. Errors are displayed to SYSOUT and cause the program to ABEND via 9999-ABEND.

    TR_TYPE

    )

   || REPEAT('0',4)
                    AS CHAR(60))
FROM  CARDDEMO.TRANSACTION_TYPE_CATEGORY
ORDER BY
      TRC_TYPE_CODE
    , TRC_TYPE_CATEGORY;

6. Business Rules

BR-01: Referential Integrity – No Orphaned Categories

The TRANSACTION_TYPE_CATEGORY table has a FOREIGN KEY … ON DELETE RESTRICT constraint referencing TRANSACTION_TYPE.TR_TYPE. Deleting a transaction type code that still has associated categories will be rejected by DB2 with a non-zero SQLCODE. The CTLI screen surfaces this as a delete failure message.

BR-02: Dual-Source Synchronisation Requirement

Online changes (CTTU / CTLI) and batch changes (MNTTRDB2) update DB2 only. The TRANEXTR job must be executed to propagate those changes to the VSAM files read by the core application's COTRN screens. There is no real-time sync; the lag is one batch cycle.

BR-03: DB2 Cursor Discipline

Both COTRTLIC cursors (C-TR-TYPE-FORWARD and C-TR-TYPE-BACKWARD) must be explicitly OPENed before each FETCH loop and CLOSEd afterward. SQLCA must be checked after every EXEC SQL; SQLCODE 100 signals end-of-data (not an error). Any other non-zero SQLCODE must abort the operation and surface a formatted message via 9999-FORMAT-DB2-MESSAGE.

BR-04: Admin-Only Access

Both online transactions (CTTU and CTLI) set CDEMO-USRTYP-ADMIN in the COMMAREA on entry, meaning they are intended for administrator accounts only (e.g., ADMIN001). They are accessible from Admin Menu (CA00) options 5 and 6, which are not displayed to non-admin users.

BR-05: Input Validation Rules

  • TR_TYPE must be exactly 2 alphanumeric characters (validated via INSPECT against LIT-ALL-ALPHANUM-FROM-X).
  • TR_DESCRIPTION must be non-blank; maximum 50 characters.
  • In MNTTRDB2 input file: column 1 must be A, U, D, or *. Any other value triggers a 9999-ABEND.

BR-06: Concurrent Update Protection

COTRTUPC reads the current record into a work area before presenting it for update. On PF5 (commit), it re-reads the record and compares the stored copy. If the description has changed between the read and the commit (i.e., another user updated the record), the program rejects the update with message "Record changed by someone else. Please review".

BR-07: GDG Backup on TRANEXTR

Before overwriting the VSAM output files, TRANEXTR backs up the current version of TRANTYPE.PS to TRANTYPE.BKUP(+1) and TRANCATG.PS to TRANCATG.PS.BKUP(+1) using IEBGENER. This preserves one generation of history. Steps run conditionally on RC=0 from the previous step (COND=(0,NE)).

9. Development Guidelines

DB2 Embedded SQL Patterns

All SQL in this module uses static embedded SQL compiled at precompile time into a DBRM (Database Request Module). This is the recommended pattern for CICS COBOL programs because:

  • Access paths are determined at bind time, yielding predictable performance.
  • Static SQL avoids dynamic plan lookup overhead on each execution.
  • SQLCA is populated automatically after each EXEC SQL; always check SQLCODE.

Canonical SQLCA Check Pattern

       EXEC SQL
            SELECT TR_TYPE
                  ,TR_DESCRIPTION
              INTO :DCL-TR-TYPE
                  ,:DCL-TR-DESCRIPTION
              FROM CARDDEMO.TRANSACTION_TYPE
             WHERE TR_TYPE = :WS-INPUT-TTYPE
       END-EXEC

       MOVE SQLCODE TO WS-DISP-SQLCODE

       EVALUATE TRUE
           WHEN SQLCODE = ZERO
               SET FOUND-TRANTYPE-IN-TABLE TO TRUE
           WHEN SQLCODE = +100
               SET WS-RECORD-NOT-FOUND TO TRUE
           WHEN OTHER
               SET WS-DB2-ERROR TO TRUE
               MOVE 'SELECT TRAN TYPE: ' TO WS-DB2-CURRENT-ACTION
               PERFORM 9999-FORMAT-DB2-MESSAGE
                  THRU 9999-FORMAT-DB2-MESSAGE-EXIT
       END-EVALUATE

Forward Cursor Paging Pattern

       EXEC SQL
            DECLARE C-TR-TYPE-FORWARD CURSOR FOR
                SELECT TR_TYPE, TR_DESCRIPTION
                  FROM CARDDEMO.TRANSACTION_TYPE
                 WHERE TR_TYPE >= :WS-START-KEY
                   AND ((:WS-EDIT-TYPE-FLAG = '1'
                   AND   TR_TYPE = :WS-TYPE-CD-FILTER)
                    OR  (:WS-EDIT-TYPE-FLAG <> '1'))
                 ORDER BY TR_TYPE
       END-EXEC

       EXEC SQL OPEN  C-TR-TYPE-FORWARD END-EXEC
       PERFORM UNTIL WS-RECORDS-COUNT >= WS-MAX-SCREEN-LINES
                  OR SQLCODE = +100
           EXEC SQL FETCH C-TR-TYPE-FORWARD
                     INTO :DCL-TR-TYPE, :DCL-TR-DESCRIPTION
           END-EXEC
           IF SQLCODE = ZERO
               ADD 1 TO WS-RECORDS-COUNT
               ... populate screen array ...
           END-IF
       END-PERFORM
       EXEC SQL CLOSE C-TR-TYPE-FORWARD END-EXEC

Delete with RI Check Pattern

       EXEC SQL
            DELETE FROM CARDDEMO.TRANSACTION_TYPE
             WHERE TR_TYPE = :WS-DELETE-TYPE-CODE
       END-EXEC

       EVALUATE TRUE
           WHEN SQLCODE = ZERO
               SET FLG-DELETED-YES TO TRUE
           WHEN SQLCODE = -532        *> FK violation
               SET WS-RI-VIOLATION  TO TRUE
               ... display "Cannot delete; categories exist" ...
           WHEN OTHER
               SET WS-DB2-ERROR TO TRUE
               PERFORM 9999-FORMAT-DB2-MESSAGE ...
       END-EVALUATE

Coding Conventions

  • All programs use 88-level condition names for flag variables (e.g., INPUT-OK, FLG-DELETED-YES); set flags with SET … TO TRUE rather than MOVE literals.
  • Working-storage literals are grouped under 01 WS-CONSTANTS (for static strings) and 01 WS-MISC-STORAGE (for working variables).
  • Messages shown to users are defined as 88-level values on info/return message PIC X fields, making all user-facing text visible in one place.
  • DB2 common procedures (9998-PRIMING-QUERY, 9999-FORMAT-DB2-MESSAGE) are shared via copybooks CSDB2RWY (working-storage) and CSDB2RPY (procedure division), included with EXEC SQL INCLUDE.
  • CICS COMMAREA: each program copies the shared CARDDEMO-COMMAREA (from COCOM01Y) at the front, then appends a private commarea structure for program-specific state.

Adding a New Transaction Type Field

  1. Alter the DB2 table: ALTER TABLE CARDDEMO.TRANSACTION_TYPE ADD COLUMN … and update ddl/TRNTYPE.ddl.
  2. Regenerate the DCLGEN declarations and update dcl/DCLTRTYP.dcl.
  3. Update BMS map COTRTUP.bms and its copybook COTRTUP.cpy.
  4. Update COTRTUPC and COTRTLIC to bind/display the new field.
  5. Update the TRANEXTR SQL to include the new column in its CONCAT expression and adjust the output LRECL.
  6. Update COBTUPDT input record layout if batch maintenance is needed for the new field.
  7. Recompile all affected programs with the DB2 precompiler and re-bind the CARDDEMO plan.

10. Usage Examples

Example 1 – Adding a Transaction Type Online (CTTU)

1. Log in as ADMIN001 / PASSWORD
2. From CA00 main menu, press 6 (Transaction Type add/edit) → CTTU
3. Screen CTRTUPA displays a blank form.

   TRANSACTION TYPE CODE  : __
   DESCRIPTION            : __________________________________________________

   PF3=Exit  PF5=Confirm  PF12=Cancel

4. Type:  TR_TYPE = "PU"  DESCRIPTION = "Purchase"

11. Performance Considerations

OperationBudgetNotes
CTLI cursor open + 7 FETCHes < 2 seconds The unique index on TR_TYPE ensures fast range scans. Filter by description uses LIKE which may require a table scan if many rows exist.
CTTU single SELECT / INSERT / UPDATE < 1 second Single-row operations keyed on the primary key; always uses the unique index.
TRANEXTR full extract < 90 seconds DSNTIAUL unloads all rows in TR_TYPE order. Dataset size is bounded by the number of transaction types (typically small). GDG backup adds one sequential copy step.
MNTTRDB2 batch run Proportional to input file size Each input record is one SQL statement. No bulk-insert optimization; suitable for small maintenance files (<1000 records).
DB2 thread acquisition THREADLIMIT(1) The DB2ENTRY is configured with THREADLIMIT(1). In high-concurrency scenarios, consider increasing the thread limit or moving to a pool-based entry.

⚠️ Thread Limit Warning

The CSD definition sets THREADLIMIT(1) on the CARDDEMO DB2ENTRY. Under concurrent admin activity (e.g., two operators using CTLI simultaneously), one request will queue until the first releases its thread. For production environments, increase THREADLIMIT or use a shared pool entry.

12. Troubleshooting

SymptomLikely CauseResolution
CTLI / CTTU abend with SQLCODE -904 or -923 DB2 subsystem unavailable or CICS DB2 connection not established Verify DB2 subsystem DAZ1 is active; check CICS DB2ENTRY CARDDEMO is enabled; re-install CSD definitions if needed.
"Delete of record failed" on a type with no categories Stale VSAM file still references the type code (core application constraint) Run TRANEXTR to refresh VSAM files, then retry the delete after verifying the VSAM records are gone.
TRANEXTR fails at STEP40 / STEP50 DSNTIAUL plan not bound, or DB2 tables do not exist Run CREADB21 first; verify DSNTIAUL is bound using app/jcl/UTIL/BINDTIAD JCL.
MNTTRDB2 abends with "TYPE NOT VALID" Input file has a record where column 1 is not A/U/D/* Inspect the INPFILE for blank or corrupt records; correct the file and resubmit.
CTTU message "Record changed by someone else" Concurrent update; another user or batch job modified the record between the read and the commit Press Enter to re-read the current record; review, then press PF5 again to commit your changes.
Options 5/6 missing from Admin Menu CA00 CICS transactions CTLI/CTTU not defined, or programs not compiled Ensure CRDDEMOD.csd resources are installed and programs are compiled/linked into the LOADLIB.
COTRTLIC / COTRTUPC compile error on EXEC SQL DB2 precompiler not run before COBOL compile Use a two-step compile JCL: DSNHPC precompile first, then IGYCRCTL. Check DSNHPC return code ≤ 4.
