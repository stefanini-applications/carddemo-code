📈 Reports

User Story Development Guide — Transaction Reports and Statements

📋 Overview

Module ID: reports

Purpose: Provides online report request capability (CICS) that submits batch jobs for transaction detail reports and customer statements. The batch programs read TRANSACT VSAM and produce formatted output.

Business Context: Reports bridge the online and batch worlds — cardholder requests from CICS screen trigger job submission via Internal Reader; the report runs as a batch job and output goes to SYSOUT or GDG dataset. CREASTMT produces customer-facing statements.

🎯 User Story Templates

Request Transaction Report

As a cardholder, I want to request a transaction report for a date range so that I can reconcile my account activity for a specific period.
Acceptance Criteria:
  • Transaction CR00 (CORPT00C) presents report request screen with date range fields
  • User enters start date and end date in YYYY-MM-DD format
  • On submit, CICS submits TRANREPT JCL via Internal Reader
  • Confirmation message shown: "Report submitted successfully"
  • Invalid date format rejected with field-level error message
  • Report output available in system SYSOUT after job completion

Transaction Detail Report (Batch)

As a compliance officer, I want a full transaction detail report for all accounts in a date range so I can audit system activity.
Acceptance Criteria:
  • TRANREPT (CBTRN03C) reads TRANSACT VSAM sequentially
  • Report includes: TRAN-ID, TRAN-CARD-NUM (masked), TRAN-AMT, TRAN-TYPE-CD, TRAN-MERCHANT-NAME, TRAN-ORIG-TS
  • Transactions filtered by TRAN-ORIG-TS within requested date range
  • Report total line shows transaction count and total amount
  • CBTRN03C uses TRANREPT.prc procedure for standard JCL step

Customer Statement

As an operations manager, I want to generate monthly customer statements so that cardholders receive their billing summary.
Acceptance Criteria:
  • CREASTMT (CBSTM03A) runs in batch and produces formatted statement
  • Statement includes: account summary, opening/closing balance, transaction detail, payment due information
  • Output written to GDG dataset for archiving and distribution
  • One statement per account; accounts without activity still receive statement with balance only

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Add account number filter to report request screen
  • Add report format selection (summary vs detail)
  • Add transaction count to TRANREPT header

⚠️ Medium (3-5 pts)

  • Add category breakdown subtotals to report
  • Report request status inquiry (was job submitted/completed)
  • Parameterize report output destination (SYSOUT class)

🔴 Complex (5-8 pts)

  • Scheduled auto-generation of statements via scheduler
  • PDF output via TXT2PDF JCL utility integration
  • Exception report: accounts over credit limit or past due
