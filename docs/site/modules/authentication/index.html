🔐 Authentication

User Story Development Guide · CardDemo System

📋 Module Overview

Purpose: User login/logout and session security for all CICS transactions.

Business Context: Every bank operator must authenticate before accessing any card, account, or transaction function. Authentication validates credentials against the USRSEC VSAM dataset and establishes the session context (user type) used for access control throughout the session.

Transaction ID: CC00

Key Programs: COSGN00C (signon), COMEN01C (main menu)

Key BMS Mapsets: COSGN00 COMEN01

VSAM Dataset: USRSEC.VSAM.KSDS

🎯 User Story Development Guide

📝 US Templates for Authentication

Login Success:
As a bank operator, I want to log in with my User ID and password so that I can access authorized card management functions.
Login Failure Feedback:
As a bank operator, I want to see a clear error message when my credentials are invalid so that I know my login failed and can retry.
Session Exit:
As a bank operator, I want to exit the application by pressing PF3 so that my terminal session is properly closed.
Admin Routing:
As a system admin, I want to be routed to the Admin Menu after login so that I can access user management and system administration functions.

⚡ Development Acceleration Factors

  • COSGN00C: Fully implemented signon program — changes are typically additive (e.g., adding 2FA) rather than replacement
  • CSUSR01Y copybook: Reusable user record definition shared with user-management module
  • COMMAREA pattern: Established state-passing mechanism; any new session attribute can be added to the existing COMMAREA structure
  • RACF integration: OS-level security already integrated; application-level validation layers on top

✅ Acceptance Criteria Patterns

Successful Login:
  • Given valid User ID and password, when Enter is pressed, then the user is navigated to Main Menu (CM00)
  • User type (Admin/User) must be stored in COMMAREA for use by subsequent screens
Invalid Credentials:
  • Given invalid credentials, when Enter is pressed, then error message "INVALID CREDENTIALS" displayed in RED on line 22
  • Password field must be cleared; User ID field must remain populated
  • Screen must remain on CC00 (no navigation)
PF3 Exit:
  • Given any state of login screen, when PF3 is pressed, then the CICS session is terminated cleanly
Empty Fields:
  • Given blank User ID or Password, when Enter is pressed, then "USER ID REQUIRED" or "PASSWORD REQUIRED" message displayed

🔧 Technical Foundation

🏗️ Key Components

  • COSGN00C: Main signon CICS program — reads USRSEC VSAM, validates credentials, sets COMMAREA user type, routes to CM00
  • COSGN00 (BMS mapset): Login screen definition — User ID field (USRIDINI), password field (PASSWDINI, masked), error message (ERRMSGI)
  • CSUSR01Y (copybook): User security record layout (User ID, name, encrypted password, user type A/U)
  • COMEN01C: Main menu — reads user type from COMMAREA to display appropriate options

🔗 Dependencies

  • USRSEC.VSAM.KSDS: User security file — must be initialized via DUSRSECJ batch job before first use
  • RACF: OS-level resource security; COSGN00C validates at application layer on top of RACF
  • COMMAREA: Must be correctly initialized on CC00 entry; shared by all downstream transactions

🌐 Screen Interface

Transaction: CC00  Program: COSGN00C  Mapset: COSGN00

Fields:
  USRIDINI   - User ID input (PIC X(08), FSET)
  PASSWDINI  - Password input (PIC X(08), FSET, masked/DARK)
  ERRMSGI    - Error message output (ASKIP, color RED/GREEN)
  CURADATE   - Current date display (ASKIP)
  TRNNAMEI   - Transaction name display (ASKIP)

Navigation:
  Enter → Validate credentials → CM00 (success) or redisplay (fail)
  PF3   → EXEC CICS RETURN (terminate session)
