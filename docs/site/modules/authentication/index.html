🔐 Authentication

User Story Development Guide — CardDemo Mainframe Application

📋 Overview

Module ID: authentication

Purpose: User signon, credential validation against VSAM USRSEC file, and role-based routing to user main menu (CM00) or admin menu (CA00).

Business Context: Entry point for all CardDemo users. Authenticates both regular users (card management) and admin users (user management). Uses RACF for environment-level security; USRSEC VSAM for application-level credentials.

CICS Transaction: CC00 Program: COSGN00C BMS Map: COSGN00 Data: USRSEC VSAM KSDS

🎯 User Story Development Guide

📝 US Templates

  • As a user, I want to log in with my user ID and password so that I can access the CardDemo application securely
  • As an admin, I want invalid login attempts to display a clear error message so that I know when I've entered incorrect credentials
  • As an admin user, I want to be routed to the admin menu after login so that I can manage users and system settings
  • As a regular user, I want to be routed to the main menu after login so that I can manage my credit cards and transactions

⚡ Development Acceleration Factors

  • COSGN00C: Complete working signon implementation available as reference
  • COCOM01Y: Common communication area copybook shared across all CICS programs — reuse for COMMAREA definitions
  • CSUSR01Y: User record copybook with 80-byte layout — reuse for any user data access
  • Pseudo-conversational pattern: Check EIBCALEN = 0 for first invocation; return via EXEC CICS RETURN TRANSID COMMAREA

📊 Complexity Guidelines

  • Simple (1-2 pts): Display signon screen, read user record from VSAM
  • Medium (3-5 pts): Validate credentials, check user type, route to appropriate menu with COMMAREA
  • Complex (5-8 pts): Add login attempt tracking, session timeout, or RACF integration changes

🔗 Dependencies

  • USRSEC VSAM file: Must be open in CICS and initialized (DUSRSECJ batch job)
  • CICS COMMAREA: Passes user context to downstream programs (COMEN01C, COADM01C)
  • RACF: Environment-level security must be configured

📋 Data Model

01  SEC-USER-DATA.
    05  SEC-USR-ID    PIC X(08).  -- VSAM key
    05  SEC-USR-FNAME PIC X(20).
    05  SEC-USR-LNAME PIC X(20).
    05  SEC-USR-PWD   PIC X(08).
    05  SEC-USR-TYPE  PIC X(01).  -- 'U'=user, 'A'=admin
    05  SEC-USR-FILLER PIC X(23).

⚙️ Business Rules

  • User ID and password validated against USRSEC VSAM KSDS via EXEC CICS READ
  • Response code NOTFND or password mismatch → error message, re-display signon
  • SEC-USR-TYPE = 'U' → route to CM00 (COMEN01C) with COMMAREA
  • SEC-USR-TYPE = 'A' → route to CA00 (COADM01C) with COMMAREA
  • No account lockout in base version; each invalid attempt shows error immediately
  • Default test credentials: ADMIN001/PASSWORD (admin), USER0001/PASSWORD (regular user)

✅ Acceptance Criteria Patterns

  • Given a valid user ID and password, when the user submits the signon screen, then they are routed to the appropriate menu (CM00 or CA00) based on their user type
  • Given an invalid password, when the user submits signon, then an error message is displayed and the signon screen is re-presented
  • Given a non-existent user ID, when the user submits signon, then an appropriate "user not found" error message is displayed
  • Given blank user ID or password fields, when the user submits signon, then a validation error is displayed before any VSAM access is attempted
