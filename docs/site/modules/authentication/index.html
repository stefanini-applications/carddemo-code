🔐 Authentication

User Story Development Guide — Signon & Session Management

📋 Overview

Module ID: authentication

Purpose: Controls access to the CardDemo application through credential validation and session initialization. Determines user type (Regular vs Admin) and routes to the appropriate menu.

Business Context: All CardDemo users must authenticate before accessing any function. This is the security gateway for the entire application, enforced via RACF at the OS level and USRSEC VSAM at the application level.

Entry Point: CICS transaction CC00 → program COSGN00C

🎯 User Story Templates

Cardholder Login

As a cardholder, I want to sign into CardDemo with my user ID and password so that I can manage my credit card accounts.
Acceptance Criteria:
  • Given valid credentials (USER0001/PASSWORD), when I submit the signon screen (CC00), then I am taken to the Main Menu (CM00)
  • Given invalid credentials, when I submit, then the signon screen redisplays with an error message and no navigation occurs
  • Given a valid admin credential (ADMIN001/PASSWORD), when I submit, then I am taken to the Admin Menu (CA00)
  • Credentials are validated against USRSEC VSAM file using SEC-USR-ID as key

Admin Login

As an administrator, I want to sign in with my admin credentials so that I can access user management and administrative functions.
Acceptance Criteria:
  • Given admin user (SEC-USR-TYPE = 'A'), when login succeeds, then route to Admin Menu (CA00)
  • Given regular user (SEC-USR-TYPE = 'U'), when login succeeds, then route to Main Menu (CM00)
  • Admin-only transactions are unavailable to regular users

Session State

As the system, I want to maintain user session state across CICS pseudo-conversational transactions so that user context is preserved without holding CICS resources.
Acceptance Criteria:
  • User ID and type stored in CICS COMMAREA (COCOM01Y copybook)
  • Each CICS transaction receives and passes COMMAREA to next transaction
  • Session is terminated when user exits (PF3/CLEAR) back to signon

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Display login screen with existing BMS map
  • Modify error message text
  • Add password masking attribute to COSGN00 BMS map

⚠️ Medium (3-5 pts)

  • Add login attempt counter to USRSEC record
  • Add account lockout after N failed attempts
  • Add audit trail for successful/failed logins

🔴 Complex (5-8 pts)

  • Integrate with external LDAP/RACF for credential delegation
  • Implement password expiry with forced reset flow
  • Multi-factor authentication via MQ message

Data Structure – SEC-USER-DATA (CSUSR01Y)

01 SEC-USER-DATA.
   05 SEC-USR-ID       PIC X(08)  -- Login ID (VSAM key)
   05 SEC-USR-FNAME    PIC X(20)  -- First name
   05 SEC-USR-LNAME    PIC X(20)  -- Last name
   05 SEC-USR-PWD      PIC X(08)  -- Password (plain text)
   05 SEC-USR-TYPE     PIC X(01)  -- 'U'=Regular, 'A'=Admin
   05 SEC-USR-FILLER   PIC X(23)

Dependencies

  • USRSEC VSAM: Must be initialized via DUSRSECJ before first login
  • RACF: OS-level security — user must have RACF access to the CICS region
  • COCOM01Y: All downstream CICS programs depend on this COMMAREA structure

⚡ Development Acceleration Factors

  • COMEN01C pattern: All menu programs use the same COCOM01Y COMMAREA pattern — reuse for any new navigation screen
  • CSUSR01Y copybook: Pre-defined user record — no new data structure needed for user credential changes
  • BMS map pattern: Follow COSGN00 map structure for any new screen additions
  • Default users: ADMIN001/PASSWORD and USER0001/PASSWORD available for testing without RACF setup
