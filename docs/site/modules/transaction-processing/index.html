💰 Transaction Processing

User Story Development Guide — Online Entry, View & Batch Posting

← Back to Documentation Index

Module ID: transaction-processing

Purpose: Provides online viewing and entry of transactions (CT00–CT02) and batch daily transaction posting (CBTRN01C/02C) that updates account balances, category balances, and writes rejects for invalid records.

🎯 User Story Development Guide

📝 US Templates

  • List Transactions: As a card holder, I want to view a paginated list of my recent transactions so that I can review my spending history.
  • View Transaction: As a card holder, I want to view transaction details so that I can verify merchant name, amount, and timestamp.
  • Add Transaction: As a card holder, I want to add a transaction so that I can record a purchase against my card.
  • Filter Transactions: As a card holder, I want to filter transactions by date range so that I can find transactions from a specific period.
  • Batch Post: As a batch system, I want to post all daily transactions so that account balances are updated with the day's activity.
  • Reject Invalid: As a batch system, I want to write invalid transactions to a rejects file so that errors can be reviewed and reprocessed.

⚡ Development Acceleration Factors

  • CVTRA05Y copybook: Complete transaction record layout shared by online and batch programs
  • COTRN00C pagination: Forward-page navigation already implemented; extend for backward paging
  • CBTRN01C validation logic: Card, account, and cross-reference validation reusable for new transaction types
  • TRANCATG / TRANTYPE reference files: Pre-loaded; lookup pattern established in existing programs

🔧 Technical Foundation

🏗️ Key Components

  • COTRN00C — Transaction list with forward pagination; reads TRANSACT VSAM sequentially from a start key
  • COTRN01C — Transaction detail view; direct KSDS read by TRAN-ID
  • COTRN02C — Transaction add; generates TRAN-ID, writes to TRANSACT KSDS
  • CBTRN01C (batch) — Reads DALYTRAN; validates vs CARDFILE, CUSTFILE, XREFFILE, ACCTFILE; posts to TRANSACT + updates ACCTDAT + TCATBALF; rejects to DALYREJS
  • CBTRN02C (batch) — Alternate transaction posting with different validation path
  • CBTRN03C (batch) — Transaction file maintenance / combine operations

🔗 Dependencies

  • TRANSACT (KSDS): Online transaction file; primary store for CT-series programs
  • DALYTRAN (Sequential): Daily transaction input for batch posting
  • DALYREJS (Sequential): Batch reject output file
  • ACCTDAT (KSDS): Account balances updated during batch posting
  • TCATBALF (KSDS): Transaction category balances updated during batch posting
  • TRANCATG / TRANTYPE (KSDS): Reference data for category/type descriptions
  • DISCGRP (KSDS): Disclosure groups used for interest rate calculation
  • Card Management module: CARDXREF required to resolve card → account

🌐 CICS Transactions

CT00  → COTRN00C  — Transaction list (paginated, forward)
CT01  → COTRN01C  — Transaction detail view
CT02  → COTRN02C  — Add new transaction

📋 Transaction Data Model (CVTRA05Y, 350 bytes)

TRAN-RECORD:
  TRAN-ID           PIC X(16)       — Transaction ID (KSDS key)
  TRAN-TYPE-CD      PIC X(02)       — Transaction type (e.g., 'PU'=Purchase)
  TRAN-CAT-CD       PIC 9(04)       — Category code (e.g., 0001=Retail)
  TRAN-SOURCE       PIC X(10)       — Source system identifier
  TRAN-DESC         PIC X(100)      — Transaction description
  TRAN-AMT          PIC S9(09)V99   — Amount (+debit, -credit)
  TRAN-MERCHANT-ID  PIC 9(09)       — Merchant identifier
  TRAN-MERCHANT-NAME PIC X(50)      — Merchant name
  TRAN-MERCHANT-CITY PIC X(50)      — Merchant city
  TRAN-MERCHANT-ZIP  PIC X(10)      — Merchant ZIP
  TRAN-CARD-NUM     PIC X(16)       — Card used
  TRAN-ORIG-TS      PIC X(26)       — Origination timestamp
  TRAN-PROC-TS      PIC X(26)       — Processing timestamp

📋 Business Rules

  • BR-TRAN-01: Every transaction must have a unique 16-char TRAN-ID
  • BR-TRAN-02: TRAN-TYPE-CD (2 chars) must exist in TRANTYPE reference file
  • BR-TRAN-03: TRAN-CAT-CD must exist in TRANCATG for the given type
  • BR-TRAN-04: Transactions that fail validation during batch posting go to DALYREJS
  • BR-TRAN-05: TCATBALF is updated per category for running balance totals
  • BR-TRAN-06: Transaction amounts are signed: positive = debit (charge), negative = credit (payment)

✅ Acceptance Criteria Examples

  • List Display: Given an authenticated user, when CT00 is accessed, then transactions for the user's account are displayed in reverse chronological order with paging
  • Add Transaction: Given a valid card number, type, category, and amount, when CT02 is submitted, then the transaction is written to TRANSACT with a generated TRAN-ID and displayed in the list
  • Batch Rejection: Given a transaction with an unknown card number in DALYTRAN, when POSTTRAN runs, then the record is written to DALYREJS with an error code
  • Balance Update: Given POSTTRAN completes, when account balance is viewed, then ACCT-CURR-BAL reflects all posted transactions
  • Performance: Transaction list must display first page within 3 seconds; batch POSTTRAN must complete within the nightly batch window
