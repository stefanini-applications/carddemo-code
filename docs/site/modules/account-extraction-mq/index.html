📡 Account Extraction MQ

User Story Development Guide — CardDemo Optional Module

⚠️ Optional Module — Requires IBM MQ and CICS-MQ connection configured.

📋 Overview

Module ID: account-extraction-mq

Purpose: Demonstrates asynchronous integration patterns using MQ — system date inquiry (CDRD) and account details inquiry (CDRA) via request/response message queuing.

Business Context: Shows how external systems can query mainframe data asynchronously via MQ without direct VSAM access. Pattern commonly used for distributed application integration and service extraction during modernization.

CDRD: CODATE01 (date inquiry) CDRA: COACCT01 (account inquiry) MQ Request/Response pattern

🎯 User Story Development Guide

📝 US Templates

  • As an external system, I want to query the mainframe system date via MQ so that I can synchronize timestamps without direct system access
  • As an external system, I want to query account details via MQ so that I can retrieve account data for downstream processing
  • As a developer, I want MQ request/response correlation to work reliably so that responses are matched to the correct requesting system
  • As a system operator, I want MQ queues monitored for depth so that unprocessed messages are detected promptly

⚡ Development Acceleration Factors

  • CODATE01/COACCT01: Complete MQ request/response implementations available as reference
  • Message correlation: REQUEST-ID field in message format used for request/response matching
  • MQ queue definitions: CARDDEMO.REQUEST.QUEUE and CARDDEMO.RESPONSE.QUEUE patterns established
  • CICS-MQ resources: MQCONN and MQQUEUE definitions available in CRDDEMOM.csd

📊 Complexity Guidelines

  • Simple (1-2 pts): Date inquiry via MQ (CDRD) — simple request/response
  • Medium (3-5 pts): Account data inquiry (CDRA) — VSAM read + MQ response packaging
  • Complex (5-8 pts): Error handling for MQ unavailable, timeout scenarios, message dead-letter queue handling

�� MQ Message Formats

-- Date Request/Response:
01 DATE-REQUEST-MSG.
   05 REQUEST-TYPE  PIC X(4) VALUE 'DATE'.
   05 REQUEST-ID    PIC X(8).

01 DATE-RESPONSE-MSG.
   05 RESPONSE-TYPE PIC X(4) VALUE 'DATE'.
   05 RESPONSE-ID   PIC X(8).
   05 SYSTEM-DATE   PIC X(10).  -- YYYY-MM-DD

-- Account Request/Response:
01 ACCT-REQUEST-MSG.
   05 REQUEST-TYPE  PIC X(4) VALUE 'ACCT'.
   05 REQUEST-ID    PIC X(8).
   05 ACCOUNT-NUMBER PIC X(11).

01 ACCT-RESPONSE-MSG.
   05 RESPONSE-TYPE PIC X(4) VALUE 'ACCT'.
   05 RESPONSE-ID   PIC X(8).
   05 ACCOUNT-DATA  PIC X(300). -- Full ACCOUNT-RECORD

⚙️ Business Rules

  • REQUEST-ID in request message must be echoed in RESPONSE-ID for correlation
  • CDRD reads system date from CICS (EXEC CICS ASKTIME / FORMATTIME)
  • CDRA reads ACCTFILE by account number from request message; NOTFND returns error response
  • Response sent to reply-to queue specified in MQ MQMD header
  • MQ queues defined: CARDDEMO.REQUEST.QUEUE (requests in), CARDDEMO.RESPONSE.QUEUE (responses out)

✅ Acceptance Criteria Patterns

  • Given a date request on CARDDEMO.REQUEST.QUEUE, when CDRD processes it, then a response with current system date is placed on the reply queue with matching REQUEST-ID
  • Given a valid account number in the account request, when CDRA processes it, then full account data (300 bytes) is returned in the response message
  • Given an account number not in ACCTFILE, when CDRA processes the request, then an error response with "Account not found" indication is sent to the reply queue
  • Given MQ is unavailable, when CDRD or CDRA is invoked, then appropriate CICS error handling returns a non-zero condition code
