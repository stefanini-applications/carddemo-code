📦 Authorization Extension

Guide for Developing User Stories

📋 Overview

Purpose: Adds an IMS/DB2/MQ-based authorization simulator that processes merchant requests, logs decisions, and surfaces fraud markings.

Business Context: Ideal for teams validating MQ-based credit card authorizations, two-phase commit behavior, and fraud reporting in modernization exercises.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • MQ request/response: As a payment gateway integrator, I want to send `AUTH-CARD` requests to `AWS.M2.CARDDEMO.PAUTH.REQUEST` and receive approval codes so that downstream systems can finish the sale.
  • Fraud tagging: As a fraud analyst, I want to flag a suspicious authorization from `CPVD` and push it into `AUTHFRDS` for reporting.
  • Data cleanup: As an operator, I want to run `CBPAUP0J` after hours to purge expired authorizations and reset credit buffers.

⚡ Development Acceleration Factors

  • MQ payload templates: Copy the request/response structures defined in `app/app-authorization-ims-db2-mq/README.md` to keep correlation IDs consistent.
  • IMS/DB2 scaffolding: Existing DBDs (`DBPAUTP0`, `DBPAUTX0`) and PSBs (`PSBPAUTB`, `PSBPAUTL`) provide a ready-made schema for new segments.
  • Shared purge job: `CBPAUP0J` already exists and can be extended if new cleanup logic is required.

📊 Complexity Guidelines

  • Simple (1-2 pts): Update a BMS map (`COPAU00`, `COPAU01`) to surface a new field, provided MQ payload remains unchanged.
  • Medium (3-5 pts): Adjust IMS segments or DB2 table columns and ensure `COPAUA0C`/`COPAUS2C` handle the extra fields.
  • Complex (5-8 pts): Introduce a new MQ queue or call into an external fraud engine, requiring message transformation and CICS/MQ resource definitions.

✅ Acceptance Criteria Examples

  • `CP00` reads an MQ `AUTH-DATE` request, applies business rules, and writes IMS `PAUTSUM0`/`PAUTDTL1` segments with no abends.
  • `CPVD` marks fraud (PF5) and ensures `AUTHFRDS` has a corresponding row with `FRAUD_RPT_DATE` and `MATCH_STATUS = 'F'`.
  • `CBPAUP0J` purges all authorizations older than the current system date and releases MQ queue resources.

🔧 Technical Foundation

🏗️ Key Components

  • CICS programs: `COPAUA0C`, `COPAUS0C`, `COPAUS1C`, `COPAUS2C`, `CBPAUP0C` orchestrate MQ inflight requests, summary/detail browsing, fraud marking, and batch purge.
  • IMS artifacts: DBDs `DBPAUTP0`, `DBPAUTX0`; segments `PAUTSUM0`, `PAUTDTL1`; PSBs `PSBPAUTB`, `PSBPAUTL`.
  • DB2 schema: Table `AUTHFRDS` plus index `XAUTHFRD`; used for analytics on fraud-marked cases.
  • MQ queues: `AWS.M2.CARDDEMO.PAUTH.REQUEST`, `AWS.M2.CARDDEMO.PAUTH.REPLY` with comma-separated payloads (authorization, approval, reason).
  • BMS screens: `COPAU00` (summary), `COPAU01` (details) which reuse PF key conventions.

🔗 Dependencies

  • Base CardDemo (`carddemo-core`) for account/card detail lookups.
  • IMS DB to persist hierarchical authorization data.
  • DB2 subsystem for fraud reporting.
  • IBM MQ for request/response communication.

🌐 Public Interfaces


CICS:
- CP00 / COPAUA0C  -> Authorization request processor (MQ trigger)
- CPVS / COPAUS0C  -> Authorization summary screen (scroll with PF7/PF8)
- CPVD / COPAUS1C  -> Authorization detail and fraud tagging (PF5)
Batch:
- CBPAUP0J / CBPAUP0C -> Purge expired authorizations
MQ:
- AWS.M2.CARDDEMO.PAUTH.REQUEST -> Accepts AUTH-DATE, CARD-NUM, etc.
- AWS.M2.CARDDEMO.PAUTH.REPLY   -> Sends CARD-NUM, TRANSACTION-ID, AUTH-RESP-CODE, APPROVED-AMT
