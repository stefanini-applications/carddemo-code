1) Module Overview and Purpose

The authorization module provides real-time authorization processing for card transactions. Requests arrive through MQ, are evaluated against account availability, and are recorded as pending authorizations in IMS. Operators can review and drill into pending items through CICS/BMS transactions. Fraud status can be toggled from the detail screen and is persisted in DB2.

The module exists as an optional extension to the CardDemo core and focuses on modernization patterns: queue-driven orchestration, multi-store persistence (IMS + DB2), and supportable batch retention.

Main transaction IDs: CP00 (intake), CPVS (summary), CPVD (detail).

Primary User Outcomes

  • Accept external authorization requests over MQ.
  • Return near-real-time approval/decline response data.
  • Retain pending authorization state for review and reconciliation.
  • Track confirmed/removed fraud in DB2 for analytics/audit workflows.
  • Purge stale pending data to keep IMS datasets lean.

2) Architectural Context

High-Level Flow

  1. An MQ-compatible producer sends a comma-separated authorization request message.
  2. COPAUA0C reads the message via MQGET and extracts fields with UNSTRING.
  3. Program reads card/account/customer context from VSAM cross-reference and master files.
  4. Decision logic computes available credit and builds response fields.
  5. Program sends response through MQPUT1 using inbound ReplyToQ and saved CorrelId.
  6. Program updates IMS summary segment and inserts IMS detail segment.
  7. Operators inspect entries in CPVS/CPVD; PF5 in detail links to COPAUS2C for DB2 fraud write/update.
  8. Batch job CBPAUP0J runs CBPAUP0C to remove expired pending records.

Subsystem Placement

CICS Online

Programs: COPAUA0C, COPAUS0C, COPAUS1C, COPAUS2C.

Mapsets: COPAU00, COPAU01.

IMS Data

DBD: DBPAUTP0 (HIDAM primary), DBPAUTX0 (index).

Segments: PAUTSUM0 root + PAUTDTL1 child.

DB2 Fraud

Table: CARDDEMO.AUTHFRDS.

Index: CARDDEMO.XAUTHFRD.

Batch

Job: CBPAUP0J.

Program: CBPAUP0C in BMP mode (DFSRRC00).

Internal Navigation Model

  • CPVS page can XCTL to COPAUS1C when selection flag is S.
  • CPVD PF3 returns to summary via COMMAREA handoff.
  • CPVD PF8 steps to next detail record by IMS GNP.
  • CPVD PF5 toggles fraud status and links to DB2 writer.

  • ACCOUNT-RECORD provides ACCT-CURR-BAL, ACCT-CREDIT-LIMIT, ACCT-ACTIVE-STATUS.

7) Business Rules

7.1 Authorization Decision

  • Default starts as approval path (SET APPROVE-AUTH TO TRUE).
  • If summary exists, available = PA-CREDIT-LIMIT - PA-CREDIT-BALANCE.
  • If summary missing but account exists, available = ACCT-CREDIT-LIMIT - ACCT-CURR-BAL.
  • If amount exceeds available, decline with AUTH-RESP-CODE = '05' and reason 4100.
  • If xref/account/customer lookup fails, decline reason maps to 3100.
  • On approval, response code 00 and approved amount equals transaction amount.

7.2 IMS Persistence

  • Summary segment inserted on first auth per account or replaced on subsequent auths.
  • Detail segment inserted for every processed request with inverse-date/time key.
  • Match status set to pending (P) for approval or declined (D) for decline.

7.3 Fraud Toggle Behavior

  • PF5 in CPVD toggles PA-AUTH-FRAUD between confirmed (F) and removed (R).
  • COPAUS1C sends fraud payload via CICS LINK to COPAUS2C.
  • COPAUS2C INSERTs DB2 row; duplicate key triggers UPDATE fallback.
  • On DB2 success, CPVD updates IMS detail and takes SYNCPOINT; else rollback.

7.4 Purge and Retention

  • Batch computes original auth day from inverse date key and compares to current day.
  • If day difference >= configured expiry days, detail record qualifies for delete.
  • Summary counters/amounts are decremented when child details are deleted.
  • If no counts remain, summary record is deleted.
  • Checkpoint frequency is configurable through SYSIN controls.
Purge compares YYDDD values arithmetically. Cross-year behavior should be validated in production schedules.

9) Development Guidelines and Patterns

9.1 Message Parsing and Compatibility

  • Preserve CSV field order unless all producers and consumers are upgraded in lockstep.
  • Prefer additive fields at end of payload with guarded parsing strategy when extending.
  • If mandatory validation is tightened, define explicit reject codes and caller contract updates.

9.2 IMS Update Discipline

  • Schedule PSB once per transactional unit and release/term appropriately.
  • Handle PSB-SCHEDULED-MORE-THAN-ONCE branch consistently (existing pattern present).
  • Keep summary counters synchronized with detail insert/delete paths.

9.3 Fraud Path Consistency

  • DB2 write success should always align with IMS fraud marker update in same user action.
  • Retain rollback on DB2 failure to avoid UI showing fraud state not present in DB2.
  • If adding new fraud actions, extend action code domain beyond F/R deliberately.

9.4 UI Patterns

  • Summary list supports exactly five visible rows per page in current map definition.
  • Selection semantics are strict: valid select indicator is S or s.
  • PF key instructions are embedded in BMS map text; update map and program behavior together.

9.5 Error Logging

  • CP00 logs structured errors to TD queue CSSL via WRITEQ TD.
  • Error severity flags distinguish warning vs critical and can trigger early return.
  • When adding new failures, include event key and reason/resp codes for supportability.

   INSERT INTO CARDDEMO.AUTHFRDS (...)
   VALUES (...)
END-EXEC
IF SQLCODE = ZERO
   SET WS-FRD-UPDT-SUCCESS TO TRUE
ELSE
   IF SQLCODE = -803
      PERFORM FRAUD-UPDATE
   ELSE
      SET WS-FRD-UPDT-FAILED TO TRUE
   END-IF
END-IF

10.5 Purge Qualification Rule (Batch)

COMPUTE WS-AUTH-DATE = 99999 - PA-AUTH-DATE-9C
COMPUTE WS-DAY-DIFF = CURRENT-YYDDD - WS-AUTH-DATE
IF WS-DAY-DIFF >= WS-EXPIRY-DAYS
   SET QUALIFIED-FOR-DELETE TO TRUE
END-IF

10.6 Typical Operator Journey

  1. Run transaction CPVS.
  2. Enter account ID and press ENTER.
  3. Review five-row summary list and navigate PF7/PF8 if needed.
  4. Type S against one row and press ENTER to open CPVD.
  5. Press PF5 to mark/remove fraud or PF8 to inspect next auth.
  6. Press PF3 to return to summary list.

11) Operations and Support Notes

11.1 Runtime Signals

  • CP00 tracks MQ and IMS errors with location codes (M001, I00x, etc.).
  • Critical errors can trigger immediate terminate and return.
  • Batch prints processed/deleted summary and detail counters.

11.2 Operational Risks

  • Queue payload contract drift can silently degrade decision quality.
  • Missing DB2TRAN binding for CPVD breaks fraud persistence.
  • Large purge window can retain stale records and increase browse cost.
  • Aggressive purge window can remove records needed for operational review.

11.3 Recovery Considerations

  • CPVD fraud update path uses SYNCPOINT and rollback control.
  • Batch job uses IMS checkpoints to improve restartability.
  • CP00 response queue send failure is logged as critical MQ error.

12) Validation Checklist

  • Verify CP00 receives request and produces correlated reply.
  • Validate approve/decline transitions across at least 3 amount scenarios.
  • Confirm PAUTSUM0/PAUTDTL1 are inserted/updated as expected.
  • Confirm CPVS paging and selection behavior across multiple pages.
  • Confirm CPVD PF5 updates DB2 and IMS consistently.
  • Confirm duplicate fraud insert path uses update fallback (SQLCODE -803).
  • Run CBPAUP0J in test mode and verify deletion + checkpoint counts.
  • Verify no regressions to core account lookup dependencies.
