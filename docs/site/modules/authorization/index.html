🛡️ Authorization (IMS/DB2/MQ)

User Story Development Guide — CardDemo Optional Module

⚠️ Optional Module — Requires IMS DB, DB2, and MQ infrastructure in addition to the base CardDemo application.

📋 Overview

Module ID: authorization

Purpose: Real-time credit card authorization processing via MQ, with IMS hierarchical database for authorization storage and DB2 for fraud analytics. Simulates end-to-end authorization flow from POS request to approval/decline.

Business Context: Extends CardDemo with real-world authorization patterns: incoming MQ request → CICS processing → IMS DB insert → DB2 fraud logging → MQ response. Includes manual fraud flagging and batch purge of expired authorizations.

CPVS: COPAUS0C (summary) CPVD: COPAUS1C (detail) CP00: COPAUA0C (MQ processor) IMS HIDAM + DB2 + MQ

🎯 User Story Development Guide

📝 US Templates

  • As a regular user, I want to view pending authorizations on my account so that I can track unmatched charges
  • As a regular user, I want to see authorization details (merchant, amount, status) so that I can verify a specific authorization
  • As a regular user, I want to flag a suspicious authorization as fraudulent so that it can be investigated
  • As a system, I want authorization requests processed in real time via MQ so that merchants receive immediate approve/decline responses
  • As a system operator, I want expired authorizations purged daily so that available credit is restored for customers

⚡ Development Acceleration Factors

  • COPAUA0C: Complete MQ trigger processor — MQ GET → business logic → IMS insert → MQ PUT response
  • IMS DBD definitions: DBPAUTP0 (HIDAM primary) and DBPAUTX0 (HIDAM index) pre-defined in app/app-authorization-ims-db2-mq/ims/
  • DB2 DDL: AUTHFRDS table and index pre-defined in ddl/AUTHFRDS.ddl for fraud storage
  • MQ queues: CARDDEMO.AUTH.REQUEST.QUEUE and CARDDEMO.AUTH.RESPONSE.QUEUE patterns established
  • Copybooks CCPAURQY/CCPAURLY: MQ message request/response structures defined

📊 Complexity Guidelines

  • Simple (1-2 pts): Authorization summary view (CPVS) — read and display IMS records
  • Medium (3-5 pts): Authorization detail with fraud flag update (CPVD) — IMS update + DB2 insert
  • Complex (5-8 pts): MQ request processor with business rules and two-phase IMS+DB2 commit (CP00)
  • Very Complex (8-13 pts): Batch purge with credit restoration across IMS + VSAM (CBPAUP0C/CBPAUP0J)

⚙️ Authorization Flow

External POS → MQ Request Queue
    → COPAUA0C (CICS MQ trigger)
    → Validate: VSAM XREFFILE + ACCTFILE
    → Apply business rules (credit check, account status)
    → INSERT IMS HIDAM (authorization record)
    → MQ Response Queue (APPROVE/DECLINE)

User reviews pending auths (COPAUS0C/COPAUS1C):
    → Read IMS authorization records
    → Flag fraud → INSERT DB2 AUTHFRDS + UPDATE IMS

Batch purge (CBPAUP0C daily):
    → READ IMS expired records
    → UPDATE ACCTFILE (restore available credit)
    → DELETE IMS records

⚙️ Business Rules

  • Authorization approved if: account active + available credit ≥ request amount
  • Authorization response sent to reply-to queue specified in MQ request message
  • IMS uses HIDAM structure with PSBPAUTB PSB for DL/I calls
  • Fraud flag set via COPAUS1C → fraud record inserted in DB2 AUTHFRDS table
  • Expired authorizations purged by CBPAUP0C; available credit restored to account
  • Two-phase commit required for IMS + DB2 simultaneous updates

✅ Acceptance Criteria Patterns

  • Given a valid MQ authorization request, when COPAUA0C processes it, then authorization is stored in IMS and response sent to reply queue within SLA
  • Given insufficient available credit, when authorization request is processed, then DECLINE response is sent and no IMS record is created
  • Given a pending authorization, when user flags it as fraud via CPVD, then IMS record is updated and DB2 AUTHFRDS record is inserted
  • Given expired authorizations exist in IMS, when CBPAUP0C runs, then expired records are deleted and corresponding account available credit is restored
