🛡️ Authorization Optional Module

User Story Development Guide — Real-Time Credit Card Authorization with IMS, DB2, and MQ

📋 Overview

Module ID: authorization

Purpose: Provides real-time credit card authorization processing via IBM MQ. Authorization requests from external POS/merchant systems are received via MQ, processed by CICS, stored in IMS DB, and fraud analytics tracked in DB2.

Business Context: This optional module simulates the authorization leg of credit card payment processing. It demonstrates multi-technology integration (CICS + IMS + DB2 + MQ) and is designed for testing modernization scenarios involving authorization services.

📦 Prerequisites: Requires base CardDemo installed plus: IBM MQ configured and accessible from CICS, IMS DB setup and loaded, DB2 database with fraud analytics tables created via CREADB21/DEFGDGD JCL.

🎯 User Story Templates

Process Authorization Request

As a merchant POS system, I want to send an authorization request via MQ so that a credit card transaction can be approved or declined before processing.
Acceptance Criteria:
  • MQ-compatible client sends authorization request to configured MQ request queue
  • CP00 (COPAUA0C) triggered by MQ message arrival
  • CICS validates card number via CARDXREF VSAM → finds account → checks credit limit
  • Business rules applied: amount ≤ available credit, account active, no fraud flags
  • Authorization record stored in IMS DB hierarchical structure
  • Approve/Decline response written to MQ reply queue within SLA

View Pending Authorizations

As a cardholder, I want to view my pending authorizations so that I can see charges that are approved but not yet posted.
Acceptance Criteria:
  • Transaction CPVS (COPAUS0C) displays summary of pending authorizations from IMS DB
  • List shows: authorization ID, amount, merchant, timestamp, status
  • Cardholder can select authorization to view details (CPVD)

View Authorization Details

As a fraud analyst, I want to view and process authorization details so that I can investigate suspicious activity and update authorization status.
Acceptance Criteria:
  • Transaction CPVD (COPAUS1C) shows full authorization record from IMS DB
  • Analyst can update authorization status; change written back to IMS
  • Fraud-related details inserted to DB2 fraud analytics table
  • Status changes logged with timestamp and user ID

Purge Expired Authorizations

As a batch operations analyst, I want to run the authorization purge job so that expired authorization records are removed from IMS DB.
Acceptance Criteria:
  • CBPAUP0J (CBPAUP0C) reads IMS DB for authorizations past expiry threshold
  • Expired records deleted from IMS DB
  • Purge count written to SYSOUT for audit trail
  • RC=0 for clean run; RC=4 if some records could not be purged (logged)

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Add authorization count display to CPVS summary screen
  • Add status filter to pending authorization list
  • Format authorization amount display

⚠️ Medium (3-5 pts)

  • Add new fraud rule to authorization business logic
  • Add authorization expiry configuration parameter
  • Add secondary DB2 analytics query to CPVD details screen

🔴 Complex (5-8 pts)

  • Add authorization reversal flow (new MQ message type)
  • Batch posting of approved authorizations to TRANSACT VSAM
  • IMS DB structure extension (new segment type for new data)
