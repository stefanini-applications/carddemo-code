🛡️ Authorization Optional Module

User Story Development Guide · CardDemo System · IMS + DB2 + IBM MQ

📋 Module Overview

Purpose: Real-time credit card authorization via IBM MQ request/reply. Authorization history stored in IMS HIDAM database. Fraud records logged to DB2. Online CICS screens allow fraud analysts to review and flag authorizations.

Business Context: When a cardholder swipes at a POS terminal, an authorization request arrives on the MQ queue. This module validates the request against the account credit limit, applies fraud rules, and returns an approved/declined response — all within 2 seconds. Fraud analysts can review authorization history and manually flag suspicious transactions.

Location: app/app-authorization-ims-db2-mq/

Transaction IDs: CP00 (MQ trigger) CPVS (Summary) CPVD (Detail)

Key Programs: COPAUA0C COPAUA1C COPAUS0C COPAUS1C

⚠️ Prerequisites: Requires IBM MQ (MQCONN resource), IMS DB (PSBPAUTL/PSBPAUTB), and DB2 (AUTHFRDS table) — these are optional infrastructure components not in the core VSAM-only deployment.

🔄 Authorization Flow

POS Terminal → MQ Queue (PAUTH.REQUEST)
  → CP00 MQ Trigger → COPAUA0C
    → Validate against ACCTDATA VSAM (credit limit check)
    → Apply fraud rules
    → Write to IMS DB (PAUTSUM0 root + PAUTDTL1 child)
    → Log to DB2 AUTHFRDS (if fraud flag conditions met)
    → MQ Reply (PAUTH.REPLY) → POS Terminal (Approved/Declined)

Fraud Analyst Review:
  CPVS (list) → CPVD (detail) → PF5 (flag fraud) → DB2 AUTHFRDS update

🎯 User Story Development Guide

📝 US Templates

As the authorization system, I want to receive POS authorization requests from MQ so that I can approve or decline transactions in real time.
As a fraud analyst, I want to view a list of pending authorizations so that I can identify suspicious authorization patterns.
As a fraud analyst, I want to mark an authorization as fraudulent via PF5 so that the fraud record is written to DB2 for investigation and reporting.
As a fraud analyst, I want to see full authorization details (account, amount, merchant, timestamp) so that I can make an informed fraud determination.

✅ Acceptance Criteria Patterns

Authorization Processing:
  • Authorization response returned to PAUTH.REPLY within 2 seconds of PAUTH.REQUEST receipt
  • Approved if: account active (status Y) AND transaction amount ≤ available credit
  • Declined if: account inactive OR would exceed credit limit
  • IMS DB PAUTSUM0 record written for every authorization attempt (approved or declined)
  • PAUTDTL1 child segment written with full transaction detail
Fraud Flagging (CPVD, PF5):
  • Pressing PF5 on detail screen writes record to DB2 AUTHFRDS table
  • Confirmation message "FRAUD FLAG SET SUCCESSFULLY" shown in GREEN
  • IMS and DB2 updated via two-phase commit (both succeed or both roll back)

🔧 Technical Foundation

🏗️ Key Components

  • COPAUA0C/COPAUA1C: MQ-triggered authorization processors — MQGET from PAUTH.REQUEST, validate, write IMS + DB2, MQPUT to PAUTH.REPLY
  • COPAUS0C: CICS online — reads IMS DB via DL/I calls, displays authorization summary list
  • COPAUS1C: CICS online — authorization detail view with PF5 fraud flag capability
  • CCPAURQY: Authorization request copybook (account ID, amount, merchant, timestamp)
  • CIPAUSMY: IMS segment layout copybook (PAUTSUM0 fields)
  • IMS DBDs: DBPAUTP0 (HIDAM primary), DBPAUTX0 (index)
  • DB2 Table: AUTHFRDS in database AWSTSFRD

🔗 Dependencies

  • account-management: ACCTDATA VSAM read for credit limit validation
  • IBM MQ: MQCONN (MQ01) CICS resource required; queues AWS.M2.CARDDEMO.PAUTH.REQUEST / PAUTH.REPLY
  • IMS DB: PSBs PSBPAUTL (online) and PSBPAUTB (batch) must be installed
  • DB2: AUTHFRDS table + XAUTHFRD index in AWSTSFRD database
  • app-vsam-mq: CODATE01 / COACCT01 — MQ helper programs for date/account queries

📁 Source Files
