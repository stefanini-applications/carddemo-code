CICS IMS HIDAM DB2 IBM MQ Optional Module app/app-authorization-ims-db2-mq

A complete MQ-triggered credit-card authorization pipeline for the CardDemo platform. Ingests real-time authorization requests from IBM MQ, enforces card and account validation rules, persists pending authorizations in IMS HIDAM, marks fraud in DB2, exposes 3270 BMS summary/detail screens, and provides a nightly batch purge job.

Overview & Purpose

The authorization module is an optional CardDemo extension that demonstrates real-time, multi-subsystem integration on z/OS. When deployed, it adds:

  • An MQ listener (COPAUA0C, transaction CP00) that processes up to 500 pending authorization requests per invocation from queue AWS.M2.CARDDEMO.PAUTH.REQUEST.
  • Card and account validation against VSAM datasets (CARDDAT, ACCTDAT, CUSTDAT, CCXREF).
  • IMS HIDAM persistence of approved/declined pending authorizations in segments PAUTSUM0 (summary per account) and PAUTDTL1 (detail per authorization event).
  • A reply message written back to AWS.M2.CARDDEMO.PAUTH.REPLY containing approval status, response codes, and approved amount.
  • 3270 BMS screens for operators to browse pending authorizations and mark fraud (CPVS / CPVD).
  • DB2 fraud tracking in table CARDDEMO.AUTHFRDS with unique index CARDDEMO.XAUTHFRD.
  • A batch IMS BMP job (CBPAUP0J) that purges expired authorization segments and restores hold amounts.
Business Context: This module simulates the real-time authorization path typical of mainframe payment processing. Modernization teams use it to demonstrate two-phase commit patterns (IMS + DB2), MQ request/response correlation, and fraud detection workflows when migrating to cloud-native architectures.

The authorization module sits alongside the core CardDemo application and depends on three external subsystems (IMS, DB2, MQ) that must be installed before the module is operational.

Processing Flow (Step by Step)

  1. External MQ client places a comma-delimited authorization request on PAUTH.REQUEST.
  2. COPAUA0C (transaction CP00) is triggered via CICS-MQ, opens the request queue, and loops over available messages (limit: 500 per run).
  3. Card number is validated via VSAM cross-reference (CCXREF), account status checked in ACCTDAT, and available credit compared to the transaction amount.
  4. Decline reasons applied: INSUFFICIENT-FUND, CARD-NOT-ACTIVE, ACCOUNT-CLOSED, CARD-FRAUD, MERCHANT-FRAUD (codes 4100 through 5200).
  5. Authorization record written to IMS HIDAM as a PAUTDTL1 segment under the account root PAUTSUM0; summary counters and balances updated accordingly.
  6. Reply message placed on PAUTH.REPLY with auth ID code, response code, response reason, and approved amount.
  7. Operators use CPVS / CPVD screens to browse pending authorizations and optionally flag them as fraudulent via COPAUS2C, which inserts a row in CARDDEMO.AUTHFRDS.
  8. Nightly batch job CBPAUP0J purges expired PAUTDTL1/PAUTSUM0 segments from IMS.

Key Components

COPAUA0C.cbl

Type: CICS COBOL IMS MQ Program
Transaction: CP00
PSB: PSBPAUTB (PCB offset +1)

Card Authorization Decision Engine. Opens MQ request queue, iterates over pending messages (up to 500), validates card/account/credit limit, writes IMS segments, and posts replies. Primary entry point for the authorization pipeline.

VSAM files: ACCTDAT, CUSTDAT, CARDDAT, CARDAIX, CCXREF

COPAUS0C.cbl

Type: CICS COBOL IMS BMS Program
Transaction: CPVS
BMS Map: COPAU00 / COPAU0A

Pending Authorization Summary Screen. Retrieves PAUTSUM0 root segments from IMS for a given account/card, renders a paginated list (PF7/PF8) of pending authorizations with amounts, dates, and approval status. Navigates to detail via COPAUS1C.

COPAUS1C.cbl

Type: CICS COBOL IMS BMS Program
Transaction: CPVD
BMS Map: COPAU01

Pending Authorization Detail Screen. Reads a specific PAUTDTL1 segment by authorization key. Displays all merchant information, transaction amount, and decline reason code from an internal table of 10 entries (0000=APPROVED through 9000=UNKNOWN). Offers PF key to invoke fraud marking.

COPAUS2C.cbl

Type: CICS COBOL IMS DB2 Program
SQL Includes: SQLCA, AUTHFRDS

Mark Authorization as Fraud. Receives a COMMAREA containing account ID, customer ID, and authorization detail record. Inserts new row into CARDDEMO.AUTHFRDS and updates IMS PAUTDTL1 segment PA-AUTH-FRAUD to 'F'.

CBPAUP0C.cbl / CBPAUP0J.jcl

Type: Batch COBOL IMS BMP
PSB: PSBPAUTB (PCB offset +2)

Purge Expired Pending Authorizations. Reads all PAUTSUM0 root segments, evaluates each PAUTDTL1 child against the expiry threshold (SYSIN parameter), deletes expired segments. Reports summary and detail record counts to SYSOUT.

COPAU00.bms / COPAU01.bms

Type: BMS Mapset Definitions
Location: app/app-authorization-ims-db2-mq/bms/

COPAU00 defines the summary screen map COPAU0A (24x80 format with date/time header, authorization list, PF key legend). COPAU01 defines the detail map with full merchant and authorization information fields.

IMS Database Artifacts

DBDs: DBPAUTP0, DBPAUTX0, PADFLDBD, PASFLDBD
PSBs: PSBPAUTB, PSBPAUTL, DLIGSAMP, PAUTBUNL

Business Rules

Authorization Decision Rules (COPAUA0C)

RuleConditionResponse Code / Action
Invalid Card Card number not found in VSAM CCXREF 3100 INVALID CARD; decline
Insufficient Funds Transaction amount exceeds available credit (CREDIT-LIMIT minus CREDIT-BALANCE) 4100 INSUFFICNT FUND; decline
Card Not Active Card status flag inactive in CARDDAT 4200 CARD NOT ACTIVE; decline
Account Closed Account status in ACCTDAT is closed 4300 ACCOUNT CLOSED; decline
Daily Limit Exceeded Cumulative approved authorizations exceed daily cap 4400 EXCED DAILY LMT; decline
Card Fraud Flag Card previously flagged as fraud in IMS segments 5100 CARD FRAUD; decline
Merchant Fraud Flag Merchant ID matches a known fraudulent merchant 5200 MERCHANT FRAUD; decline
Lost Card Card reported as lost 5300 LOST CARD; decline
Approved All checks pass; credit available 0000 APPROVED; write IMS, send reply with approved amount

Validation Rules (COPAUA0C)

  • Required fields: MQ requests must supply card number, merchant ID, transaction amount, and transaction ID. Incomplete payloads are rejected before any IMS interaction.
  • Processing limit: Maximum 500 messages per CP00 invocation (WS-REQSTS-PROCESS-LIMIT) prevents unbounded queue drain.
  • Amount parsing: Transaction amount must be parseable from 13-character alphanumeric field; non-numeric values cause decline code 9000.

Fraud Marking Rules (COPAUS2C)

  • Fraud marking inserts a full row to AUTHFRDS with AUTH_FRAUD = 'Y' and FRAUD_RPT_DATE.
  • Corresponding IMS PAUTDTL1 segment has PA-AUTH-FRAUD = 'F' (PA-FRAUD-CONFIRMED).
  • Fraud removal sets PA-AUTH-FRAUD = 'R' (PA-FRAUD-REMOVED).
  • Both the DB2 insert and IMS segment update are within the same CICS unit-of-work; either both commit or both back out.

Retention / Purge Rules (CBPAUP0C)

  • Expiry threshold in SYSIN parameter (Julian day format YYDDD).
  • Any PAUTDTL1 whose authorization date is older than CURRENT-YYDDD minus WS-EXPIRY-DAYS qualifies for deletion.
  • If all detail segments under a PAUTSUM0 root are deleted, the root segment is also deleted.
  • IMS checkpoint calls issued every N records (controlled by WK-CHKPT-ID-CTR) for recoverability.

IMS Database Schema

IMS HIDAM database DBPAUTP0 uses VSAM as the underlying access method with a two-level segment hierarchy:

DBD: DBPAUTP0  ACCESS=(HIDAM,VSAM)  DATASET DD1=DDPAUTP0,SIZE=4096
|
+-- SEGM: PAUTSUM0  (100 bytes, ROOT)
|      KEY: ACCNTID bytes 1-6, COMP-3, unique
|      Pointer: LCHILD to DBPAUTX0 secondary index (PAUTINDX, POINTER=INDX)
|
+-- SEGM: PAUTDTL1  (200 bytes, CHILD of PAUTSUM0)
       KEY: PAUT9CTS bytes 1-8, CHAR, unique
            = composite Julian date (COMP-3) + time (COMP-3)

Secondary Index DBD: DBPAUTX0
   Provides alternate access path into PAUTSUM0 (e.g. by card number)

Development Guidelines

Adding a New MQ Request Field

  1. Add the field to CCPAURQY.cpy in app/app-authorization-ims-db2-mq/cpy/.
  2. If it must be persisted, add it to CIPAUDTY.cpy and regenerate IMS DBD/PSB (DBPAUTP0); reload using LOADPADB.JCL.
  3. If DB2 tracking is needed, add the column to AUTHFRDS.ddl and update DCL in app/app-authorization-ims-db2-mq/dcl/.
  4. Update COPAUA0C.cbl to map the new field from the MQ buffer to the IMS segment.
  5. Update BMS maps and COPAUS0C/COPAUS1C if the field should be displayed.

IMS Segment Regeneration Steps

1. Edit DBD:  app/app-authorization-ims-db2-mq/ims/DBPAUTP0.dbd
2. DBDGEN:    Submit DBPAUTP0.jcl (IMS DBD generation utility)
3. Edit PSB:  app/app-authorization-ims-db2-mq/ims/PSBPAUTB.psb
4. PSBGEN:    Use IMS PSB generation utility
5. ACBGEN:    IMS Application Control Block generation
6. NEWCOPY:   CEMT NEWCOPY for CICS programs after recompile
7. Verify:    Send a test MQ message; check IMS and BMS output

Coding Conventions

  • IMS status codes MUST use 88-level condition names (STATUS-OK, SEGMENT-NOT-FOUND, END-OF-DB, RETRY-CONDITION, etc.) rather than literal two-character strings in EVALUATE branches.
  • All MQ API calls use CICS-managed connections (MQCONN via CSD resource definitions) to ensure CICS unit-of-work consistency; never call MQCONN directly from application code.
  • Error conditions write structured records via CCPAUERY.cpy layout with subsystem indicator (A/C/I/D/M/F) and severity level (L/I/W/C).
  • COMMAREA is used for program-to-program communication (COPAUS1C to COPAUS2C). Matching LINKAGE SECTION structures must be maintained in both programs.
  • IMS root segment key (ACCNTID) is COMP-3 packed decimal. Always convert correctly from account ID numeric fields before IMS GU/GN calls.
Warning: The PAUTSUM0 segment key is COMP-3 packed decimal (ACCNTID). Incorrect packing will return IMS status GE (segment not found) with no clear diagnostic. Always validate key field conversion when adding new IMS access paths.

       PROCEDURE DIVISION.
      *-- Open MQ request queue
           MOVE WS-REQUEST-QNAME TO MQM-OD-REQUEST-OBJECTNAME
           CALL 'MQOPEN' USING MQM-HCONN-REQUEST
                               MQM-OD-REQUEST
                               WS-OPTIONS
                               W01-HOBJ-REQUEST
                               WS-COMPCODE
                               WS-REASON
           IF WS-COMPCODE = MQCC-OK
               SET WS-REQUEST-MQ-OPEN TO TRUE
           END-IF

      *-- Process messages until limit or queue empty
           PERFORM UNTIL WS-LOOP-END
                      OR WS-MSG-PROCESSED >= WS-REQSTS-PROCESS-LIMIT
               CALL 'MQGET' USING W01-HCONN-REQUEST
                                  W01-HOBJ-REQUEST
                                  MQM-MD-REQUEST
                                  MQM-GMO-REQUEST

                                  W01-GET-BUFFER
                                  W01-DATALEN

                                  WS-REASON
               IF WS-COMPCODE = MQCC-OK
                   ADD 1 TO WS-MSG-PROCESSED
                   PERFORM 2000-PROCESS-AUTH-REQUEST

Error Handling

All programs use the structured CCPAUERY.cpy error logging layout with subsystem and severity codes.

IMS Error Handling

88-level ConditionIMS StatusAction
STATUS-OK' ' or 'FW'Continue normal processing
SEGMENT-NOT-FOUND'GE'Handle as expected no-data condition
DUPLICATE-SEGMENT-FOUND'II'Log warning; skip insert
DATABASE-UNAVAILABLE'BA'RETRY-CONDITION; delay and retry
COULD-NOT-SCHEDULE-PSB'TE'RETRY-CONDITION; delay and retry
END-OF-DB'GB'Normal end-of-scan termination

MQ Error Handling

  • MQGET returns MQCC-FAILED with reason MQRC-NO-MSG-AVAILABLE when the queue is empty — this is the normal loop-exit condition, not an error.
  • All other MQ failures capture WS-COMPCODE and WS-REASON, log to CCPAUERY with ERR-MQ subsystem indicator, and terminate gracefully.
  • Messages that cannot be processed are backed out to the dead-letter queue via the CICS backout mechanism.

DB2 Error Handling (COPAUS2C)

  • All SQL operations check SQLCA via WS-SQLCODE and WS-SQLSTATE.
  • Non-zero SQLCODE sets ERR-FLG-ON and returns error status in the COMMAREA to the calling screen.
  • Fraud INSERT and IMS segment update are in the same CICS unit-of-work; failure in either rolls back both.

User Story Examples

Real-time Authorization (Medium, 3-5 pts)
As a cloud POS emulator, I want to submit an MQ message to PAUTH.REQUEST and receive an approval/decline reply within 500 ms so that I can simulate real-time point-of-sale authorization scenarios.

Acceptance Criteria:
  • Reply message contains a valid AUTH-ID-CODE and AUTH-RESP-CODE.
  • Approved transactions are written to IMS PAUTDTL1 with PA-AUTH-APPROVED = '00'.
  • Declined transactions have a non-zero response code matching the decline reason table.
  • Processing time under nominal load stays below 500 ms.
Fraud Marking (Medium, 3-5 pts)
As a fraud analyst, I want to mark a pending authorization as fraudulent via the CPVD detail screen so that the transaction is persisted in AUTHFRDS for analytics and the IMS segment is updated accordingly.

Acceptance Criteria:
  • Row inserted in CARDDEMO.AUTHFRDS with AUTH_FRAUD = 'Y' and FRAUD_RPT_DATE.
  • IMS PAUTDTL1 segment has PA-AUTH-FRAUD = 'F' (PA-FRAUD-CONFIRMED).
  • BMS message line confirms fraud marking succeeded.
  • Undo operation sets PA-AUTH-FRAUD = 'R' (PA-FRAUD-REMOVED).
Nightly Purge (Simple, 1-2 pts)
As a batch operator, I want CBPAUP0J to purge authorizations older than the configured threshold from IMS so that the HIDAM database stays within its allocated VSAM extent.

Acceptance Criteria:
  • Only segments older than the SYSIN-configured expiry threshold are deleted.
  • SYSOUT shows counts of summary and detail segments deleted per run.
  • Active/unexpired records are not removed.
  • Job completes within the nightly batch window with IMS checkpoints taken.
Decline Reason Display (Simple, 1-2 pts)
As an operator, I want the CPVD detail screen to display a human-readable decline reason so I can quickly assess whether to escalate a transaction.

Acceptance Criteria:
  • All 10 reason codes (0000 through 9000) map to readable descriptions on the BMS screen.
  • Code 0000 shows "APPROVED".
  • Unrecognized codes display as "9000 UNKNOWN".
New Merchant Field (Complex, 5-8 pts)
As a product owner, I want the authorization request to carry an additional merchant latitude/longitude field so that fraud models can incorporate geolocation data.

Acceptance Criteria:

  • CIPAUDTY copybook and IMS DBPAUTP0 DBD regenerated; PAUTDTL1 segment stores new fields.
