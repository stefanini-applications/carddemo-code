🛡️ Authorization Module

Guidance for stories driving MQ, IMS, and DB2 integration scenarios.

📋 Overview

Purpose: Deliver a realistic credit-card authorization pipeline that ingests MQ requests, enforces business rules, writes pending authorizations to IMS, flags fraud in DB2 (`AUTHFRDS`), and exposes summary/detail screens plus a purge job.

Business Context: This optional module showcases two-phase commit (IMS + DB2) and MQ request/response patterns that modernization teams can demo when connecting cloud POS events to a mainframe core.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Real-time Authorization: As a fraud analyst, I want to see pending requests in `CPVS` so I can escalate suspicious transactions before they settle.
  • MQ Integration: As a cloud POS emulator, I want to post a comma-delimited message to `PAUTH.REQUEST` and validate the reply so I can test declines and approvals.
  • Data Purge: As an operator, I want `CBPAUP0J` to purge expired authorizations nightly so IMS/VSAM datasets do not overflow.

⚡ Development Acceleration Factors

  • Message formats: `app/app-authorization-ims-db2-mq/README.md` documents request/response CSV fields (card number, merchant data, response codes).
  • Diagrams: `diagrams/auth_flow.png`, `auth_summary.png`, `auth_details.png`, and fraud marking images show the user journey.
  • Copybooks & BMS: Copybooks such as `CIPAUSMY`/`CIPAUDTY` align IMS segments with BMS map layouts `COPAU00`/`COPAU01`.
  • Automation: JCL `CBPAUP0J` purges expired authorizations, and sample MQ definitions live under `app/app-authorization-ims-db2-mq/mq`.

📊 Complexity Guidelines

  • Simple (1-2 pts): Update a BMS label or PF key action for `CPVS` without touching MQ payloads.
  • Medium (3-5 pts): Add a new MQ field (e.g., merchant ZIP) to the request, adjust `COPAUA0C`, and verify IMS segments and DB2 inserts.
  • Complex (5-8 pts): Extend the fraud-marking flow to another downstream queue, ensuring DB2 transactions (`COPAUS2C`) and IMS updates stay two-phase consistent.

🔧 Technical Foundation

🏗️ Key Components

  • `COPAUA0C` processes MQ requests from `PAUTH.REQUEST` and writes pending entries into IMS segments (PAUTSUM0/PAUTDTL1).
  • `COPAUS0C`/`COPAUS1C` render summary/detail screens for operators to inspect authorizations.
  • `COPAUS2C` marks fraud and inserts a row into DB2 table `AUTHFRDS`; `CBPAUP0C`/`CBPAUP0J` purge stale data.
  • IMS DBDs (`DBPAUTP0`, `DBPAUTX0`) and PSBs (`PSBPAUTB`, `PSBPAUTL`) under `app/app-authorization-ims-db2-mq/ims`.
  • MQ resources defined in `app/app-authorization-ims-db2-mq/mq` with queues `AWS.M2.CARDDEMO.PAUTH.REQUEST/REPLY` and a connection to CICS.

🔗 Dependencies

  • IMS HIDAM for pending authorization storage.
  • DB2 for fraud analytics (`AUTHFRDS`, `XAUTHFRD`).
  • MQ for request/response connectivity (demo clients, pos emulators).
  • VSAM copybooks for account/customer data.

🗂️ Public Interfaces

CP00 - MQ-triggered authorization intake (requests must carry card, merchant, amount)
CPVS - Pending authorization summary screen
CPVD - Authorization detail + fraud flag screen
CBPAUP0J - Batch purge of expired authorizations
AWS.M2.CARDDEMO.PAUTH.REQUEST / .REPLY - MQ queues documented in README with CSV payloads

✅ Acceptance Criteria Examples

  • MQ payloads missing required fields are rejected with a BMS error and recorded in `CCPAUERY` logs.
  • Marking a fraud case writes to DB2 `AUTHFRDS.AUTH_FRAUD` and updates the IMS segment without leaving stale locks.
  • `CBPAUP0J` deletes only authorizations older than the retention window documented in `README.md`, keeping IMS/VSAM alignment.
