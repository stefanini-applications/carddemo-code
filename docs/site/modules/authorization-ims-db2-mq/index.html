🔒 Authorization IMS/DB2/MQ

User Story Development Guide — Credit Card Authorization with IMS, Db2, and MQ

← Back to Documentation Index
⚠️ Optional Module — Requires IMS DB, IBM Db2, and IBM MQ infrastructure

Module ID: authorization-ims-db2-mq

Purpose: Simulates credit card authorization processing using IBM MQ for request/response messaging, IMS DB for pending authorization records, and IBM Db2 for transaction logging. Includes a batch purge job for expired authorizations.

🎯 User Story Development Guide

📝 US Templates

  • View Pending Authorizations: As a card holder, I want to view a list of pending authorizations so that I can see charges awaiting approval.
  • Authorization Details: As a card holder, I want to view authorization details so that I can verify or dispute a specific charge.
  • Process Authorization: As a system, I want to process incoming authorization requests via MQ so that purchase approvals are handled in real-time.
  • Purge Expired: As a batch system, I want to purge expired authorization records so that IMS DB and Db2 stay clean.

⚡ Development Acceleration Factors

  • CCPAURQY / CCPAURLY copybooks: Authorization request and reply message structures already defined
  • IMSFUNCS copybook: IMS function codes and PCB patterns — reusable for any IMS operation
  • CIPAUDTY / CIPAUSMY: IMS segment definitions for authorization records
  • COPAUA0C MQ pattern: Complete MQ trigger → receive → process → reply pattern implemented

🔧 Technical Foundation

🏗️ Key Components

  • COPAUA0C (CICS) — MQ-triggered authorization processor; receives request, reads/writes IMS DB, logs to Db2, sends MQ reply
  • COPAUS0C (CICS) — Pending authorization summary; reads IMS DB via PCB
  • COPAUS1C (CICS) — Authorization detail; reads IMS and updates Db2 on approve/reject
  • COPAUS2C (CICS) — Supplemental authorization view
  • CBPAUP0C (batch) — Purge expired authorizations from IMS DB and Db2
  • COPAU00 / COPAU01 (BMS maps) — Authorization summary and detail screens
  • DBPAUTP0.dbd / DBPAUTX0.dbd (IMS DBD) — Authorization pending database
  • PSBPAUTB.psb / PSBPAUTL.psb (IMS PSB) — Program specification blocks
  • AUTHFRDS.ddl / AUTHFRDS.dcl — Db2 table for authorization records

🔗 Dependencies

  • IBM MQ: Authorization requests arrive on defined queues; replies sent to reply-to queue
  • IMS DB: DBPAUTP0 database stores pending authorization records; accessed via DLI calls
  • IBM Db2: AUTHFRDS table logs processed authorizations
  • VSAM CARDDAT: Card validation during authorization processing

🌐 CICS Transactions

CPVS  → COPAUS0C  — Pending authorization summary (reads IMS + VSAM)
CPVD  → COPAUS1C  — Authorization detail (updates IMS + inserts Db2)
CP00  → COPAUA0C  — Process auth requests (MQ trigger, IMS insert/update)

📋 Authorization Message Structures

CCPAURQY — Authorization Request:
  - Card number, merchant ID, amount, timestamp
  - Request type, source system

CCPAURLY — Authorization Reply:
  - Approval/decline code
  - Authorization ID, response timestamp

CCPAUERY — Authorization Enquiry:
  - Query parameters for pending auth lookup

📋 Business Rules

  • BR-AUTH-EXT-01: Authorization requests arrive via MQ and trigger COPAUA0C processing
  • BR-AUTH-EXT-02: Pending authorizations are stored in IMS DB (DBPAUTP0)
  • BR-AUTH-EXT-03: Processed authorizations are logged to Db2 AUTHFRDS table
  • BR-AUTH-EXT-04: Expired authorizations are purged by CBPAUP0J batch job
  • BR-AUTH-EXT-05: MQ correlation ID links requests to replies

✅ Acceptance Criteria Examples

  • View Pending: Given pending authorizations exist in IMS DB, when card holder navigates to CPVS, then all pending authorizations for their account are listed
  • Authorization Detail: Given an authorization is selected, when CPVD is displayed, then merchant, amount, and timestamp are shown; user can approve or dispute
  • MQ Processing: Given an authorization request arrives on the MQ queue, when COPAUA0C triggers, then the authorization is stored in IMS and a reply is sent on the reply-to queue
  • Purge Job: Given CBPAUP0J runs, then authorizations older than the expiry threshold are removed from IMS DB and Db2
