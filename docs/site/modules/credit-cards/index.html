Overview and Purpose

The credit-cards module provides the complete online CICS interface for listing, viewing, and updating credit card records within the CardDemo mainframe application. It is one of the core functional modules available to regular (non-admin) users after successful authentication.

Each credit card is stored as a record in CARDFILE (VSAM KSDS) and is linked to an account via the XREFFILE cross-reference file. The cross-reference also maps each card to its owner customer, enabling efficient navigation across the accounts–cards–customers relationship.

Module ID: credit-cards
Layer: Online CICS Business Logic
User Role: Regular User (authenticated via COSGN00C, SEC-USR-TYPE = 'U')
Source version: CardDemo_v1.0-15-g27d6c6f-68 (2022-07-19)

Architectural Context

The credit-cards module follows the pseudo-conversational CICS pattern used throughout CardDemo. Each user interaction triggers a transaction, the program processes the request, sends a BMS map to the terminal, and returns with a COMMAREA preserving state for the next interaction. 

Terminal ──CCLI──► COCRDLIC ──XCTL──► COCRDSLC (view)
                                   ──XCTL──► COCRDUPC (update)
                                   ──XCTL──► COMEN01C (main menu on PF3)

COCRDSLC ──XCTL──► COCRDLIC (back to list)
                ──XCTL──► COCRDUPC (direct update from detail)
                ──XCTL──► COMEN01C (main menu on PF3)

COCRDUPC ──XCTL──► COCRDSLC (back to detail)
                ──XCTL──► COMEN01C (main menu on PF3)

Business Rules

Card Listing (COCRDLIC / CCLI)

  • A regular user sees only cards linked to their own account — XREFFILE AIX is browsed using the account ID from COMMAREA.
  • An admin user can list all cards across all accounts — browse starts from the beginning of CARDFILE.
  • Up to 7 cards are displayed per page; PF7 (backward) and PF8 (forward) enable pagination.
  • Optional filter fields: Account ID (ACCTSID) and Card Number (CARDID) can narrow the list.
  • Selection action codes: S = view detail (routes to COCRDSLC), U = update (routes to COCRDUPC).
  • Only one selection is allowed per page; selecting multiple rows triggers an error message.
  • The filter account ID must be a non-zero 11-digit numeric value when provided.
  • The filter card number must be a 16-digit numeric value when provided.

Card Detail View (COCRDSLC / CCDL)

  • The user must provide a valid account number (non-zero, 11-digit numeric).
  • An optional card number (16-digit numeric) further narrows the search.
  • When only account ID is supplied, the first matching card is found by browsing CARDAIX.
  • When both account and card number are supplied, CARDFILE is read directly by card number, then validated against the account.
  • If no card is found for the given search condition, a "Did not find cards for this search condition" message is displayed.
  • The screen displays: card number, account ID, CVV, embossed name, expiry date (YYYY-MM-DD), and active status.

Card Update (COCRDUPC / CCUP)

  • Editable fields: CARD-EMBOSSED-NAME, CARD-ACTIVE-STATUS, CARD-EXPIRAION-DATE (month and year separately).
  • Card number and account ID are display-only — they cannot be changed after lookup.
  • CARD-ACTIVE-STATUS must be 'Y' or 'N' only.
  • CARD-EMBOSSED-NAME can contain only alphabetic characters and spaces (no digits or special characters).
  • Expiry month must be between 1 and 12 inclusive.
  • Expiry year must be between 1950 and 2099 inclusive.
  • Changes are validated on Enter; the user must press PF5 to confirm and commit the update.
  • If no changes are detected compared to the fetched record, a "No change detected" message is shown and no REWRITE occurs.
  • A successful update displays "Changes committed to database"; a failure shows "Changes unsuccessful. Please try again".

Cross-Reference Consistency

  • CARDFILE and XREFFILE must remain in sync — each CARD-NUM in CARDFILE should have a corresponding XREF-CARD-NUM in XREFFILE.
  • The current online programs only update CARDFILE (REWRITE). XREFFILE updates are handled during data loading (batch JCL: XREFFILE.jcl).
  • CVV is stored as plain numeric in CARDFILE — no encryption in the base application.

COMMAREA and State Management

Credit-cards programs use a two-part COMMAREA:

  1. CARDDEMO-COMMAREA (from COCOM01Y) — system-wide state including user ID, user type, and cross-module routing fields. Occupies the first N bytes.
  2. WS-THIS-PROGCOMMAREA (from CVCRD01Y) — module-specific context including the current account ID (CC-ACCT-ID), card number (CC-CARD-NUM), customer ID (CC-CUST-ID), AID key, error messages, and next-program routing. Appended after CARDDEMO-COMMAREA.
*> COMMAREA layout:
*> |--- CARDDEMO-COMMAREA (system, fixed length) ---|--- CC-WORK-AREAS (module) ---|

*> On CICS RETURN:
MOVE CARDDEMO-COMMAREA    TO WS-COMMAREA
MOVE WS-THIS-PROGCOMMAREA TO
     WS-COMMAREA(LENGTH OF CARDDEMO-COMMAREA + 1 :
                 LENGTH OF WS-THIS-PROGCOMMAREA)

EXEC CICS RETURN
     TRANSID  (LIT-THISTRANID)
     COMMAREA (WS-COMMAREA)
     LENGTH   (LENGTH OF WS-COMMAREA)
END-EXEC

*> On re-entry, split COMMAREA back:
MOVE DFHCOMMAREA(1:LENGTH OF CARDDEMO-COMMAREA) TO CARDDEMO-COMMAREA
MOVE DFHCOMMAREA(LENGTH OF CARDDEMO-COMMAREA + 1:
                 LENGTH OF WS-THIS-PROGCOMMAREA) TO WS-THIS-PROGCOMMAREA

The EIBCALEN register is checked at entry to distinguish the first invocation from subsequent pseudo-conversational re-entries: 

IF EIBCALEN = 0
    *> First invocation — fresh start, initialize COMMAReas
    INITIALIZE CARDDEMO-COMMAREA
               WS-THIS-PROGCOMMAREA
ELSE
    *> Subsequent invocation — restore state from COMMAREA
    MOVE DFHCOMMAREA... TO ...
END-IF

Input Validation Logic

Account ID Validation

*> From COCRDSLC / COCRDUPC
IF CC-ACCT-ID = SPACES OR LOW-VALUES
   SET WS-PROMPT-FOR-ACCT TO TRUE       *> 'Account number not provided'
ELSE IF CC-ACCT-ID-N = 0
   SET SEARCHED-ACCT-ZEROES TO TRUE     *> 'Account number must be a non zero 11 digit number'
ELSE IF CC-ACCT-ID NOT NUMERIC
   SET SEARCHED-ACCT-NOT-NUMERIC TO TRUE
END-IF

Card Number Validation

*> Card number, if provided, must be 16-digit numeric
IF CC-CARD-NUM NOT = SPACES
   IF CC-CARD-NUM NOT NUMERIC
      SET SEARCHED-CARD-NOT-NUMERIC TO TRUE
      *> 'Card number if supplied must be a 16 digit number'
   END-IF
END-IF

Card Name Validation (Update only)

*> Name must be alphabetic characters and spaces only
MOVE CARD-EMBOSSED-NAME TO CARD-NAME-CHECK
INSPECT CARD-NAME-CHECK
    REPLACING ALL SPACES BY 'A'
IF CARD-NAME-CHECK NOT ALPHABETIC
   SET WS-NAME-MUST-BE-ALPHA TO TRUE
   *> 'Card name can only contain alphabets and spaces'
END-IF

Active Status Validation (Update only)

*> Must be 'Y' or 'N'
IF CARD-ACTIVE-STATUS NOT = 'Y' AND
   CARD-ACTIVE-STATUS NOT = 'N'
   SET CARD-STATUS-MUST-BE-YES-NO TO TRUE
   *> 'Card Active Status must be Y or N'
END-IF

Expiry Date Validation (Update only)

*> Month: 1-12
MOVE CARD-EXPIRY-MONTH TO CARD-MONTH-CHECK
IF NOT VALID-MONTH      *> 88-level: VALUES 1 THRU 12
   SET CARD-EXPIRY-MONTH-NOT-VALID TO TRUE

*> Year: 1950-2099
MOVE CARD-EXPIRY-YEAR TO CARD-YEAR-CHECK
IF NOT VALID-YEAR       *> 88-level: VALUES 1950 THRU 2099
   SET CARD-EXPIRY-YEAR-NOT-VALID TO TRUE

Error Handling

*> Standard pattern for all file operations:

Development Patterns

Pseudo-Conversational CICS Pattern

All three programs follow the same entry/exit pattern:

  1. Check EIBCALEN — if 0, first invocation; initialize COMMAReas and display initial screen.
  2. Otherwise, restore state from COMMAREA (CARDDEMO + module-specific).
  3. Read the incoming BMS map (EXEC CICS RECEIVE MAP).
  4. Identify the AID key (EIBAID) and route to the appropriate processing paragraph.
  5. Validate inputs, access VSAM files, prepare output.
  6. Send the output BMS map (EXEC CICS SEND MAP).
  7. Save state back to combined COMMAREA and EXEC CICS RETURN TRANSID(...) COMMAREA(...).

AIX Browse Pattern (Pagination)

COCRDLIC maintains two read-direction routines:

  • 9000-READ-FORWARD — STARTBR then READNEXT loop, stops when 7 records collected or ENDFILE.
  • 9100-READ-BACKWARDS — STARTBR then READPREV loop, reads backwards for PF7 (page up).

The RID field (WS-CARD-RID) stores the composite key (ACCT-ID + CARD-NUM) to resume browsing from the correct position on the next page.

Two-Pass Update Pattern

COCRDUPC uses a confirm-before-commit approach to prevent accidental updates:

  1. Pass 1 (Enter): Validate all input fields. If valid, display confirmation prompt ("Changes validated. Press F5 to save").
  2. Pass 2 (PF5): Re-read CARDFILE with UPDATE intent, apply changes, REWRITE.

This pattern prevents partial updates and gives users a chance to review changes before they are committed.

             SET READ-LOOP-EXIT TO TRUE
        WHEN OTHER
             *> Handle error
             SET READ-LOOP-EXIT TO TRUE
    END-EVALUATE
END-PERFORM

EXEC CICS ENDBR DATASET('CARDAIX ') END-EXEC

Example: Updating a Card Record

*> Step 1: Read with UPDATE intent (holds exclusive lock)
EXEC CICS READ
     DATASET  ('CARDDAT ')
     INTO     (CARD-RECORD)
     LENGTH   (LENGTH OF CARD-RECORD)
     RIDFLD   (WS-CARD-RID-CARDNUM)
     UPDATE
     RESP     (WS-RESP-CD)
END-EXEC

IF WS-RESP-CD = DFHRESP(NORMAL)
    *> Apply validated updates
    MOVE NEW-EMBOSSED-NAME  TO CARD-EMBOSSED-NAME
    MOVE NEW-ACTIVE-STATUS  TO CARD-ACTIVE-STATUS
    MOVE NEW-EXPIRY-YYYY    TO CARD-EXPIRAION-DATE(1:4)
    MOVE '-'                TO CARD-EXPIRAION-DATE(5:1)
    MOVE NEW-EXPIRY-MM      TO CARD-EXPIRAION-DATE(6:2)
    MOVE '-01'              TO CARD-EXPIRAION-DATE(8:3)

    *> Step 2: Write back
    EXEC CICS REWRITE
         DATASET  ('CARDDAT ')
         FROM     (CARD-RECORD)
         LENGTH   (LENGTH OF CARD-RECORD)
         RESP     (WS-RESP-CD)
    END-EXEC

    IF WS-RESP-CD = DFHRESP(NORMAL)
        SET CONFIRM-UPDATE-SUCCESS TO TRUE
    ELSE
        SET INFORM-FAILURE TO TRUE
    END-IF
END-IF

Technical Debt and Known Issues

  • Typo in field name: CARD-EXPIRAION-DATE (missing 't' in expiration). This appears in CVACT02Y and all programs. Must be preserved for binary compatibility with existing VSAM data files.
  • CVV plain-text storage: CARD-CVV-CD is stored as an unencrypted 3-digit numeric. This is a security concern for any modernization effort.
  • No real-time account balance link: Card activation/deactivation in COCRDUPC does not trigger any real-time recalculation of available credit. Balance remains as-is until next batch cycle (POSTTRAN).
  • XREFFILE not updated online: The update program (COCRDUPC) only REWRITEs CARDFILE. If a card-to-account mapping change were needed, XREFFILE would require a separate update — not currently implemented in the online path.
  • Single-user session model: CICS pseudo-conversational pattern with COMMAREA provides no concurrent session management. Simultaneous updates to the same card by two users could result in last-write-wins behaviour.
  • Mixed COBOL coding styles: The application intentionally uses varied COBOL patterns and coding styles to exercise modernization tooling coverage. Some paragraphs use GO TO, others use PERFORM; some use 88-level conditions extensively while others use inline IF chains.

DEFINE FILE(XREFFILE) GROUP(CARDDEMO)  DSNAME(...)

JCL for VSAM Load

CARDFILE is loaded via app/jcl/CARDFILE.jcl (IDCAMS REPRO from flat file). XREFFILE is loaded similarly. These jobs must run before CICS can open the files.

RACF Security

Access to CCLI, CCDL, and CCUP transactions is controlled by RACF. Regular users (SEC-USR-TYPE = 'U') have access to all three. The CARDFILE and XREFFILE VSAM datasets must be profiled in RACF for CICS access.
