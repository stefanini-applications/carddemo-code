📦 Core Services

Guide for developing user stories grounded on the base CardDemo experience.

📋 Overview

Purpose: Ship the core credit card lifecycle: sign-on, account/card maintenance, transactions, billing, and statement generation backed by VSAM, COBOL, BMS, and JCL.

Business Context: Provides the mainframe experience that fronts end-users and admin staff when assessing modernization choices; all modernization stories should keep this base intact.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Lifecycle: As a cardholder, I want to see balances and credit limits on the `CAVW` screen so I can make payment decisions quickly.
  • Transaction Review: As an operations analyst, I want to page through `CT00` so I can audit transactions without leaving the transaction list.
  • Batch Orchestration: As a scheduler, I want `POSTTRAN`/`CREASTMT` to run successfully so nightly reconciliation completes before business hours.

⚡ Development Acceleration Factors

  • Copybook reuse: `CVACT01Y`, `CVCUS01Y`, and `CVTRA06Y` define the account, customer, and transaction payloads that any new screen can reuse.
  • BMS templates: `COSGN00`, `COMEN01`, and other mapsets show how to structure prompts, message areas, and PF keys without building everything from scratch.
  • JCL skeletons: Jobs like `POSTTRAN`, `ACCTFILE`, and `CREASTMT` already set up COPYLIB and datasets, so new steps can piggyback on their DD statements.
  • Dataset awareness: VSAM files (`AWS.M2.CARDDEMO.ACCTDATA.PS`, `.DALYTRAN.PS`, `.CARDXREF.VSAM.KSDS`) are documented and reused across flows.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new field or PF-key to an existing BMS map (e.g., display a helper text on `COSGN00`).
  • Medium (3-5 pts): Enhance `COACTVWC` validation, add code to compute a derived balance, or wrap a new status message in `ERRMSG`.
  • Complex (5-8 pts): Introduce new batch sequencing (e.g., add a step to `POSTTRAN` that writes to `TCATBALF` and ensure nightly dependencies remain stable).

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs: `COACTVWC`, `COACTUPC`, `COCRDLI`, `COCRDSLC`, `COTRN00C`, `CORPT00C` handle account/card/transaction logic.
  • BMS maps: `COSGN00`, `COMEN01`, `COTRN00`, `COACTVW`, `CU00` define each screen layout and message area.
  • Copybooks: `CVACT01Y`, `CVCUS01Y`, and `CVTRA06Y` guarantee consistent offsets across VSAM/COBOL reads.
  • Batch jobs: `POSTTRAN`, `CREASTMT`, `ACCTFILE`, `TRANIDX`, `TRANEXTR`, `TRANBKP` refresh VSAM data.

🔗 Dependencies

  • VSAM datasets (`ACCTDATA`, `CARDDATA`, `DALYTRAN`, `CARDXREF`, etc.).
  • CICS resources defined via `app/csd/*.csd` with transactions like `CC00`, `CA00`, `CT00`.
  • JCL jobs referencing `AWS.M2.CARDDEMO.LOADLIB` and spool datasets.

🌐 Public Interfaces

CC00  (COSGN00)   - Sign-on page (User ID/Password fields)
CM00  (COMEN01)   - Main menu routing
CAVW  (COACTVWC)  - Account view and edit
CT00  (COTRN00C)  - Transaction listing and paging
CB00  (COBIL00C)  - Bill payment UI (writes DALYTRAN)
POSTTRAN (CBTRN02C) - Batch posting/daily transaction processing
CREASTMT (CBSTM03A/B) - Statement generation (input from DALYTRAN)

✅ Acceptance Criteria Examples

  • Account view must show `ACCT-CURR-BAL`, `ACCT-CREDIT-LIMIT`, and respect `ACCT-ACTIVE-STATUS` (via `WS-INPUT-FLAG`).
  • Transaction list must page with PF7/PF8 and call the correct `COTRN00` mapset.
  • Batch flows (`POSTTRAN`, `CREASTMT`) must complete with SYSOUT return codes of zero and refresh `TCATBALF`/`DALYTRAN`.
