📦 Transaction Type Extension

Guide for Developing User Stories

📋 Overview

Purpose: Demonstrates DB2 static SQL, cursor navigation, and dual-storage workflows for transaction metadata while feeding the base VSAM posting pipelines.

Business Context: Helps modernization teams understand how DB2 can coexist with VSAM transaction posting and provides admin controls for categories and types.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • CRUD tooling: As an admin, I want `CTTU` to add a new transaction type so that the posting engine can reference it.
  • Bulk sync: As an operations lead, I want `TRANEXTR` to export DB2 rows into VSAM files so the COBOL posting jobs stay in sync.
  • Cursor testing: As QA, I want `CTLI` to page both forward and backward so the UI handles large result sets.

⚡ Development Acceleration Factors

  • DB2 DDL (`app/app-transaction-type-db2/ddl`): Provides table definitions and indexes to extend or version with minimal impact.
  • Control files (`ctl/`): Already contain host variable declarations that can be reused for new stored fields.
  • Batch scaffolding: `TRANEXTR` and `MNTTRDB2` illustrate DB2-to-VSAM extraction patterns that can be repurposed.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a descriptive column to `TRANSACTION_TYPE` and display it on `CTLI` without changing the cursor logic.
  • Medium (3-5 pts): Introduce a referential integrity check between `TRANSACTION_TYPE_CATEGORY` and other posted data.
  • Complex (5-8 pts): Rework `TRANEXTR` to handle incremental change data capture or add new staging tables.

✅ Acceptance Criteria Examples

  • `CTTU` commits via embedded SQL and displays an error map on SQLCODE <> 0 using the `SQLCA` handler.
  • `CTLI` forward/backward paging loads the correct `TRANSACTION_TYPE` rows while leaving cursor status fields (`MORE-FLAGS`) accurate.
  • `TRANEXTR` produces VSAM-compatible files and the COBOL posting programs (`COCRDLIC` etc.) consume the updated file without requiring structure changes.

🔧 Technical Foundation

🏗️ Key Components

  • CICS programs: `COTRTUPC` (CTTU add/edit) and `COTRTLIC` (CTLI list/update/delete).
  • DB2 tables: `CARDDEMO.TRANSACTION_TYPE`, `TRANSACTION_TYPE_CATEGORY` with FK `DELETE RESTRICT`.
  • Batch jobs: `CREADB21` creates the DB2 schema, `TRANEXTR` extracts VSAM-friendly copies, `MNTTRDB2` provides offline maintenance.
  • Control/DDL files: DB2 declarations (dcl/), DDL scripts (ddl/), and copybooks used by the precompiler.

🔗 Dependencies

  • DB2 subsystem with precompiler integration enabled in CICS.
  • Base CardDemo for VSAM posting compatibility.
  • JCL procs that compile DB2-enabled programs and run extraction jobs.

🌐 Public Interfaces


CICS:
- CTTU / COTRTUPC -> Add or edit transaction types with static SQL
- CTLI / COTRTLIC -> List, update, delete transaction types (forward/backward cursor)
Batch:
- CREADB21 -> Create DB2 schema and load tables
- TRANEXTR -> Produce VSAM files from DB2 output
- MNTTRDB2 -> Batch maintenance updates
