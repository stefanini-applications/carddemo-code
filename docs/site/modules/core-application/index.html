1. Module Overview and Purpose

Business role: core-application is the runtime heart of CardDemo. It carries the interactive account/card/transaction journeys used by cardholders, operators, and administrators before any optional extension module is considered.

Operational role: It also defines core nightly consistency operations by coupling online updates with batch jobs that rebuild and post files used by CICS transactions.

Problem solved: Maintain low-latency terminal-first card servicing in a deterministic mainframe execution model where copybook contracts, mapsets, and VSAM record layouts must remain synchronized.

What this module owns

  • Sign-on and user routing (COSGN00C, COMEN01C, COADM01C).
  • Account view/update (COACTVWC, COACTUPC).
  • Card list/update flows (COCRDLIC, COCRDUPC, plus card detail handoff).
  • Transaction list/detail/add (COTRN00C, COTRN01C, COTRN02C).
  • Bill payment posting from interactive screen (COBIL00C).
  • Admin user security CRUD (COUSR00C..COUSR03C).
  • Batch refresh/posting/statement jobs under app/jcl/.

2. Architectural Context

2.1 Runtime architecture

The module is a CICS transaction mesh. Entry programs populate and propagate CARDDEMO-COMMAREA fields (CDEMO-FROM-PROGRAM, CDEMO-TO-PROGRAM, CDEMO-PGM-CONTEXT) and transfer control via EXEC CICS XCTL.

EXEC CICS XCTL
    PROGRAM(CDEMO-TO-PROGRAM)
    COMMAREA(CARDDEMO-COMMAREA)
END-EXEC.

2.2 UI orchestration

BMS mapsets define fields, protections, and PF-key hints. COBOL programs perform SEND MAP/MAPSET, RECEIVE MAP/MAPSET, and context-aware validation.

Inbound controls

  • DFHENTER for action/submit.
  • DFHPF3 for back/exit.
  • DFHPF7/DFHPF8 for list paging.
  • DFHPF5/DFHPF12 for save/cancel in update screens.

State style

  • CDEMO-PGM-ENTER for first screen entry.
  • CDEMO-PGM-REENTER for post-input loops.
  • Screen-local state in extended commarea blocks (for paging keys and update snapshots).

2.3 Data access model

Online flows directly read/rewrite VSAM datasets such as ACCTDAT, CARDDAT, CUSTDAT, TRANSACT, and USRSEC. Cross-reference files (CXACAIX, CCXREF) resolve account/card relationships.

2.4 Batch coupling

JCL jobs provision datasets, rebuild indexes, post daily transactions, and generate statements. Shell wrappers submit those JCL streams through JES via FTP tunnel.

4. Dependencies

4.1 Internal module dependencies

  • authorization: Main menu option entry can route to optional pending authorization view program (COPAUS0C) when installed.
  • transaction-type: Admin menu copybook includes optional DB2 transaction-type maintenance options (COTRTLIC, COTRTUPC).
  • account-extractions: Not directly invoked by core screens, but shares foundational datasets and operational environment.

4.2 External platform dependencies

  • CICS runtime services (SEND, RECEIVE, XCTL, READ/READNEXT/READPREV, REWRITE, STARTBR/ENDBR).
  • VSAM KSDS datasets and AIX/PATH definitions.
  • JES for batch execution and SDSF command streams for file state control.
  • RACF + seeded user dataset records controlling access roles.
  • Shell + FTP tunnel workflow for remote JCL submission.

6.2 Business rules extracted from code paths

  • Sign-on rejects blank user ID/password and sets cursor to required field.
  • Main menu option must be numeric and in configured range; invalid options return explicit error text.
  • Account view requires non-zero numeric 11-digit account ID.
  • Account update uses staged state model: fetch details, edit, validate, confirm with PF5, then write.
  • Account update validates phone formatting, SSN segments, dates, yes/no flags, and state/ZIP cross-check.
  • Account and customer writes require successful lock reads first; failures set lock/commit state flags.
  • Card update enforces active status values and month/year constraints before rewrite.
  • Transaction add validates amount format (-99999999.99) and date format (YYYY-MM-DD) before insertion.
  • Transaction add verifies account/card key relation through cross-reference files.
  • Bill payment inserts a payment transaction and reduces account current balance in same user flow.
  • User creation requires first name, last name, user ID, password, and user type.
  • User update rewrites only when at least one field value changed.
  • User delete requires existing record fetch before delete action.
  • PF7/PF8 list screens use browse start/read-next/read-prev semantics and explicit top/bottom messaging.
  • Optional menu entries display not-installed status when target program is absent.

8. Development Guidelines and Patterns

8.1 Pattern: entry/reentry state machine

  • On first entry: initialize map output and set cursor field.
  • On reentry: receive map, parse AID, dispatch action branch.
  • On completion: set commarea context and RETURN TRANSID for same transaction loop.

8.2 Pattern: list screens

  • Use STARTBR + READNEXT for forward page.
  • Use READPREV for backward page restoration.
  • Track first/last keys of displayed page in local commarea extension.
  • Set informational messages when top/bottom is reached.

8.3 Pattern: update screens

  • Fetch existing record and keep snapshot as old state.
  • Validate new state field-by-field and then cross-field.
  • On PF5 save, re-read with UPDATE and compare old/new to detect mid-air changes.
  • Rewrite record(s), and rollback transaction when secondary write fails.

8.4 Validation rule map (selected)

Context Validation Failure behavior
Account IDMust be numeric, 11 digits, non-zeroError message and cursor reset to account field

8.5 Maintenance checklist

  1. When adding a field, update: BMS map, program IO moves, copybook layout, and any batch JCL sort/repro logic.
  2. Preserve commarea size/offset compatibility when extending module-local state.
  3. If adding menu options, update option copybook and menu rendering bounds.
  4. Keep error message length aligned with map message field sizes.
  5. Verify both interactive and batch data paths after structural changes.

9. Usage Examples and Code Snippets

9.1 Example: navigate from sign-on to account view

Step 1: Start transaction CC00.
Step 2: Enter USER0001 / PASSWORDU (or environment-specific credentials).
Step 3: On CM00, choose option 1 (Account View).
Step 4: In CAVW, enter account ID and press Enter.
Step 5: Inspect account/customer detail pane.

9.2 Example: update account data with confirmation

Step 1: Open CAUP.
Step 2: Enter account ID and press Enter (details fetched).
Step 3: Edit fields (address, phone, status, limits).
Step 4: Press Enter to validate.
Step 5: Press PF5 to commit.
Step 6: If PF12 is used before PF5, changes are canceled.

9.3 Example: add transaction from CT02

Input account or card key.
Provide required fields: type, category, source, amount, merchant, dates.
Set Confirm = Y.
Press Enter.
Expected result: new transaction inserted and ID returned in message line.

9.4 Example: run full operational cycle

./scripts/run_full_batch.sh

This script submits close-refresh-post-open jobs in an opinionated order and waits between critical steps.

9.5 Example: key CICS read pattern

EXEC CICS READ
    DATASET(WS-USRSEC-FILE)
    INTO(SEC-USER-DATA)
    RIDFLD(SEC-USR-ID)
    KEYLENGTH(LENGTH OF SEC-USR-ID)
    RESP(WS-RESP-CD)
    RESP2(WS-REAS-CD)
END-EXEC.

9.6 Example: BMS send/receive pattern

EXEC CICS RECEIVE
    MAP('COUSR2A')
    MAPSET('COUSR02')
    INTO(COUSR2AI)
END-EXEC.

EXEC CICS SEND
    MAP('COUSR2A')
    MAPSET('COUSR02')
    FROM(COUSR2AO)
    ERASE
    CURSOR
END-EXEC.

10.2 Operational checklist

Use this as a quick runbook for validation before and after making changes in this module.

  • Validate sign-on and menu dispatch with admin and regular user IDs.
  • Validate account view and update flows with a known account present in cross-reference data.
  • Validate card list pagination (PF7/PF8) and update path from row selection.
  • Validate transaction list/detail/add and ensure inserted transaction is browseable.
  • Validate bill payment path updates transaction history and account balance.
  • Validate user list/create/update/delete in admin menu.
  • Run posting/refresh jobs and verify CICS files are reopened after batch window.
  • Verify no record-layout mismatch after copybook changes.
  • Verify no map field length mismatch after BMS changes.
  • Verify top/bottom page messaging on every list screen.
