Core Application

📦 User Story Development Guide

Modernization anchor for account, card, transaction, payment, and admin screens.

📋 Overview

Purpose: Keep the legacy green-screen workflows airtight while enabling Product Owners to scope stories that exercise sign-on, account/card management, transaction history, statements, and admin CRUD.

Business Context: This is the primary demo surface for finance operators and analysts exploring migration patterns—every modernization US touches these flows before optional extensions run.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Visibility: As a cardholder, I want to scroll through `CT00` so I can reconcile purchases before a statement posts.
  • Card Management: As an operations analyst, I want to deactivate a card via `CCUP` so we can prevent further usage immediately.
  • Admin Workflow: As an admin, I want to provision a new user (`CU01`) with the right RACF role so they can run posting and reporting jobs.

⚡ Development Acceleration Factors

  • BMS + COBOL pairs: Maps and programs live in `app/bms/` and `app/cbl/`, so you can reuse `COACTVW` patterns.
  • Copybooks: Shared data structures live under `app/cpy/` (e.g., `CUSTOMER-RECORD`, `DALYTRAN-RECORD`)—align new fields with those definitions.
  • Sample data & jobs: The `app/jcl/` directory hosts refresh jobs (`ACCTFILE`, `TRANBKP`), and `scripts/run_full_batch.sh` chains them for quick smoke testing.
  • Scheduler templates: CA-7/Control-M definitions in `app/scheduler/` show how to sequence nightly runs.

📊 Complexity Guidelines

  • Simple (1-2 pts): Adjust a BMS label, update validation text, or surface a new message via `COBIL00C` (no dataset change).
  • Medium (3-5 pts): Add a new account field (update copybook + VSAM layout) or augment `CAUP` with additional validation logic.
  • Complex (5-8 pts): Introduce a new transactional field that flows through `CT00`/`POSTTRAN` and requires a VSAM restructure plus JCL job update.

🔧 Technical Foundation

🏗️ Key Components

  • Sign-on/Menu: `COSGN00C` (CC00) routes to `COMEN01C`, the hub for user navigation.
  • Account/Card Layers: `COACTVW`/`COACTUP` and `COCRDLI`/`COCRDSLC` manage read/write access to VSAM accounts/cards.
  • Transaction Views: `COTRN00C`-series, `CORPT00C`, `CBIL00C` tie to `DALYTRAN` datasets and statement logic.
  • Admin CRUD: `CA00`, `CU00`-`CU03` for user maintenance and `CA00` menu gating.
  • Batch Jobs: `POSTTRAN`, `CREASTMT`, `TRANIDX`, `OPENFIL` executed via `app/jcl/` and optionally through `scripts/run_full_batch.sh`.

🔗 Dependencies

  • CICS region, BMS maps, and copybooks (`app/cpy`) provide the runtime plumbing.
  • VSAM datasets (`app/data/*`) store accounts, cards, customers, transactions, and security records.
  • JCL job streams (`app/jcl/`) refresh datasets and run batch programs.
  • RACF controls access, referencing the HLQ-named datasets defined in `README.md`.

🌐 Public Interfaces

CC00 (COSGN00C)       - Sign-on and menu routing
CM00 (COMEN01C)       - Main menu
CAVW/CAUP (COACTVW/C)  - Account view/update
CCLI/CCDL/CCUP         - Card list/view/update flows
CT00-CT02              - Transaction list/detail/add
CB00                   - Bill payment
CA00/CU00-CU03         - Admin menu, user CRUD
POSTTRAN, CREASTMT, TRANIDX, OPENFIL (batch jobs)

✅ Acceptance Criteria Examples

  • Users reach `CM00` only after RACF validates credentials stored in `AWS.M2.CARDDEMO.USRSEC.PS`.
  • `CAUP` writes updated fields into `CVACT01Y`, maintaining field lengths from the existing copybook.
  • `POSTTRAN` finishes within the nightly budget, and `scripts/run_full_batch.sh` reports success for each individual JCL submission.
