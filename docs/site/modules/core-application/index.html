1. Overview & Purpose

The core-application module is the foundational credit-card management system of the CardDemo platform. It delivers all primary interactive flows — sign-on, account/card management, transaction history, bill payments, statement generation, and administrative user maintenance — implemented as CICS COBOL programs rendered on IBM 3270 green-screen terminals via BMS mapsets.

This module is the anchor for all modernization scenarios. Every optional extension module (authorization, transaction-type, account-extractions) depends on the VSAM datasets and COBOL programs established here. It targets the IBM z/OS CICS runtime and interacts exclusively with VSAM KSDS files for persistence.

Modernization Note: When migrating to a modern platform (AWS Mainframe Modernization / Micro Focus), this module's COBOL programs, BMS maps, copybooks, and VSAM datasets are the primary migration targets. The batch JCL jobs are candidates for AWS Batch or Step Functions orchestration.

Interaction Pattern

All interactive flows follow the CICS pseudo-conversational pattern:

1
Terminal issues a transaction ID (e.g., CC00, CT00).
2
CICS dispatches the associated COBOL program; program reads DFHCOMMAREA context.
3
Program validates input, performs VSAM I/O, and issues EXEC CICS SEND MAPSET.
4
CICS returns control to terminal; program terminates (pseudo-conversational).
5
Next PF key press re-enters the program with updated COMMAREA and map data.

3. Key Components

3.1 CICS Online Programs (app/cbl/)

COSGN00C — Sign-On

Transaction: CC00

Presents the CardDemo sign-on screen (COSGN00 BMS map). Validates user ID and password against the VSAM security file (USRSEC). On success, populates CARDDEMO-COMMAREA with user type (A=Admin, U=User) and routes to CM00 or CA00.

Files: app/cbl/COSGN00C.cbl, app/bms/COSGN00.bms

COMEN01C — Main Menu

Transaction: CM00

Hub for regular user navigation. Displays numbered menu options that map to account, card, transaction, billing, and reporting flows. Reads COMMAREA for the authenticated user context and enforces role-based branching.

Files: app/cbl/COMEN01C.cbl, app/bms/COMEN01.bms

COACTVWC — Account View

Transaction: CAVW

Reads account and cross-reference records from VSAM. Accepts an account ID or card number, looks up the cross-reference file (CXACAIX), then retrieves the full ACCOUNT-RECORD from ACCTDAT. Supports both admin (all accounts) and user (own account only) contexts.

Files: app/cbl/COACTVWC.cbl, app/bms/COACTVW.bms

COACTUPC — Account Update

Transaction: CAUP

Full-featured account update screen. Validates mandatory and optional fields using COBOL EVALUATE and 88-level condition names, then rewrites the ACCOUNT-RECORD in the VSAM KSDS. Enforces ownership — only the matching user or admin can update.

Files: app/cbl/COACTUPC.cbl, app/bms/COACTUP.bms

COCRDLIC — Card List

Transaction: CCLI

Lists credit cards associated with an account. Admin users see all cards; regular users see only cards linked to their account ID. Supports PF7/PF8 paging and S-selector for card detail navigation.

Files: app/cbl/COCRDLIC.cbl, app/bms/COCRDLI.bms

COCRDSLC — Card Select

Transaction: CCSL

Handles the single-card selection from the card list, passing the card number and account context in COMMAREA to downstream update/detail programs.

Files: app/cbl/COCRDSLC.cbl, app/bms/COCRDSL.bms

COCRDUPC — Card Update

Transaction: CCUP

Updates card attributes (embossed name, active status, expiration date) in the CARDDAT VSAM file. Validates card number format and active-status values via copybook condition names.

Files: app/cbl/COCRDUPC.cbl, app/bms/COCRDUP.bms

COTRN00C — Transaction List

Transaction: CT00

Reads from the TRANSACT VSAM KSDS and displays paginated transaction rows. Each row shows transaction ID, type code, amount, and date. PF7/PF8 navigate pages; S-selector forwards to CT01 detail.

Files: app/cbl/COTRN00C.cbl, app/bms/COTRN00.bms

COTRN01C — Transaction Detail

Transaction: CT01

Shows the full detail of a single transaction record from the TRANSACT file, including merchant name, city, ZIP, original and processing timestamps, and category code.

Files: app/cbl/COTRN01C.cbl, app/bms/COTRN01.bms

COTRN02C — Transaction Add

Transaction: CT02

Adds a new transaction to the TRANSACT VSAM file. Generates a 16-character transaction ID, validates the card number via CARDXREF, and writes the full DALYTRAN-RECORD.

Files: app/cbl/COTRN02C.cbl, app/bms/COTRN02.bms

COBIL00C — Bill Payment

Transaction: CB00

Processes online bill payments. Reads the current account balance from ACCTDAT via the cross-reference index (CXACAIX), confirms payment amount with the user, writes a payment transaction to TRANSACT, and updates the account's current balance.

Files: app/cbl/COBIL00C.cbl, app/bms/COBIL00.bms

COADM01C — Admin Menu

Transaction: CA00

Entry point for administrative users. Provides numbered options routing to user management functions (CU00–CU03). Reads COMMAREA user type to enforce admin-only access — non-admin users receive an error if they attempt to access this transaction directly.

Files: app/cbl/COADM01C.cbl, app/bms/COADM01.bms

COUSR00C — User List

Transaction: CU00

Lists all users from the USRSEC VSAM security file. Displays user ID, first/last name, and user type. Supports 10-record paged display with PF7/PF8 navigation and S-selector for edit/delete.

Files: app/cbl/COUSR00C.cbl, app/bms/COUSR00.bms

COUSR01C / COUSR02C / COUSR03C

Transactions: CU01 / CU02 / CU03

Create, update, and delete user records in the USRSEC VSAM file. Write/rewrite/delete operations use the SEC-USER-DATA structure from copybook CSUSR01Y. User type (A/U) enforces downstream RACF-equivalent authorization.

Files: app/cbl/COUSR01C.cbl, app/cbl/COUSR02C.cbl, app/cbl/COUSR03C.cbl

CORPT00C — Reports

Transaction: CR00

Provides a reporting/inquiry screen for transaction summaries. Acts as a gateway to statement and reporting outputs generated by the batch tier.

Files: app/cbl/CORPT00C.cbl, app/bms/CORPT00.bms

CSUTLDTC — Date/Utility

Utility COBOL program providing date calculation and formatting services shared across the CICS programs. Called via EXEC CICS LINK to convert dates to display formats.

Files: app/cbl/CSUTLDTC.cbl

4. Public APIs & Interfaces

In the CICS environment, "APIs" are CICS transaction IDs. External programs communicate via DFHCOMMAREA (the CARDDEMO-COMMAREA defined in COCOM01Y.cpy).

4.1 Online CICS Transactions

CC00 — Sign-On (Program: COSGN00C)
Entry point for all users. Accepts User ID + Password, validates against USRSEC VSAM, populates COMMAREA, transfers to CM00 (user) or CA00 (admin). Invoked with EIBCALEN = 0 on first entry.
* Input: COSGN0AI map fields: USERIDI, PASSWDI
* Output: Routes to COMEN01C or COADM01C via EXEC CICS XCTL
* COMMAREA fields set: CDEMO-USER-ID, CDEMO-USER-TYPE, CDEMO-FROM-TRANID
CM00 — Main Menu (Program: COMEN01C)
Regular-user hub. Routes to account view (option 1), card list (option 2), transaction list (option 3), bill payment (option 4), reports (option 5). Pressing PF3 returns to sign-on.
CAVW — Account View (Program: COACTVWC)
Read account master. Input: account ID or card number. Reads XREF file to resolve card→account, then fetches ACCOUNT-RECORD from ACCTDAT. Output: displays all account fields on COACTVW BMS map.
* Key COMMAREA inputs: CDEMO-ACCT-ID or CDEMO-CARD-NUM
* Key file I/O:
*   CXACAIX (alternate index, key=card number) → ACCT-ID
*   ACCTDAT (KSDS, key=ACCT-ID) → ACCOUNT-RECORD
CAUP — Account Update (Program: COACTUPC)
Write account master. Validates all editable fields (credit limits, zip code, group ID, status) before REWRITE to ACCTDAT. Returns error messages on BMS message line for each failing validation rule.
CCLI — Card List (Program: COCRDLIC)
Paginated list of cards. Admin sees all cards from CARDDAT; regular user sees only cards where CARD-ACCT-ID matches session account. PF7=prev page, PF8=next page, S=select for update.
CCUP — Card Update (Program: COCRDUPC)
Update card record in CARDDAT VSAM. Editable fields: CARD-EMBOSSED-NAME, CARD-ACTIVE-STATUS, CARD-EXPIRAION-DATE. Validates card number length (16 digits) and active status (Y/N).
CT00 / CT01 / CT02 — Transaction List / Detail / Add (COTRN00C / 01C / 02C)
Full transaction lifecycle: list transactions from TRANSACT KSDS (CT00), drill into detail view (CT01), add new transaction (CT02). CT02 generates a 16-char TRAN-ID and validates card number via CARDXREF before writing to TRANSACT.
* CT02 key validations:
*   DALYTRAN-CARD-NUM  must resolve via XREFFILE
*   DALYTRAN-AMT       must be numeric, non-zero
*   DALYTRAN-TYPE-CD   2-char alphanumeric
*   DALYTRAN-CAT-CD    4-digit category code
CB00 — Bill Payment (Program: COBIL00C)
Online bill payment. Looks up account via CXACAIX, presents current balance, accepts payment amount, writes a payment transaction record to TRANSACT, and updates ACCT-CURR-BAL in ACCTDAT. Two-step confirmation required (enter once to review, enter again to confirm).
* Files used: TRANSACT (write), ACCTDAT (read/rewrite), CXACAIX (read)
* Transaction type code written: 'PY' (payment)
* Balance update: ACCT-CURR-BAL = ACCT-CURR-BAL - payment amount
CA00 / CU00–CU03 — Admin & User Management (COADM01C, COUSRnnC)
Administrative user lifecycle. CU00 lists users from USRSEC; CU01 adds new user (WRITE); CU02 updates existing user (REWRITE); CU03 deletes user (DELETE). All operations restricted to CDEMO-USRTYP-ADMIN sessions.

6. Business Rules

Authentication & Authorization

  • All sessions must authenticate via CC00; programs check EIBCALEN and redirect to sign-on if COMMAREA is absent.
  • Admin functions (CA00, CU00–CU03) verify CDEMO-USRTYP-ADMIN in COMMAREA; non-admin users receive an error.
  • Default demo credentials stored in AWS.M2.CARDDEMO.USRSEC.PS: ADMIN001/PASSWORD (admin), USER0001/PASSWORD (regular user).

Account Ownership

  • COACTUPC verifies the requesting user ID matches the account before writing. Admins bypass this check.
  • Card list (CCLI) filters to account-owned cards for non-admin users using the cross-reference file.
  • Bill payment (CB00) resolves account from the card number in session context, not from user input alone.

Transaction Processing

  • Transaction IDs are 16 characters; CT02 generates them using CICS ABSTIME + account context to ensure uniqueness.
  • Only transactions with valid card numbers resolvable via CARDXREF are accepted by CT02.
  • Type codes are 2-character alphanumeric: PU=Purchase, PY=Payment, others as defined in transaction-type module.
  • Category codes (DALYTRAN-CAT-CD) are 4-digit; values sourced from the transaction-type DB2 module when installed.

Batch Consistency

  • POSTTRAN must run after dataset refresh jobs (ACCTFILE, TRANBKP) to post against current data.
  • CREASTMT depends on POSTTRAN completing so statements reflect the latest transaction balances.
  • CICS files must be closed via CLOSEFIL JCL before any VSAM REPRO or DEFINE CLUSTER operations.
  • TCATBALF VSAM is updated by CBTRN02C and must be refreshed before reporting queries.

Validation Rules

  • ZIP code: ACCT-ADDR-ZIP must be 10 characters; CUST-ADDR-ZIP follows the same constraint from copybook definition.
  • Credit limit: ACCT-CREDIT-LIMIT must be positive; ACCT-CASH-CREDIT-LIMIT must not exceed total credit limit.
  • FICO score: CUST-FICO-CREDIT-SCORE range 300–850 (PIC 9(03)).
  • Dates: Validated using CSUTLDTC utility program; must be YYYY-MM-DD format.

8.2 Nightly Batch Sequence (scripts/run_full_batch.sh)

The shell script submits JCL jobs via FTP to JES2 in this order:

1. CLOSEFIL   — Close CICS files
2. ACCTFILE   — Refresh Account VSAM
3. CARDFILE   — Refresh Card VSAM
4. XREFFILE   — Refresh Card Cross-Reference VSAM
5. CUSTFILE   — Refresh Customer VSAM
6. TRANBKP    — Backup + Refresh Transaction VSAM
7. POSTTRAN   — Post daily transactions
8. CREASTMT   — Generate statements
9. TRANIDX    — Rebuild transaction indexes
10. OPENFIL   — Reopen CICS files
Prerequisite: An FTP tunnel to the JES2 port (default: 2121) must be active before running run_full_batch.sh. The script checks for the tunnel with ps f | grep -c "2121:".

10. Development Guidelines & Patterns

Adding a New Field to an Existing Screen

1
Update the relevant copybook in app/cpy/ (e.g., add a field to CVACT01Y.cpy). Verify total record length still matches the VSAM RECORDSIZE definition.
2
Add the corresponding BMS field in app/bms/ map source. Reassemble the mapset.
3
Update the COBOL program to MOVE/validate the new field; use existing 88-level condition names for validation flags.
4
If the field is persistent, update the VSAM DEFINE CLUSTER in the relevant app/jcl/*.jcl file with the new RECORDSIZE, then reload data via the refresh JCL.
5
Run scripts/run_full_batch.sh after reloading data to verify posting still completes cleanly.

Pseudo-Conversational Pattern

* Standard CICS program entry pattern used in all online programs:
MAIN-PARA.
    IF EIBCALEN = 0
        * First invocation - send map with empty fields
        PERFORM SEND-SCREEN-INITIAL
    ELSE
        * Re-entry after user interaction
        MOVE DFHCOMMAREA TO WS-COMMAREA
        PERFORM PROCESS-INPUT
        PERFORM VALIDATE-FIELDS
        IF NOT ERR-FLG-ON
            PERFORM UPDATE-VSAM-FILE
        END-IF
        PERFORM SEND-SCREEN-WITH-RESULTS
    END-IF.

    EXEC CICS RETURN
        TRANSID(WS-TRANID)
        COMMAREA(WS-COMMAREA)
        LENGTH(LENGTH OF WS-COMMAREA)
    END-EXEC.

Error Handling Pattern

* Standard error handling used across all programs:
HANDLE-VSAM-ERROR.
    EVALUATE WS-RESP-CD
        WHEN DFHRESP(NORMAL)
            CONTINUE
        WHEN DFHRESP(NOTFND)
            MOVE 'Record not found' TO WS-MESSAGE
            SET ERR-FLG-ON TO TRUE
        WHEN DFHRESP(DUPKEY)
            MOVE 'Duplicate key error' TO WS-MESSAGE
            SET ERR-FLG-ON TO TRUE
        WHEN OTHER
            MOVE 'Unexpected CICS error' TO WS-MESSAGE
            SET ERR-FLG-ON TO TRUE
    END-EVALUATE.

11. Usage Examples & Code Snippets

Example: XCTL (Transfer Control) Between Programs

* From COSGN00C - transfer to main menu after successful signon:
MOVE 'COMEN01C' TO CDEMO-TO-PROGRAM
MOVE 'CM00'     TO CDEMO-TO-TRANID

EXEC CICS XCTL
    PROGRAM(CDEMO-TO-PROGRAM)
    COMMAREA(CARDDEMO-COMMAREA)
    LENGTH(LENGTH OF CARDDEMO-COMMAREA)
END-EXEC.

Example: VSAM Read via Alternate Index

* From COACTVWC - look up account by card number via CXACAIX:
MOVE CC-CARD-NUM TO WS-CARD-RID-CARDNUM

EXEC CICS READ
    FILE('CXACAIX')
    INTO(CARD-XREF-RECORD)
    RIDFLD(WS-CARD-RID-CARDNUM)
    RESP(WS-RESP-CD)
    RESP2(WS-REAS-CD)
END-EXEC

IF WS-RESP-CD = DFHRESP(NORMAL)
    MOVE XREF-ACCT-ID TO CDEMO-ACCT-ID
    PERFORM READ-ACCOUNT-RECORD
ELSE
    MOVE 'Card number not found' TO WS-MESSAGE
    SET ERR-FLG-ON TO TRUE
END-IF.

Example: Paginated Browse of TRANSACT VSAM

* From COTRN00C - browse transaction file:
EXEC CICS STARTBR
    FILE('TRANSACT')
    RIDFLD(WS-TRAN-ID-FILTER)
    RESP(WS-RESP-CD)
END-EXEC

PERFORM VARYING WS-IDX FROM 1 BY 1
    UNTIL WS-IDX > 10 OR TRANSACT-EOF
    EXEC CICS READNEXT
        FILE('TRANSACT')
        INTO(WS-TRAN-RECORD)
        RIDFLD(WS-TRAN-ID-FILTER)
        RESP(WS-RESP-CD)
    END-EXEC
    IF WS-RESP-CD = DFHRESP(ENDFILE)
        SET TRANSACT-EOF TO TRUE
    ELSE
        PERFORM POPULATE-SCREEN-ROW
    END-IF
END-PERFORM

EXEC CICS ENDBR FILE('TRANSACT') END-EXEC.

Example: Bill Payment Update

* From COBIL00C - update account balance after payment:
COMPUTE ACCT-CURR-BAL =
    ACCT-CURR-BAL - WS-PAYMENT-AMT

EXEC CICS REWRITE
    FILE('ACCTDAT')
    FROM(ACCOUNT-RECORD)
    LENGTH(LENGTH OF ACCOUNT-RECORD)
    RESP(WS-RESP-CD)
END-EXEC

IF WS-RESP-CD = DFHRESP(NORMAL)
    MOVE 'Payment processed successfully' TO WS-MESSAGE
ELSE
    MOVE 'Payment update failed - contact admin' TO WS-MESSAGE
    SET ERR-FLG-ON TO TRUE
END-IF.
