🏦 Accounts

User Story Development Guide — CardDemo Mainframe Application

📋 Overview

Module ID: accounts

Purpose: View and update credit card account information including current balance, credit limit, cash credit limit, account dates, group ID, and status.

Business Context: Central entity in CardDemo — every card and transaction links back to an account. Account balances are updated by batch POSTTRAN; online view reflects the last batch-processed state.

Transaction CAVW: COACTVWC Transaction CAUP: COACTUPC VSAM: ACCTFILE (KSDS) Copybook: CVACT01Y

🎯 User Story Development Guide

�� US Templates

  • As a regular user, I want to view my account details so that I can see my current balance and available credit
  • As a regular user, I want to view my account open and expiration dates so that I know when my account was established
  • As a regular user, I want to update my account ZIP code so that my billing address stays current
  • As a regular user, I want to see my credit limit and cash credit limit so that I can plan my spending

⚡ Development Acceleration Factors

  • COACTVWC: Reference implementation for VSAM account read + BMS display
  • CVACT01Y copybook: Complete 300-byte account record layout ready to use
  • COMMAREA pattern: User context (user ID, user type) passed from authentication COMMAREA
  • COACTUP/COACTVW BMS maps: Screen layouts defined — field names available in generated copybooks

📊 Complexity Guidelines

  • Simple (1-2 pts): Display account view from single VSAM KSDS read (CAVW)
  • Medium (3-5 pts): Account update with field validation, VSAM REWRITE (CAUP)
  • Complex (5-8 pts): Cross-module impact — e.g., account status change affecting card and transaction eligibility

📋 Account Data Model

01  ACCOUNT-RECORD.               -- 300 bytes, VSAM KSDS
    05  ACCT-ID                PIC 9(11).      -- Primary key
    05  ACCT-ACTIVE-STATUS     PIC X(01).      -- 'Y'=active
    05  ACCT-CURR-BAL          PIC S9(10)V99.  -- Current balance
    05  ACCT-CREDIT-LIMIT      PIC S9(10)V99.  -- Credit limit
    05  ACCT-CASH-CREDIT-LIMIT PIC S9(10)V99.  -- Cash limit
    05  ACCT-OPEN-DATE         PIC X(10).      -- YYYY-MM-DD
    05  ACCT-EXPIRAION-DATE    PIC X(10).      -- YYYY-MM-DD (note: typo in source)
    05  ACCT-REISSUE-DATE      PIC X(10).      -- YYYY-MM-DD
    05  ACCT-CURR-CYC-CREDIT   PIC S9(10)V99.
    05  ACCT-CURR-CYC-DEBIT    PIC S9(10)V99.
    05  ACCT-ADDR-ZIP          PIC X(10).
    05  ACCT-GROUP-ID          PIC X(10).      -- Links to DISCGRP interest rates
    05  FILLER                 PIC X(178).

⚙️ Business Rules

  • Account read requires ACCT-ID from user context (COMMAREA) or direct entry
  • NOTFND response from VSAM READ → display "Account not found" message
  • ACCT-ACTIVE-STATUS='Y' required for normal operations
  • ACCT-CURR-BAL updated only by batch POSTTRAN — online value may lag by 1 cycle
  • ACCT-GROUP-ID determines applicable interest rates in DISCGRP file
  • Field-level validation required on update (numeric fields, date formats)

✅ Acceptance Criteria Patterns

  • Given a valid account ID, when account view is requested, then all account fields are displayed including current balance and credit limit
  • Given an invalid account ID, when account view is requested, then an error message "Account not found" is displayed
  • Given valid updated values, when account update is submitted, then the record is rewritten to ACCTFILE and a success message is displayed
  • Given non-numeric data in a numeric field, when update is submitted, then a field-level error is displayed without updating VSAM
