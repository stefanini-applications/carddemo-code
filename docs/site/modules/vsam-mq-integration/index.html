📨 VSAM/MQ Integration

User Story Development Guide — Account & Date Inquiry via IBM MQ

← Back to Documentation Index
⚠️ Optional Module — Requires IBM MQ infrastructure

Module ID: vsam-mq-integration

Purpose: Demonstrates asynchronous MQ request/response patterns for external system access to CardDemo data. Provides account data inquiry (CDRA) and system date inquiry (CDRD) via MQ messages, reading VSAM data and returning results on reply queues.

🎯 User Story Development Guide

📝 US Templates

  • Account Inquiry via MQ: As an external system, I want to query account details via MQ so that I can access account data without direct VSAM access.
  • Date Inquiry via MQ: As an external system, I want to query the mainframe system date via MQ so that I have an authoritative timestamp.
  • Async Integration: As a system integrator, I want to use MQ request/response so that downstream systems can access CardDemo data asynchronously.

📊 Complexity Guidelines

1–2 pts
Simple
  • Add additional account field to MQ reply message
  • Change MQ queue name configuration
3–5 pts

🔧 Technical Foundation

🏗️ Key Components

  • COACCT01 (CICS) — Account details inquiry via MQ (CDRA transaction); reads ACCTDAT VSAM, formats reply message, puts to MQ reply-to queue
  • CODATE01 (CICS) — System date inquiry via MQ (CDRD transaction); retrieves CICS date/time, puts to MQ reply-to queue

🔗 Dependencies

  • IBM MQ: Request and reply queues must be defined; CICS must be configured with MQ connection
  • ACCTDAT (VSAM KSDS): COACCT01 reads account record by account ID from MQ request message
  • CICS ABSTIME: CODATE01 uses CICS time services for authoritative system date

🌐 CICS Transactions

CDRD  → CODATE01  — System date inquiry via MQ (request/response)
CDRA  → COACCT01  — Account details inquiry via MQ (request/response)

📋 MQ Message Flow

Date Inquiry (CODATE01 / CDRD):
  1. MQ message arrives on CDRD request queue
  2. CODATE01 triggered by CICS MQ trigger mechanism
  3. CICS ABSTIME called for current date/time
  4. Reply message put to MQ reply-to queue (from MQMD)

Account Inquiry (COACCT01 / CDRA):
  1. MQ message arrives with account ID on CDRA queue
  2. COACCT01 triggered by CICS MQ trigger
  3. ACCTDAT read using account ID from message
  4. Account details formatted into reply message
  5. Reply put to MQ reply-to queue

📋 Business Rules

  • BR-MQ-01: MQ correlation ID from request message must be copied to reply message
  • BR-MQ-02: Reply-to queue name is taken from MQMD (MQ Message Descriptor) of the incoming request
  • BR-MQ-03: If account is not found, an appropriate error code is returned in the reply
  • BR-MQ-04: Date format in reply uses CICS standard date format

✅ Acceptance Criteria Examples

  • Date Inquiry: Given a properly formatted MQ message on the CDRD queue, when CODATE01 processes it, then the current mainframe date/time is returned on the reply-to queue with the original correlation ID
  • Account Inquiry: Given a valid account ID in the MQ request message, when COACCT01 processes it, then account details (balance, limits, status) are returned on the reply-to queue
  • Account Not Found: Given an unknown account ID in the request, when COACCT01 processes it, then an error code is returned in the reply message
  • Performance: MQ request/response round trip must complete within 5 seconds
