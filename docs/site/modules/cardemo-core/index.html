📦 CardDemo Core

Guide for Developing User Stories

📋 Overview

Purpose: The base CardDemo module powers sign-on, account/card management, transactions, statements, and the administrator menu inside CICS.

Business Context: Provides the working data store and workflow for credit card lifecycle management that optional extensions build upon (transaction types, authorizations, MQ extracts).

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Account Management: As a cardholder, I want to view and edit my contact information so that future statements land at my preferred address.
  • Transaction Posting: As an operator, I want to run the nightly posting job and see clean JES output so that daily balances reflect today’s activity.
  • Admin Maintenance: As an admin, I want to manage user records (CU00-CU03) to ensure support staff can log in with appropriate permissions.

⚡ Development Acceleration Factors

  • BMS map reuse: Most screens follow the PF-key pattern (Enter, PF3/PF12, PF7/PF8) so new forms can copy structure from `app/bms`/`app/cpy-bms`.
  • Copybooks: Data layouts such as `CVACT01Y` and `CVTRAN06Y` are shared across `cbl` programs, reducing duplication when adding fields.
  • Scripts: `scripts/run_full_batch.sh` sequences dataset refreshes and posting jobs, so story acceptance can include re-running the script rather than manually submitting multiple JCL files.

📊 Complexity Guidelines

  • Simple (1-2 pts): CRUD changes on an existing BMS screen or validation rule within a single COBOL program (e.g., add a warning message on `COACTVW`).
  • Medium (3-5 pts): Add new business logic that affects multiple transactions or update copybooks plus maps (e.g., new date format in account update plus `ACCTFILE` refresh).
  • Complex (5-8 pts): Introduce a new dataset or modify the batch pipeline (e.g., new sequence after `TRANBKP` that reorganizes VSAM indexes or integrates DB2 metadata).

✅ Acceptance Criteria Examples

  • Sign-on (`CC00`) validates credentials and routes to `CM00` within one CICS transaction; rejects invalid passwords with descriptive BMS message.
  • `COACTUPC` persists updates to `AWS.M2.CARDDEMO.ACCTDATA.PS` and leaves the user on the detail screen with a success banner.
  • Running `scripts/run_full_batch.sh` replays the 16-job sequence, waits 5–10 seconds between submissions, and ends with JES return code 0000 for each job.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs (`app/cbl/`): Each transaction has its own entrypoint (e.g., `COACTVW`, `COCRDLIC`, `CORPT00`).
  • BMS mapsets (`app/bms/`, `app/cpy-bms/`): Define 3270 screens rendered inside CICS.
  • VSAM datasets (`app/data/`): `AWS.M2.CARDDEMO.ACCTDATA.PS`, `CARDDATA.PS`, `DALYTRAN.PS`, and supporting cross-reference files.
  • Batch JCL (`app/jcl/`): `ACCTFILE`, `CARDFILE`, `TRANBKP`, `POSTTRAN`, `INTCALC`, `TRANIDX`, `OPENFIL`, among others.
  • Automation scripts: `scripts/run_full_batch.sh` for sequential JES submission and helper templates like `scripts/compile_batch.jcl.template`.

🔗 Dependencies

  • CICS region with the CardDemo group (`CARDDEMO`).
  • VSAM files provisioned with the provided HLQs (e.g., `AWS.M2.CARDDEMO.ACCTDATA.PS`).
  • JCL/PROCs configured to compile and execute COBOL/CICS jobs.
  • Optional MQ/DB2/IMS (when dependent modules are enabled).

🌐 Public Interfaces


CICS:
- CC00 (COSGN00C) Sign-on
- CM00 (COMEN01C) Main menu
- CAVW/CAUP (COACTVWC/COACTUPC) Account view/update
- CCLI/CCDL (COCRDLIC/COCRDSLC) Card list/detail
- CT00-CT02 (COTRN00C/COTRN01C/COTRN02C) Transaction lifecycle
- CB00 (COBIL00C) Bill payment
- CA00 (COADM01C) Admin menu
- CU00-CU03 (COUSR00C-COUSR03C) User management
Batch:
- ACCTFILE, CARDFILE, TRANBKP, POSTTRAN, INTCALC, TRANIDX, OPENFIL, WAITSTEP, TRANEXTR
