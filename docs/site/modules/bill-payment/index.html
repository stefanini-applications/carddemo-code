💸 Bill Payment

User Story Development Guide · CardDemo System

📋 Module Overview

Purpose: Process customer bill payments and apply them to account balances. Creates a corresponding transaction record and updates the account balance immediately.

Business Context: When customers call to make a payment by phone, CSRs use this screen to process the payment in real time. The payment immediately reduces the account balance and creates an auditable transaction record.

Transaction ID: CB00

Key Programs: COBIL00C LNKBILI LNKLBIL

VSAM Datasets: ACCTDATA.VSAM.KSDS, TRANSACT.VSAM.KSDS

🎯 User Story Development Guide

📝 US Templates

As a CSR, I want to process a customer's phone payment so that their account balance is updated immediately.
As a CSR, I want the payment date to default to today so that I don't need to manually enter the current date for standard payments.
As a CSR, I want to see the current balance before posting a payment so that I can confirm the payment does not exceed the outstanding balance.

✅ Acceptance Criteria Patterns

Payment Processing (CB00):
  • Payment amount must be positive numeric; reject non-numeric or zero amounts
  • Payment must not exceed current outstanding balance; display error if over-payment attempted
  • Account balance in ACCTDATA VSAM reduced by payment amount on successful posting
  • Transaction record written to TRANSACT VSAM with payment transaction type code
  • Payment date defaults to system date; must be in YYYY-MM-DD format if overridden
  • "PAYMENT PROCESSED SUCCESSFULLY" shown in GREEN on success
  • PF3 must discard input and return to Main Menu

🔧 Technical Foundation

🏗️ Key Components

  • COBIL00C: Main bill payment CICS program — reads account, validates payment, calls linkage programs, updates ACCTDATA VSAM and TRANSACT VSAM
  • LNKBILI / LNKLBIL: Linkage section programs for billing calculations (interest/timespan calculations using assembler COBDATFT)
  • LNKBILI: Called via EXEC CICS LINK; handles billing date and timespan logic

🔗 Dependencies

  • account-management: ACCTDATA VSAM must have a valid record for the account ID
  • transaction-management: Payment creates a record in TRANSACT VSAM; transaction type code for payment must exist in TRANTYPE
  • batch-processing: INTCALC batch reads balances updated by CB00; avoid concurrent access
