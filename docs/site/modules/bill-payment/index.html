💵 Bill Payment

User Story Development Guide — Account Balance Payment

← Back to Documentation Index

Module ID: bill-payment

Purpose: Enables card holders to pay their credit card bill. The payment reduces the current account balance and records a payment transaction in the TRANSACT VSAM file.

🎯 User Story Development Guide

📝 US Templates

  • View Balance: As a card holder, I want to see my current balance before paying so that I know the amount due.
  • Pay Bill: As a card holder, I want to pay my full outstanding balance so that I avoid interest charges.
  • Confirm Payment: As a card holder, I want to confirm my payment before it is processed so that I can avoid mistakes.
  • Payment Receipt: As a card holder, I want a transaction ID generated for my payment so that I have a record of the transaction.
  • Balance Update: As a system, I want to update the account balance immediately after payment so that the balance reflects the payment.

⚡ Development Acceleration Factors

  • COBIL00C payment flow: Two-step confirmation pattern (enter → confirm → post) already implemented
  • CVACT01Y + CVACT03Y: Account lookup via card number using existing CXACAIX AIX pattern
  • CVTRA05Y: Transaction record structure reused — payment is written as a standard TRAN-RECORD
  • Timestamp-based TRAN-ID: Transaction ID generation from system time established in COBIL00C

🔧 Technical Foundation

🏗️ Key Components

  • COBIL00C — Bill payment program; reads account via CXACAIX, generates TRAN-ID, posts to TRANSACT, updates ACCTDAT
  • COBIL00 (BMS map) — Bill payment 3270 screen; displays account balance, confirmation prompt
  • CVACT01Y (copybook) — Account record (reads ACCTDAT for current balance)
  • CVACT03Y (copybook) — Card-account XREF (resolves card → account)
  • CVTRA05Y (copybook) — Transaction record (payment written as TRAN-RECORD)

🔗 Dependencies

  • TRANSACT (KSDS): Payment transaction written here
  • ACCTDAT (KSDS): ACCT-CURR-BAL decremented by payment amount
  • CXACAIX (AIX): Resolves card number → account ID for balance lookup
  • Transaction Processing module: Same TRANSACT file and TRAN-RECORD structure
  • Account Management module: Account record structure and balance field

🌐 CICS Transactions

CB00  → COBIL00C  — Bill payment (two-step: display → confirm → post)

📋 Payment Flow

1. User navigates to CB00
2. COBIL00C reads account via CXACAIX (card→account lookup)
3. Current balance displayed on COBIL00 screen
4. User confirms payment (ENTER key on confirmation prompt)
5. COBIL00C generates TRAN-ID from ABSTIME timestamp
6. Payment TRAN-RECORD written to TRANSACT
7. ACCTDAT updated: ACCT-CURR-BAL reduced by payment amount
8. Confirmation message displayed to user

📋 Business Rules

  • BR-BILL-01: Bill payment posts a full-balance payment transaction to TRANSACT
  • BR-BILL-02: Payment requires user confirmation before being written (two-step: display → confirm)
  • BR-BILL-03: TRAN-ID is generated from the CICS ABSTIME clock ensuring uniqueness
  • BR-BILL-04: Payment reduces ACCT-CURR-BAL in ACCTDAT immediately upon confirmation
  • BR-BILL-05: Payment transaction uses a specific TRAN-TYPE-CD to identify it as a payment

✅ Acceptance Criteria Examples

  • Balance Display: Given an authenticated user navigates to CB00, then the current outstanding balance is displayed on the payment screen
  • Confirmation Required: Given the payment screen is displayed, when the user presses ENTER without confirming, then a confirmation prompt is shown before the payment is posted
  • Payment Posted: Given payment is confirmed, then a payment TRAN-RECORD is written to TRANSACT and ACCT-CURR-BAL is reduced to zero (full payment)
  • Transaction ID: Given payment is completed, then a unique 16-character TRAN-ID is assigned and displayed
  • Performance: Payment confirmation and posting must complete within 3 seconds
