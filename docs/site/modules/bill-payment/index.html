🧾 Bill Payment

User Story Development Guide — Credit Card Bill Payment Processing

📋 Overview

Module ID: bill-payment

Purpose: Processes bill payments that reduce the outstanding balance on a credit card account. Payments are recorded as credit transactions in the TRANSACT VSAM file and update the ACCTFILE balance.

Business Context: Bill payment is the primary mechanism for cardholders to reduce their balance. It must validate account status before accepting payment, and record the payment as a transaction for complete audit trail.

Entry Point: CICS transaction CB00 → program COBIL00C

🎯 User Story Templates

Make a Bill Payment

As a cardholder, I want to make a payment toward my credit card balance so that my outstanding balance is reduced.
Acceptance Criteria:
  • Transaction CB00 (COBIL00C) presents payment screen with account balance and payment amount field
  • System validates account is active (ACCT-ACTIVE-STATUS = 'Y') before accepting payment
  • Payment amount must be positive and not exceed current balance
  • After successful payment: ACCT-CURR-BAL decreased by payment amount in ACCTFILE
  • Payment recorded as TRAN-RECORD in TRANSACT VSAM with negative TRAN-AMT (credit)
  • ACCT-CURR-CYC-CREDIT incremented by payment amount
  • Confirmation message displayed with new balance

Payment Validation

As the system, I want to validate payment inputs before processing so that invalid payments are rejected with clear error messages.
Acceptance Criteria:
  • Payment amount of $0.00 is rejected with message "Payment amount must be greater than zero"
  • Payment on inactive account (ACCT-ACTIVE-STATUS != 'Y') is rejected with "Account is not active"
  • Non-numeric payment amount rejected with appropriate field error
  • On validation error, screen redisplays with entered data; no updates made to VSAM

Payment History Visibility

As a cardholder, I want to see my recent payments in the transaction history so that I can confirm payments have been applied.
Acceptance Criteria:
  • Payments appear in transaction list (CT00) with negative TRAN-AMT value
  • TRAN-DESC for payments includes payment reference
  • Payment TRAN-TYPE-CD distinguishes payments from purchases

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Display minimum payment due on payment screen
  • Add payment confirmation prompt before submitting
  • Format currency display with dollar sign

⚠️ Medium (3-5 pts)

  • Add payment scheduling (future date)
  • Overpayment warning when payment exceeds balance
  • Add EFT account reference field to payment record

🔴 Complex (5-8 pts)

  • Auto-payment setup (recurring monthly payments)
  • Payment allocation across multiple cards on one account
  • MQ-based payment confirmation to external EFT system

🔧 Technical Foundation

Key Components

VSAM Files Accessed

FileAccessPurpose
ACCTFILERead + UpdateValidate account status; update ACCT-CURR-BAL and cycle credits
CARDFILEReadValidate card/account linkage
TRANSACTWriteRecord payment as TRAN-RECORD with negative amount

Payment Transaction Record Pattern

TRAN-TYPE-CD     = "PA"   (Payment - example code)
TRAN-AMT         = negative value (credit to account)
TRAN-SOURCE      = "ONLINE"
TRAN-ORIG-TS     = current CICS timestamp
TRAN-CARD-NUM    = cardholder's primary card number
TRAN-DESC        = "BILL PAYMENT - [reference]"

Dependencies

  • accounts module: ACCTFILE and CVACT01Y copybook must be available
  • transactions module: TRANSACT VSAM and CVTRA05Y copybook for payment recording
  • credit-cards module: CARDFILE for card-to-account linkage validation
  • authentication module: User session (COCOM01Y) must be established
