🏷️ Transaction Types Optional Module

User Story Development Guide · CardDemo System · DB2-backed

📋 Module Overview

Purpose: Online CICS management of the DB2-backed transaction type catalog. Allows admins to add, edit, and delete transaction type codes and categories via interactive CICS screens.

Business Context: Transaction types define the nature of card transactions (e.g., purchase, refund, cash advance, fee, interest). When new product lines or transaction categories are introduced, admins need to add codes without code changes or batch updates. This module enables catalog management through the online system with DB2 providing referential integrity.

Location: app/app-transaction-type-db2/

Transaction IDs: CTLI (List) CTTU (Add/Edit)

Key Programs: COTRTLIC COTRTUPC

DB2 Tables: TRANSACTION_TYPE, TRANSACTION_TYPE_CATEGORY (schema: CARDDEMO)

⚠️ Prerequisites: DB2 subsystem required. COADM01C (Admin Menu) options 5/6 enabled only with this module installed. Requires Admin user type (SEC-USR-TYPE = 'A').

🎯 User Story Development Guide

📝 US Templates

As an admin, I want to add a new transaction type code so that card systems can process a new transaction category.
As an admin, I want to update a transaction type description so that it reflects current business terminology without a code deployment.
As an admin, I want to delete an obsolete transaction type so that it no longer appears in transaction entry dropdowns or validation.
As an admin, I want to list all transaction types with paging so that I can review the complete catalog.

✅ Acceptance Criteria Patterns

Transaction Type List (CTLI):
  • DB2 cursor pages forward (PF8) and backward (PF7) — different from VSAM STARTBR/READNEXT pattern
  • Inline selection: 'U' on row navigates to CTTU edit; 'D' on row triggers delete confirmation
  • At end of catalog, PF8 shows "NO MORE RECORDS"
  • Admin user type required; non-admin users must see "NOT AUTHORIZED" message
Add/Edit Type (CTTU):
  • Type code is 2-character, unique in DB2 TRANSACTION_TYPE table
  • Duplicate type code shows "TYPE CODE ALREADY EXISTS" error (DB2 PK violation handled)
  • Description field required (VARCHAR 50); blank description shows validation error
  • "RECORD SAVED SUCCESSFULLY" shown in GREEN on INSERT or UPDATE
  • Delete requires confirmation — row must have no dependent transactions in TRANSACT VSAM

🔧 Technical Foundation

🏗️ Key Components

  • COTRTLIC: Uses DB2 forward and backward cursors (two separate DECLARE CURSOR statements) for paging — distinguishes this from VSAM STARTBR/READNEXT pattern used elsewhere
  • COTRTUPC: DB2 INSERT (new type) or UPDATE (existing type) with SQLCA error handling; validates SQLCODE for duplicate key (SQLCODE = -803)
  • DCLTRTYP.dcl: DB2 host variable declaration copybook for TRANSACTION_TYPE
  • DCLTRCAT.dcl: DB2 host variable declaration copybook for TRANSACTION_TYPE_CATEGORY

🔗 Dependencies

  • transaction-management: TRAN-TYPE-CD in CVTRA05Y references TRANSACTION_TYPE.TRAN_TYPE_CD when DB2 module installed; referential integrity enforced in DB2
  • user-management: Admin user type (SEC-USR-TYPE = 'A') required; COADM01C checks this before showing menu options 5/6
  • DB2 subsystem: DEFINE DB2ENTRY(DB201PLN) in CICS CSD required; CICS DB2 attachment must be configured
