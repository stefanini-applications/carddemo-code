🗄️ Transaction Type DB2 Optional Module

User Story Development Guide — DB2 Transaction Type Reference Data Management

📋 Overview

Module ID: transaction-type-db2

Purpose: Manages transaction type reference data in DB2 relational tables, providing an alternative to VSAM-based storage. Demonstrates embedded static SQL patterns, cursor processing, and DB2 CRUD in a CICS environment.

Business Context: Transaction types are reference data used to categorize all financial transactions. This optional module shows how reference data management can be moved from VSAM to DB2, enabling richer querying and maintenance capabilities while maintaining backward compatibility via the TRANEXTR extraction job.

📦 Prerequisites: Base CardDemo installed plus: DB2 subsystem configured and accessible from CICS, DB2 database and tables created via CREADB21 JCL, COBTUPDT precompiled with DB2 precompiler.

🎯 User Story Templates

List Transaction Types

As an admin, I want to list all transaction types from DB2 so that I can review the current reference data catalog.
Acceptance Criteria:
  • Transaction CTLI (COTRTLIC) opens DB2 forward cursor to read transaction type records
  • List shows: TRAN-TYPE code, description (TRAN-TYPE-DESC)
  • Forward and backward scrolling via PF keys (cursor repositioned)
  • Admin can select a type to edit or delete from the list
  • Admin menu option 5 navigates to CTLI

Add Transaction Type

As an admin, I want to add a new transaction type to DB2 so that new transaction categories are available in the system.
Acceptance Criteria:
  • Transaction CTTU (COTRTUPC) provides add form
  • Required fields: TRAN-TYPE (2-char code), TRAN-TYPE-DESC (up to 50 chars)
  • Duplicate TRAN-TYPE code rejected with "Transaction type already exists"
  • SQL INSERT executed with host variables; SQLCA checked for errors
  • Success: confirmation message displayed
  • Admin menu option 6 navigates to CTTU

Update Transaction Type

As an admin, I want to update a transaction type description so that the UI shows accurate labels.
Acceptance Criteria:
  • CTTU or CTLI allows editing TRAN-TYPE-DESC for existing record
  • SQL UPDATE executed with WHERE TRAN-TYPE = :host-variable
  • SQLCA checked; successful update confirmed on screen
  • TRAN-TYPE (key) cannot be changed — only description

Delete Transaction Type

As an admin, I want to delete an obsolete transaction type from DB2 so that inactive types no longer appear in the reference data.
Acceptance Criteria:
  • CTLI allows delete of selected type with confirmation
  • SQL DELETE executed; SQLCA verified
  • Deletion of type in use by existing transactions warns admin (RI consideration)

Extract to VSAM

As a batch operator, I want to run TRANEXTR so that DB2 transaction types are synchronized to VSAM for the base application to use.
Acceptance Criteria:
  • TRANEXTR (DSNTIAUL) extracts all rows from DB2 transaction type table
  • Output file formatted compatibly with CVTRA03Y copybook
  • TRANTYPE and TRANCATG VSAM files updated with extracted data
  • RC=0 on success; error output to SYSOUT on failure

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Add record count display to CTLI list screen
  • Add search by type code prefix
  • Format type code as uppercase on entry

⚠️ Medium (3-5 pts)

  • Add active/inactive flag to transaction type table
  • Filter CTLI list by active status
  • Add audit timestamp columns (CREATED_TS, UPDATED_TS)

🔴 Complex (5-8 pts)

  • Stored procedure for bulk type maintenance
  • Dynamic SQL implementation for parameterized queries
  • Bi-directional sync: VSAM changes reflected to DB2

DB2 Integration Patterns Demonstrated

  • Static Embedded SQL: Host variables in EXEC SQL statements, precompiled with DB2 precompiler
  • Cursor Processing: COTRTLIC demonstrates forward SCROLL and backward SCROLL cursors
  • SQLCA Handling: SQLCODE and SQLERRM checked after each SQL operation
  • CRUD Operations: All four operations (INSERT, SELECT, UPDATE, DELETE) demonstrated
