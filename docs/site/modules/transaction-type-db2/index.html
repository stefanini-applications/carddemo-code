🗄️ Transaction Type DB2

User Story Development Guide — Admin Transaction Type Management with Db2

← Back to Documentation Index
⚠️ Optional Module — Requires IBM Db2 infrastructure

Module ID: transaction-type-db2

Purpose: Admin module for managing transaction type and category reference data stored in IBM Db2 tables. Demonstrates CICS-Db2 integration patterns including cursor operations, inserts, updates, and deletes.

🎯 User Story Development Guide

📝 US Templates

  • List Types: As an admin, I want to list all transaction types so that I can review available charge categories.
  • Add Type: As an admin, I want to add a new transaction type so that new charge categories are available for transactions.
  • Update Type: As an admin, I want to update a transaction type description so that it accurately reflects the charge category.
  • Delete Type: As an admin, I want to delete an obsolete transaction type so that it is no longer available for new transactions.
  • Sync to VSAM: As a batch system, I want to extract Db2 transaction types to VSAM so that the online COBOL programs can access them.

🔧 Technical Foundation

🏗️ Key Components

  • COTRTLIC (CICS) — List/update/delete transaction types; uses Db2 cursor for paginated list; demonstrates DELETE with WHERE clause
  • COTRTUPC (CICS) — Add/edit transaction types; demonstrates INSERT and UPDATE on Db2
  • COBTUPDT (batch) — Batch update of transaction types in Db2 from sequential input
  • COTRTLI / COTRTUP (BMS maps) — Transaction type list and update screens
  • CSDB2RPY / CSDB2RWY (copybooks) — Db2 communication area for read and write
  • DCLTRTYP.dcl / DCLTRCAT.dcl (DCLGEN) — Db2 table column declarations
  • TRNTYPE.ddl / TRNTYCAT.ddl (DDL) — Db2 table definitions
  • XTRNTYPE.ddl / XTRNTYCAT.ddl — Extended index definitions

🔗 Dependencies

  • IBM Db2: TRNTYPE and TRNTYCAT tables; CICS RCT (Resource Control Table) must be configured for Db2 access
  • TRANEXTR JCL: Extracts Db2 data to sequential file for VSAM load
  • TRANCATG / TRANTYPE (VSAM): VSAM counterparts loaded from Db2 via TRANEXTR
  • Transaction Processing module: TRAN-TYPE-CD and TRAN-CAT-CD validated against these tables

🌐 CICS Transactions

CTTU  → COTRTUPC  — Transaction type add/edit (INSERT/UPDATE on Db2)
CTLI  → COTRTLIC  — Transaction type list/update/delete (cursor, DELETE on Db2)

📋 Db2 Table Structures

TRNTYPE (Transaction Type):
  TRAN-TYPE       CHAR(2)         — Transaction type code (PK)
  TRAN-TYPE-DESC  CHAR(50)        — Description

TRNTYCAT (Transaction Category):
  TRAN-TYPE-CD    CHAR(2)         — Transaction type (composite PK)
  TRAN-CAT-CD     DECIMAL(4)      — Category code (composite PK)
  TRAN-CAT-DESC   CHAR(50)        — Category description

📋 VSAM Sync Pattern

1. Admin updates type/category via CTTU/CTLI (Db2 updated)
2. Nightly TRANEXTR JCL extracts Db2 → sequential file
3. TRANCATG/TRANTYPE JCL loads sequential file → VSAM
4. Online COBOL programs read from VSAM for lookups

✅ Acceptance Criteria Examples

  • List Types: Given Db2 TRNTYPE table has records, when admin navigates to CTLI, then all transaction types are listed with code and description
  • Add Type: Given a unique 2-char type code and description, when admin submits CTTU, then the record is inserted into Db2 TRNTYPE
  • Delete Type: Given an existing type code, when admin deletes via CTLI, then the Db2 record is removed and no longer appears in the list
  • VSAM Sync: Given Db2 types are updated and TRANEXTR runs, then VSAM TRANTYPE file reflects the current Db2 data
