🗃️ Transaction Type DB2

User Story Development Guide — CardDemo Optional Module

⚠️ Optional Module — Requires DB2 subsystem. Extends the base Admin Menu (CA00) with transaction type management options 5 and 6.

📋 Overview

Module ID: transaction-type-db2

Purpose: Admin maintenance of transaction type and category reference data in DB2 tables, with batch extraction to VSAM for integration with the base transaction processing system.

Business Context: Demonstrates DB2 integration in CICS — static embedded SQL, cursor processing, CRUD operations. Batch TRANEXTR extracts DB2 data to VSAM files used by POSTTRAN and TRANREPT.

CTLI: COTRTLIC (list/update/delete) CTTU: COTRTUPC (add/edit) DB2 Tables: TRANTYPE, TRANCAT

🎯 User Story Development Guide

📝 US Templates

  • As an admin, I want to list all transaction types in DB2 so that I can manage reference data
  • As an admin, I want to add a new transaction type to DB2 so that new payment methods can be categorized
  • As an admin, I want to update a transaction type description so that the reference data stays accurate
  • As an admin, I want to delete an obsolete transaction type so that it no longer appears as a valid option
  • As a system operator, I want DB2 transaction types extracted to VSAM so that batch processing uses the latest reference data

⚡ Development Acceleration Factors

  • COTRTLIC: Demonstrates forward/backward cursor processing — reference for scrollable list with DB2
  • COTRTUPC: DB2 INSERT and UPDATE with static embedded SQL — precompiler pattern
  • CSDB2RPY/CSDB2RWY copybooks: DB2 reply and request structures for CICS-DB2 integration
  • DDL files: TRNTYPE.ddl, TRNTYCAT.ddl — table definitions available for reference
  • DCL files: DCLTRTYP.dcl, DCLTRCAT.dcl — DB2 declarations for COBOL precompiler

📊 Complexity Guidelines

  • Simple (1-2 pts): Display DB2 transaction type list with forward cursor (CTLI view)
  • Medium (3-5 pts): Add/edit transaction type with DB2 INSERT/UPDATE (CTTU); delete with DB2 DELETE (CTLI)
  • Complex (5-8 pts): Bidirectional cursor navigation in CTLI; TRANEXTR batch extraction to VSAM

⚙️ Business Rules

  • Transaction type code is 2-character alphanumeric (matches TRAN-TYPE-CD in CVTRA03Y)
  • Transaction category linked to transaction type — both must be consistent
  • DB2 changes must be extracted via TRANEXTR to VSAM before they affect batch POSTTRAN processing
  • CICS programs use DB2 precompiler (static SQL) — recompile required for schema changes
  • SQLCA checked after every SQL statement — error conditions handled and displayed to user
  • Admin menu options 5 (CTTU) and 6 (CTLI) only appear when this optional module is installed

✅ Acceptance Criteria Patterns

  • Given transaction types exist in DB2, when CTLI is invoked, then all types are listed with code and description
  • Given a new unique transaction type code, when admin submits CTTU add form, then record is inserted in DB2 TRANTYPE table
  • Given a duplicate transaction type code, when admin submits CTTU add form, then DB2 SQLCODE -803 (unique constraint) error is displayed
  • Given TRANEXTR batch job runs, when it completes, then VSAM TRANTYPE and TRANCATG files are updated with current DB2 data
