⚙️ Batch Processing

User Story Development Guide — Nightly Batch Cycle & Data Maintenance

← Back to Documentation Index

Module ID: batch-processing

Purpose: End-of-day and periodic batch jobs responsible for transaction posting, interest calculation, statement generation, file initialization, reference data loading, and data maintenance. Runs during the nightly batch window when CICS files are closed.

🎯 User Story Development Guide

📝 US Templates

  • Nightly Cycle: As a batch operator, I want to run the full nightly batch cycle so that all daily activity is processed before the next business day.
  • Post Transactions: As a batch system, I want to post daily transactions to account balances so that accounts reflect the day's charges.
  • Interest Calculation: As a batch system, I want to calculate interest charges so that finance charges are applied on schedule.
  • Reject Handling: As a batch operator, I want to review rejected transactions so that errors can be corrected and reprocessed.
  • File Initialize: As a batch operator, I want to initialize VSAM files with sample data so that the system is ready for testing or a new environment.
  • File Management: As a batch system, I want CLOSEFIL to run before batch and OPENFIL after so that VSAM files are properly managed between batch and online.

⚡ Development Acceleration Factors

  • CLOSEFIL/OPENFIL pattern: Standard CICS file management JCL — required bookend for all batch cycles
  • CBTRN01C validation: Multi-file cross-reference validation reusable for new transaction types
  • CSUTLDTC date utility: Date conversion/comparison utility available to all batch programs
  • COBSWAIT/MVSWAIT: Timer control for batch step sequencing
  • REPROC.prc JCL procedure: Parameterized batch procedure for REPRO operations
  • CA7/Control-M schedulers: Job scheduling definitions already provided

🔧 Technical Foundation

🏗️ Key Batch Programs

  • CBTRN01C — Daily transaction posting: reads DALYTRAN, validates vs CARDFILE/CUSTFILE/XREFFILE/ACCTFILE, posts to TRANSACT, updates ACCTDAT + TCATBALF, rejects to DALYREJS
  • CBTRN02C — Alternate posting path (same files, different validation flow)
  • CBTRN03C — Transaction file maintenance
  • CBACT01C–CBACT04C — Account file sequential reads and reporting; demonstrate COMP/COMP-3 data formats
  • CBCUS01C — Customer file sequential processing
  • CBSTM03A/B — Statement generation (two-pass sort/merge)
  • CBEXPORT / CBIMPORT — Data migration utilities
  • COBSWAIT (CICS) — Wraps MVSWAIT assembler timer routine
  • CSUTLDTC — Date conversion utility using COBDATFT assembler
  • MVSWAIT.asm — Assembler timer control for WAITSTEP JCL step
  • COBDATFT.asm — Date format conversion (called by CSUTLDTC)

🔗 VSAM Files Used by Batch

  • DALYTRAN (Sequential): Input for POSTTRAN — daily transaction records from external systems
  • DALYREJS (Sequential): Output for rejected transactions — reviewed and reprocessed manually
  • ACCTDAT (KSDS): Account balances updated during posting
  • CARDDAT (KSDS): Card validation during posting
  • CUSTDAT (KSDS): Customer validation during posting
  • CARDXREF (KSDS): Card-to-account resolution during posting
  • TRANSACT (KSDS): Target for posted transactions
  • TCATBALF (KSDS): Transaction category balances updated per post

📋 Full Batch Sequence

Nightly Batch Execution Order:
  1.  CLOSEFIL   — Close VSAM files in CICS
  2.  ACCTFILE   — Reload account data from sample (or backup)
  3.  CARDFILE   — Reload card data
  4.  XREFFILE   — Reload card-account cross-reference
  5.  CUSTFILE   — Reload customer data
  6.  TRANBKP    — Backup transaction file
  7.  TRANEXTR   — Extract Db2 transaction types (optional)
  8.  TRANCATG   — Update transaction category VSAM from extract
  9.  TRANTYPE   — Update transaction type VSAM from extract
  10. DISCGRP    — Update disclosure group VSAM
  11. TCATBALF   — Initialize category balance file
  12. DUSRSECJ   — Reset user security file
  13. POSTTRAN   — Core transaction posting (CBTRN01C)
  14. INTCALC    — Interest calculation
  15. TRANBKP    — Post-processing backup
  16. COMBTRAN   — Combine system + daily transactions
  17. CREASTMT   — Generate account statements (CBSTM03A/B)
  18. TRANIDX    — Define alternate index on transaction file
  19. OPENFIL    — Reopen VSAM files for CICS
  20. WAITSTEP   — Optional: pause before next job step

📋 Business Rules

  • BR-BATCH-01: CLOSEFIL must run before any batch update jobs; OPENFIL must run after
  • BR-BATCH-02: Batch sequence must be executed in the defined order to maintain data integrity
  • BR-BATCH-03: POSTTRAN validates card, account, and customer existence before posting each transaction
  • BR-BATCH-04: Rejected transactions in DALYREJS include an error code indicating the rejection reason
  • BR-BATCH-05: TCATBALF is reset and rebuilt each batch cycle
  • BR-BATCH-06: GDG datasets used for versioned backups of key files

✅ Acceptance Criteria Examples

  • Full Cycle: Given the batch sequence runs successfully, then CICS files are reopened and available for online access before the start of business
  • Transaction Posting: Given 1,000 records in DALYTRAN, when POSTTRAN runs, then all valid records are written to TRANSACT and ACCTDAT balances are updated; invalid records are in DALYREJS
  • Interest Calculation: Given accounts with balances > 0 after POSTTRAN, when INTCALC runs, then interest charges are computed using the DISCGRP rates for each account group
  • Performance: Full batch cycle must complete within 6 hours; POSTTRAN must process 10,000 transactions within 4 hours
  • File Safety: Given CLOSEFIL has not run, when an update batch job is attempted, then the job must not run (JCL prerequisite check)
