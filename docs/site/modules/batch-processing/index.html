⚙️ Batch Processing

User Story Development Guide — CardDemo Mainframe Application

📋 Overview

Module ID: batch-processing

Purpose: Nightly and periodic batch jobs for transaction posting, interest calculation, data file loading/refreshing, statement generation, and reporting. Executed via JCL on z/OS.

Business Context: Batch processing is the operational backbone — account balances are only updated during batch (POSTTRAN), interest is applied during batch (INTCALC), and customer statements are produced during batch (CREASTMT).

POSTTRAN: CBTRN02C INTCALC: CBACT04C CREASTMT: CBSTM03A Schedulers: CA7, Control-M

🎯 User Story Development Guide

📝 US Templates

  • As a system operator, I want daily transactions posted automatically each night so that account balances are current by morning
  • As a system operator, I want interest calculated and posted monthly so that finance charges are applied correctly
  • As a system operator, I want invalid transactions reported to a rejection file so that they can be reviewed and corrected
  • As a system operator, I want batch jobs orchestrated in the correct sequence so that dependencies are respected (e.g., POSTTRAN before INTCALC)
  • As a developer, I want batch jobs to produce clear return codes so that job scheduling can detect failures automatically

⚡ Development Acceleration Factors

  • CBTRN02C (POSTTRAN): Complete multi-file batch processing reference — DALYTRAN input, TRANSACT output, ACCTFILE update, TCATBAL update, DALYREJS rejection
  • CBACT04C (INTCALC): Interest calculation pattern using DISCGRP lookup by account group + transaction type + category
  • Job scheduler definitions: CA7 (app/scheduler/CardDemo.ca7) and Control-M (app/scheduler/CardDemo.controlm) templates available
  • COBSWAIT: Assembler-based timer (MVSWAIT) for inserting wait steps between jobs
  • CBEXPORT/CBIMPORT: Data export and import utilities for data migration scenarios

📊 Complexity Guidelines

  • Simple (1-2 pts): Data refresh job (IDCAMS REPRO of flat file to VSAM)
  • Medium (3-5 pts): Sequential processing with single VSAM update (e.g., CBCUS01C read/print)
  • Complex (5-8 pts): Multi-file processing with cross-reference lookups and rejection handling (CBTRN02C, CBACT04C)
  • Very Complex (8-13 pts): Statement generation with multiple output formats, subroutine calls, complex data structures (CBSTM03A)

⚙️ Batch Sequence and Dependencies

CLOSEFIL → [data refresh jobs] → DUSRSECJ → POSTTRAN
    → INTCALC → TRANBKP → COMBTRAN → CREASTMT
    → TRANIDX → OPENFIL

⚙️ Business Rules

  • CLOSEFIL must run before any VSAM file refresh (files must be closed before IDCAMS REPRO)
  • POSTTRAN processes DALYTRAN sequentially; invalid records (bad card, inactive account) → DALYREJS
  • TCATBALF updated by POSTTRAN; INTCALC reads TCATBALF for category-level balance data
  • Interest rate from DISCGRP: key = ACCT-GROUP-ID + TRAN-TYPE-CD + TRAN-CAT-CD
  • GDG (Generation Data Group) used for archival datasets — defined by DEFGDGB/DEFGDGD
  • Scheduler definitions available for CA7 and Control-M job scheduling systems

✅ Acceptance Criteria Patterns

  • Given DALYTRAN contains valid transactions, when POSTTRAN runs, then all valid transactions are written to TRANSACT and account balances are updated; return code = 0
  • Given DALYTRAN contains invalid transactions, when POSTTRAN runs, then invalid records are written to DALYREJS with error reason; valid records still processed
  • Given POSTTRAN has completed, when INTCALC runs, then finance charges are calculated per disclosure group rate and posted to accounts
  • Given CREASTMT runs, when CBSTM03A completes, then both STMTFILE (plain text) and HTMLFILE (HTML) are produced with RC=0
  • Given any batch job fails, when the job completes, then return code is non-zero and downstream dependent jobs should not execute
