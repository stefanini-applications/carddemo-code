📊 Reporting

User Story Development Guide — Transaction Reports & Account Statements

← Back to Documentation Index

Module ID: reporting

Purpose: Provides online report request submission (CORPT00C) which writes JCL to an extra-partition TDQ for batch execution. Batch programs (CBSTM03A/B) generate transaction statements and account summaries.

🎯 User Story Development Guide

📝 US Templates

  • Request Report: As a card holder, I want to request a transaction report for a date range so that I can review my spending period.
  • Monthly Statement: As a card holder, I want to receive a monthly statement so that I have a consolidated record of all activity.
  • Export Data: As an admin, I want to export account data so that it can be used for analysis or migration.
  • Report Status: As a card holder, I want to know when my report is being generated so that I can expect it in the output queue.

⚡ Development Acceleration Factors

  • CORPT00C TDQ pattern: JCL submission via extra-partition TDQ already implemented — new report types just need new JCL written to the queue
  • TRANREPT.prc JCL procedure: Parameterized report procedure; add new steps for new report formats
  • CBSTM03A/B sort-merge pattern: Statement generation uses two-pass sort/merge; reuse for new statement formats
  • CVEXPORT copybook: Export record structure available for data extraction

📊 Complexity Guidelines

🔧 Technical Foundation

🏗️ Key Components

  • CORPT00C (CICS) — Report request screen; accepts date range and report type; writes JCL to extra-partition TDQ (CICS WRITEQ TD)
  • CORPT00 (BMS map) — Report request 3270 screen
  • CBSTM03A (batch) — Statement generation first pass (sort/select)
  • CBSTM03B (batch) — Statement generation second pass (merge/format)
  • CBEXPORT / CBIMPORT (batch) — Data export and import between VSAM and sequential files
  • TRANREPT.prc — JCL procedure for transaction reports
  • TXT2PDF1.JCL — Optional: converts text reports to PDF format
  • FTPJCL.JCL — Optional: FTP delivery of report output

🔗 Dependencies

  • TRANSACT (KSDS): Source data for all transaction reports
  • ACCTDAT (KSDS): Account data for statement header information
  • Extra-partition TDQ: CICS queue used to submit JCL to internal reader
  • Batch Processing module: Report JCL jobs run in the batch environment

🌐 CICS Transactions & JCL Jobs

CR00  → CORPT00C  — Online report request (submits JCL via TDQ)

JCL Jobs:
  CREASTMT  — Create account statement (CBSTM03A/B)
  TRANREPT  — Transaction report (TRANREPT.prc)
  REPTFILE  — Report file utility
  CBADMCDJ  — Admin report JCL

📋 Statement Record Structure (COSTM01)

Statement records produced by CBSTM03A/B include:
  - Account header (account ID, customer name, statement date)
  - Transaction detail lines (date, description, amount)
  - Summary totals (by category, by type)
  - Balance summary (previous balance, payments, charges, new balance)

📋 Business Rules

  • BR-RPT-01: Reports are submitted via extra-partition TDQ — CICS writes JCL records that the internal reader picks up
  • BR-RPT-02: Report date range must be in YYYY-MM-DD format
  • BR-RPT-03: Report output is written to SYSOUT dataset; available via JES spool
  • BR-RPT-04: Statement totals must reconcile with TCATBALF running balances

✅ Acceptance Criteria Examples

  • Report Request: Given dates in YYYY-MM-DD format, when user submits CR00, then JCL is written to the TDQ and the report is queued for batch execution
  • Date Validation: Given an invalid date format, when user submits, then an error message is displayed and the request is not submitted
  • Statement Accuracy: Given CREASTMT runs, then statement totals match the sum of TRANSACT records for the statement period
  • Export: Given CBEXPORT runs, then all ACCTDAT records are written to the sequential output file in the CVEXPORT format
