📨 MQ Integration Optional Module

User Story Development Guide — Asynchronous MQ Request/Response Patterns

📋 Overview

Module ID: mq-integration

Purpose: Demonstrates asynchronous IBM MQ request/response integration patterns for two use cases: system date inquiry (CDRD) and account details extraction (CDRA). Shows how to decouple external systems from CICS using MQ.

Business Context: This optional module is primarily a technical demonstration for modernization scenarios. It shows how CICS applications can expose services asynchronously via MQ, enabling integration with distributed or cloud-based systems without synchronous CICS coupling.

📦 Prerequisites: Base CardDemo installed plus: IBM MQ configured and accessible from CICS, MQ queues defined (request and reply queues for CDRD and CDRA transactions).

🎯 User Story Templates

System Date Inquiry via MQ

As an external system, I want to query the current system date from CardDemo via MQ so that my timestamps are synchronized with the mainframe.
Acceptance Criteria:
  • External client sends date inquiry request to configured MQ request queue
  • CDRD (CODATE01) triggered by MQ message in CICS
  • CODATE01 retrieves current system date/time
  • Response message written to MQ reply queue in expected format
  • External client reads response and obtains mainframe system date
  • Round-trip latency meets SLA (configurable timeout)

Account Details via MQ

As an external system, I want to retrieve account details from CardDemo via MQ so that I can display account information in a distributed application.
Acceptance Criteria:
  • External client sends account inquiry request (with account ID) to MQ request queue
  • CDRA (COACCT01) triggered by MQ message in CICS
  • COACCT01 reads ACCOUNT-RECORD from ACCTFILE VSAM using provided account ID
  • Account data formatted and written to MQ reply queue
  • If account not found, error response written to reply queue
  • Response includes: ACCT-ID, ACCT-CURR-BAL, ACCT-CREDIT-LIMIT, ACCT-ACTIVE-STATUS

MQ Pattern Documentation for Architects

As a modernization architect, I want documentation of the MQ request/response pattern used in CardDemo so I can design equivalent services for cloud migration.
Acceptance Criteria:
  • Pattern documented: client sends to request queue, CICS program triggered, reads VSAM, writes to reply queue
  • Message formats documented for request and response for both CDRD and CDRA
  • Equivalent REST API design documented as migration target pattern

📊 Complexity Guidelines

✅ Simple (1-2 pts)

  • Add additional date format to CODATE01 response
  • Add account status field to COACCT01 response
  • Add MQ message ID logging

⚠️ Medium (3-5 pts)

  • Add customer inquiry via MQ (new COCU01 MQ program)
  • Add request validation and error response codes
  • Add MQ dead-letter queue handling for failed requests

🔴 Complex (5-8 pts)

  • New MQ-based transaction submission service (write to TRANSACT)
  • MQ-to-REST API bridge for cloud integration demonstration
  • Correlation ID tracking across multiple MQ round-trips

MQ Configuration
