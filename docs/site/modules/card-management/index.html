💳 Card Management

User Story Development Guide — Credit Card List, View & Update

← Back to Documentation Index

Module ID: card-management

Purpose: Enables users to list all credit cards linked to their account, view card details (number, CVV, expiry, status), and update card status or embossed name.

🎯 User Story Development Guide

📝 US Templates

  • List Cards: As a card holder, I want to see all credit cards on my account so that I can choose which card to manage.
  • View Card: As a card holder, I want to view card details including CVV and expiration so that I can use it for purchases.
  • Deactivate Card: As a card holder, I want to mark my card as inactive so that a lost or stolen card cannot be used.
  • Card Expiry: As a card holder, I want to see my card expiration date so that I know when to request renewal.
  • Cross-Reference: As a system, I want to resolve card-to-account via the CARDXREF file so that transactions can be linked to the correct account.

⚡ Development Acceleration Factors

  • CVACT02Y copybook: Card record layout shared across all card operations and batch programs
  • CVACT03Y (CARDXREF): Single cross-reference record links card → customer → account — one read resolves the full relationship
  • CARDAIX (AIX): Alternate index on CARDDAT enables account-based card lookup
  • COCRDLIC pagination: Existing list program supports scrolling through multiple cards

📊 Complexity Guidelines

1–2 pts
Simple
  • Display card type/embossed name on list screen
  • Add read-only CVV display to detail screen
3–5 pts
Medium
  • Implement card status toggle with confirmation dialog
  • Add card expiration date update with validation
5–8 pts

  • Add new card issuance flow (create card + XREF records)
  • Implement card replacement workflow (deactivate old, issue new)

🔧 Technical Foundation

🏗️ Key Components

  • COCRDLIC — Credit card list; paginates cards for an account using CARDAIX alternate index
  • COCRDSLC — Card detail view; reads by CARD-NUM KSDS key
  • COCRDUPC — Card update; CICS READ UPDATE / REWRITE on CARDDAT
  • COCRDLI / COCRDSL / COCRDUP (BMS maps) — 3270 screen layouts
  • CVACT02Y (copybook) — Card record (150 bytes)
  • CVACT03Y (copybook) — Card cross-reference record (50 bytes)

🔗 Dependencies

  • CARDDAT (KSDS): Card master file, keyed on CARD-NUM (16 chars)
  • CARDAIX (AIX): Alternate index enabling account-based lookups
  • CARDXREF (KSDS): Card-to-account-to-customer cross-reference
  • ACCTDAT (KSDS): Account status checked when displaying card validity

🌐 CICS Transactions

CCLI  → COCRDLIC  — Credit card list (paginated)
CCDL  → COCRDSLC  — Credit card detail view
CCUP  → COCRDUPC  — Credit card update

📋 Card Data Models

CARD-RECORD (CVACT02Y, 150 bytes, CARDDAT KSDS):
  CARD-NUM              PIC X(16)   — Card number (KSDS key)
  CARD-ACCT-ID          PIC 9(11)   — Linked account ID
  CARD-CVV-CD           PIC 9(03)   — CVV security code
  CARD-EMBOSSED-NAME    PIC X(50)   — Name on card
  CARD-EXPIRAION-DATE   PIC X(10)   — YYYY-MM-DD
  CARD-ACTIVE-STATUS    PIC X(01)   — 'Y'=Active, 'N'=Inactive

CARD-XREF-RECORD (CVACT03Y, 50 bytes, CARDXREF KSDS):
  XREF-CARD-NUM         PIC X(16)   — Card number (KSDS key)
  XREF-CUST-ID          PIC 9(09)   — Customer ID
  XREF-ACCT-ID          PIC 9(11)   — Account ID

📋 Business Rules

  • BR-CARD-01: Each card links to exactly one account (CARD-ACCT-ID)
  • BR-CARD-02: Card status 'Y' = active; inactive cards cannot be used for transactions
  • BR-CARD-03: CARD-EMBOSSED-NAME reflects the cardholder name as it appears on the physical card
  • BR-CARD-04: CARDXREF is required to resolve card → customer → account in batch transaction posting

✅ Acceptance Criteria Examples

  • List Cards: Given an authenticated user, when navigating to CCLI, then all cards linked to the user's account are listed with card number (masked), status, and expiration
  • View Details: Given a card is selected from the list, when CCDL is displayed, then full card details including CVV and embossed name are shown
  • Deactivate: Given an active card, when the user updates status to 'N' and confirms, then CARDDAT is updated and the card appears as inactive in the list
  • Performance: Card list and detail views must display within 3 seconds
