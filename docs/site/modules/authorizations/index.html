🚨 Authorizations

Guide for crafting stories around MQ-triggered authorizations, IMS history, and fraud analytics.

📋 Overview

Purpose: Deliver the MQ → CICS → IMS/DB2 lifecycle that simulates credit card authorization requests, displays pending/ detail screens, and marks fraudulent activity.

Business Context: Mirrors modern authorization flows so teams understand how to trace requests through MQ queues, IMS segments, and DB2 fraud tables when defining product stories.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Monitoring: As a fraud analyst, I want to view `CPVS` so I can flag pending authorizations before settlement.
  • Investigation: As an integration engineer, I want to send MQ requests to `CP00` with the documented CSV payload and inspect the response.
  • Maintenance: As an ops lead, I want to run `CBPAUP0J` to purge expirations and adjust credit availability.

⚡ Development Acceleration Factors

  • MQ format reuse: The request/response CSV order is documented and the COBOL copybooks (`CCPAURQY`, `CCPAURLY`) already parse it.
  • IMS & DB2 templates: DBD/PSB (`DBPAUTP0`, `PSBPAUTL/B`) definitions plus DB2 schema `AUTHFRDS` exist, so new segments follow the same structure.
  • Event queue wiring: MQ connections (`MQ01`, `CARDREQ`, `CARDRES`) are pre-defined in `app/app-authorization-ims-db2-mq/csd`.

📊 Complexity Guidelines

  • Simple (1-2 pts): Add a new display field to the `COPAU00` map or adjust the PF-key prompt.
  • Medium (3-5 pts): Tweak the CSV parser in `COPAUA0C` to accept a new merchant field and ensure the response still includes the original columns.
  • Complex (5-8 pts): Extend the two-phase commit across IMS and DB2 so that a fraud flag in `AUTHFRDS` only persists if both DB2 and IMS updates succeed.

🔧 Technical Foundation

🏗️ Key Components

  • COBOL programs: `COPAUA0C` (MQ trigger), `COPAUS0C`, `COPAUS1C`, and `COPAUS2C` (fraud flag updates).
  • Copybooks: `CIPAUSMY`, `CIPAUDTY`, `CCPAURQY`, `CCPAURLY` provide IMS segments and MQ payload parsing.
  • Persisted stores: IMS HIDAM `DBPAUTP0` with segments `PAUTSUM0`/`PAUTDTL1` and DB2 table `AUTHFRDS`.
  • MQ queues: `AWS.M2.CARDDEMO.PAUTH.REQUEST/REPLY` (requests, replies).

🔗 Dependencies

  • IMS HIDAM (DBDs/PSBs defined under `app/app-authorization-ims-db2-mq/ims`).
  • DB2 schema + plans (`app/app-authorization-ims-db2-mq/ddl` and `csd` entries).
  • MQ connection definitions (`app/app-authorization-ims-db2-mq/csd` and `app/app-authorization-ims-db2-mq/mq`).
  • VSAM datasets for accounts/customers referenced by the authorization programs.

🌐 Public Interfaces

CP00  (COPAUA0C) - MQ trigger that reads AWS.M2.CARDDEMO.PAUTH.REQUEST and replies to AWS.M2.CARDDEMO.PAUTH.REPLY
CPVS  (COPAUS0C) - Summary screen over IMS PAUTSUM0 segments (PF7/PF8 paging)
CPVD  (COPAUS1C) - Detail screen with PF5 fraud marking that calls COPAUS2C
CBPAUP0J - Batch purge job (CBPAUP0C) that expires authorizations and recalculates credit

✅ Acceptance Criteria Examples

  • MQ requests must parse the documented CSV payload, correlate with `REQUEST-ID`, and write a reply containing `AUTH-RESP-CODE` and `APPROVED-AMT`.
  • The authorization detail screen must allow PF5 to toggle `AUTH_FRAUD='Y'` in DB2 and mark the corresponding IMS segment.
  • `CBPAUP0J` must delete expired entries and ensure VSAM/IMS resources remain consistent.
