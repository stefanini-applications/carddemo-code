💸 Transactions

User Story Development Guide — CardDemo Mainframe Application

📋 Overview

Module ID: transactions

Purpose: Online listing, viewing, and adding of credit card transactions stored in the TRANSACT VSAM KSDS file. Daily transactions from external sources land in DALYTRAN (sequential) and are posted via batch.

Business Context: Transactions are the financial heart of CardDemo. Each transaction references a card number, transaction type, and category. Online addition goes directly to TRANSACT; batch posting processes the DALYTRAN daily file.

CT00: COTRN00C (list) CT01: COTRN01C (view) CT02: COTRN02C (add) VSAM: TRANSACT KSDS + DALYTRAN

🎯 User Story Development Guide

📝 US Templates

  • As a regular user, I want to see a list of my transactions so that I can review recent activity
  • As a regular user, I want to view detailed transaction information so that I can verify a specific charge
  • As a regular user, I want to add a new transaction manually so that I can record a purchase immediately
  • As a regular user, I want to filter transactions by date so that I can find transactions from a specific period

⚡ Development Acceleration Factors

  • CVTRA05Y copybook: Complete 350-byte TRAN-RECORD with all fields pre-defined
  • CVTRA06Y copybook: Daily transaction record (same 350-byte structure) for batch processing
  • Transaction types and categories: TRANTYPE and TRANCATG VSAM files provide reference data for validation
  • TRAN-ID generation: Must be unique 16-char ID — review COTRN02C for existing generation logic

📊 Complexity Guidelines

  • Simple (1-2 pts): Transaction view by ID (CT01)
  • Medium (3-5 pts): Transaction list with pagination and browse (CT00); transaction add with type/category validation (CT02)
  • Complex (5-8 pts): Filtered list by date range; cross-reference lookup to include merchant information

📋 Transaction Data Model

01  TRAN-RECORD.                   -- 350 bytes, VSAM KSDS
    05  TRAN-ID             PIC X(16).   -- Primary key (unique)
    05  TRAN-TYPE-CD        PIC X(02).   -- Must exist in TRANTYPE
    05  TRAN-CAT-CD         PIC 9(04).   -- Must exist in TRANCATG
    05  TRAN-SOURCE         PIC X(10).
    05  TRAN-DESC           PIC X(100).
    05  TRAN-AMT            PIC S9(09)V99.
    05  TRAN-MERCHANT-ID    PIC 9(09).
    05  TRAN-MERCHANT-NAME  PIC X(50).
    05  TRAN-MERCHANT-CITY  PIC X(50).
    05  TRAN-MERCHANT-ZIP   PIC X(10).
    05  TRAN-CARD-NUM       PIC X(16).   -- Links to CARDFILE
    05  TRAN-ORIG-TS        PIC X(26).   -- Original timestamp
    05  TRAN-PROC-TS        PIC X(26).   -- Processing timestamp
    05  FILLER              PIC X(20).

⚙️ Business Rules

  • TRAN-TYPE-CD must match a valid record in TRANTYPE VSAM file
  • TRAN-CAT-CD must match a valid record in TRANCATG VSAM file (keyed by TRAN-TYPE-CD + TRAN-CAT-CD)
  • TRAN-ID must be unique within TRANSACT file (DUPREC response from WRITE = duplicate)
  • Batch POSTTRAN (CBTRN02C) updates account balances based on TRAN-AMT; invalid records go to DALYREJS
  • Online CT02 writes directly to TRANSACT; does NOT update account balance (batch does)
  • TRAN-CARD-NUM must correspond to a valid card in CARDFILE/XREFFILE

✅ Acceptance Criteria Patterns

  • Given an account with transactions, when transaction list is requested, then transactions are displayed in chronological order with ID, amount, and description
  • Given a valid transaction ID, when transaction view is requested, then all 14 transaction fields are displayed
  • Given valid transaction data, when transaction add is submitted, then a new record is written to TRANSACT and a confirmation is displayed
  • Given an invalid TRAN-TYPE-CD, when transaction add is submitted, then a validation error "Invalid transaction type" is displayed
  • Given a duplicate TRAN-ID, when transaction add is submitted, then a "Duplicate transaction ID" error is displayed
